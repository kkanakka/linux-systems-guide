Section 1

Overview & Scale Estimation

The Top K Songs problem asks us to build a system that continuously computes the most-played songs across a massive streaming platform. Unlike simple counters, we need real-time freshness (charts update within minutes), exact historical accuracy (daily/weekly official charts), and multi-dimensional slicing (by country, genre, playlist, and time window).

Functional Requirements

  • Real-time Top 50 — Return top 50 songs globally and per-dimension, updated within 1–2 minutes of a play event.
  • Historical Charts — Daily, weekly, and monthly official charts computed from exact play counts.
  • Multi-Dimensional — Slice by Global, Country, City, Genre, Playlist, and arbitrary time periods.
  • Deduplication — A user playing the same song on repeat should count only once per 30-second window (fraud prevention).
  • API — Clients query GET /top-k?dimension=country&value=US&k=50 and receive an ordered list.

Section 2

Lambda Architecture

We adopt a Lambda Architecture to get the best of both worlds: the hot path provides sub-minute freshness using approximate counting, while the cold path computes exact counts in batch. A serving layer merges both views to give users accurate, near-real-time results.

Figure 1 — Lambda Architecture: Hot path (stream) + Cold path (batch) merging into the Serving Layer

Architecture Walkthrough

  1. Play Event Ingestion: Every song play generates an event containing user_id, song_id, timestamp, country, city, genre_ids[], and playlist_id. Events are published to a Kafka topic partitioned by song_id for ordering.
  2. Deduplication Gate: A stateful filter (keyed by user_id + song_id) drops duplicate plays within a 30-second tumbling window. This prevents repeat-play fraud from inflating counts.
  3. Hot Path (Real-Time): A Samza stream processor consumes deduplicated events. For each dimension, it updates a Count-Min Sketch (approximate frequency) and a Min-Heap of size K (top-K tracker). Results are flushed to Redis every 30 seconds.
  4. Cold Path (Batch): The same Kafka topic is also sinked to HDFS via a Kafka Connect connector. A nightly Spark batch job performs exact GROUP BY song_id aggregation, deduplicates, and writes official chart data to Venice (LinkedIn’s derived data store).
  5. Serving Layer: The API server reads from Redis for real-time queries and Venice for historical charts. For “today so far” queries, it merges last batch result + real-time delta.

Hot Path — Real-Time Stream Processing

Stream Pipeline: Kafka → Samza → Redis

The stream processor maintains in-memory data structures per partition:

  • Count-Min Sketch (CMS): A probabilistic data structure that estimates frequency of any song with bounded over-count error. Uses d=5 hash functions and w=10,000 buckets — only ~200 KB of memory per dimension.
  • Min-Heap (size K=50): Tracks the current top-50. On each event, query the CMS for the song’s estimated count; if it exceeds the heap minimum, replace and re-heapify. O(log K) per event.
  • Flush Cycle: Every 30 seconds, serialize the top-50 list and write to Redis with a TTL of 5 minutes. Clients always see data ≤ 30s stale.
// Pseudocode: Samza stream task for real-time Top-K
class TopKStreamTask implements StreamTask {
    CountMinSketch cms;       // d=5, w=10000
    MinHeap<SongCount> heap;  // capacity = K = 50

    void process(IncomingMessageEnvelope msg) {
        PlayEvent e = deserialize(msg);
        cms.add(e.songId);
        long estimatedCount = cms.estimate(e.songId);

        if (heap.size() < K) {
            heap.insert(new SongCount(e.songId, estimatedCount));
        } else if (estimatedCount > heap.peekMin().count) {
            heap.replaceMin(new SongCount(e.songId, estimatedCount));
        }
    }

    void onTimer() {  // every 30s
        redis.set("topk:global:realtime", heap.toSortedList(), TTL=300s);
    }
}

Cold Path — Batch Processing

Batch Pipeline: HDFS → Spark → Venice

The batch layer runs once daily (or hourly for faster convergence) and computes exact play counts:

  • Input: Raw play events in Parquet format on HDFS, partitioned by date.
  • Dedup: SELECT DISTINCT user_id, song_id, date to eliminate intra-day duplicates (a user playing the same song 100 times counts as 1 per day for official charts).
  • Aggregation: GROUP BY song_id, dimension with exact COUNT(*).
  • Top-K Extraction: ROW_NUMBER() OVER (PARTITION BY dimension ORDER BY count DESC) ≤ 50
  • Output: Results written to Venice as key-value pairs: dimension:value:period → [ranked song list].
-- Spark SQL: Daily Top-K per country
WITH deduped AS (
    SELECT DISTINCT user_id, song_id, country, date
    FROM play_events
    WHERE date = '2025-04-06'
),
counts AS (
    SELECT song_id, country, COUNT(*) AS play_count
    FROM deduped
    GROUP BY song_id, country
),
ranked AS (
    SELECT *, ROW_NUMBER() OVER (
        PARTITION BY country ORDER BY play_count DESC
    ) AS rank
    FROM counts
)
SELECT * FROM ranked WHERE rank <= 50;

Serving Layer & Merge Strategy

Merging Real-Time + Batch

For a query like “Top 50 in the US right now,” the serving layer performs:

  1. Fetch the last batch result from Venice (e.g., yesterday’s exact counts).
  2. Fetch the real-time delta from Redis (today’s approximate counts since midnight).
  3. Add the two: merged_count = batch_count + realtime_delta.
  4. Re-sort and return the top 50.

This gives us exact historical accuracy with near-real-time freshness. The approximation error from the CMS only affects the “today’s delta” portion, which is bounded by the CMS error guarantee: ε ≤ e/w ≈ 0.03% over-count.

Section 3

Approximate Counting Algorithms

With 100M unique songs and 1B+ events/day, maintaining exact per-song counters in the hot path would require massive memory and cross-partition coordination. Approximate counting algorithms let us estimate frequencies with bounded error using orders-of-magnitude less memory.

Count-Min Sketch (CMS)

How It Works

A Count-Min Sketch is a 2D array of counters with dimensions d × w, where d = number of independent hash functions and w = width (number of buckets per row).

        Bucket 0    Bucket 1    Bucket 2    ...    Bucket w-1
       +----------+----------+----------+-------+----------+
h1(x)  |     3    |     0    |    12    |  ...  |     7    |
       +----------+----------+----------+-------+----------+
h2(x)  |     5    |     8    |     0    |  ...  |     2    |
       +----------+----------+----------+-------+----------+
h3(x)  |     1    |     0    |     9    |  ...  |     4    |
       +----------+----------+----------+-------+----------+
h4(x)  |     0    |    11    |     3    |  ...  |     6    |
       +----------+----------+----------+-------+----------+
h5(x)  |     7    |     2    |     0    |  ...  |     1    |
       +----------+----------+----------+-------+----------+

Add(song_id): For each row i, compute h_i(song_id) mod w and increment that cell.

Estimate(song_id): Return min(counter[i][h_i(song_id)]) for i in 1..d. The minimum across rows eliminates most collision noise.

Guarantees:

  • Never under-counts (one-sided error — can only over-count).
  • Error bound: with probability 1 - δ, the estimate is at most ε · N above the true count, where ε = e/w and δ = e^(-d).
  • Space: O(d × w) counters. With d=5, w=10,000 and 4-byte counters = 200 KB.

HeavyKeeper

Optimized for Heavy Hitters

HeavyKeeper combines a Count-Min Sketch with fingerprint + decay to focus accuracy on the most frequent items (exactly what we need for Top-K):

  • Each cell stores (fingerprint, counter) instead of just a counter.
  • On collision (different item hashes to same cell), the existing counter is decayed with probability b^(-counter) (exponential decay), so low-frequency items are gradually evicted.
  • Heavy hitters survive because their high counters make decay exponentially unlikely.
  • Paired with a min-heap for top-K extraction — items that fall out of the sketch are also removed from the heap.

Advantage over CMS: Much higher accuracy for the top items (the ones we actually care about), at the cost of slightly worse accuracy for tail items (which we don’t need).

Space-Saving Algorithm

Deterministic Approximate Counting

Space-Saving maintains a fixed-size map of m (item, count) pairs:

  1. If the item already exists in the map, increment its count.
  2. If the item is new and the map is not full, insert it with count = 1.
  3. If the item is new and the map is full, find the item with the minimum count, replace it with the new item, and set the new item’s count to min_count + 1.

Guarantee: Any item with true frequency > N/m is guaranteed to be in the map. Error per item is at most N/m.

Drawback: Requires maintaining a sorted structure; harder to parallelize across partitions compared to CMS which is trivially mergeable.

Algorithm Comparison

Algorithm Space Error Type Mergeable? Best For
Count-Min Sketch O(d × w) — ~200 KB One-sided (over-count) Yes (cell-wise add) General frequency estimation
HeavyKeeper O(d × w) — ~400 KB One-sided, focused on top items Partially (fingerprint conflicts) Top-K heavy hitters specifically
Space-Saving O(m) entries — ~few KB Bounded two-sided No (requires full merge) Small K, single-node
Count-Min Sketch + Min-Heap O(d×w + K) — ~200 KB One-sided (over-count) Yes Distributed top-K (our choice)
Exact HashMap O(n) — ~800 MB for 100M songs None Yes (merge maps) Batch/cold path only

Why CMS + Min-Heap for the Hot Path?

We chose Count-Min Sketch + Min-Heap because: (1) CMS sketches are trivially mergeable — just add corresponding cells — making them ideal for distributed stream processing with multiple Samza partitions; (2) the one-sided error guarantee means we never miss a true top-K song (we might include a borderline song that doesn’t belong, but we never exclude one that does); (3) memory footprint is ~200 KB per dimension, allowing us to track dozens of dimensions per task instance.

Section 4

Multi-Dimensional Top-K

Users don’t just want “global top 50” — they want charts sliced by geography, genre, playlist context, and time. Each dimension requires its own set of counters.

Dimension Breakdown

Dimension Cardinality Counter Sets Memory (CMS per set) Total Memory
Global 1 1 200 KB 200 KB
Country ~200 200 200 KB ~40 MB
City ~5,000 (top cities) 5,000 100 KB (smaller w) ~500 MB
Genre ~50 50 200 KB ~10 MB
Playlist ~10,000 (curated) 10,000 50 KB (smaller w) ~500 MB
Time Period 3 (hourly/daily/weekly) 3 × other dims Shared with above ~3× overhead

Fan-Out Pattern

Each play event fans out to multiple counter sets. A single play of song X by a user in New York, USA on a Pop playlist updates:

  • global CMS — 1 update
  • country:US CMS — 1 update
  • city:new_york CMS — 1 update
  • genre:pop CMS — 1 update (per genre tag)
  • playlist:p_xyz CMS — 1 update

That’s ~5 CMS updates per event. At 50K events/sec peak, that’s 250K CMS increment operations/sec — trivially fast since each is just d=5 array increments in memory.

Storage & Partitioning

Partitioning Strategy

We partition the stream processing by dimension type:

  • Global + Country + Genre: Single Samza job (low cardinality, fits in one task’s memory).
  • City: Separate Samza job, partitioned by city hash. Each partition handles ~500 cities.
  • Playlist: Separate Samza job, partitioned by playlist ID. Curated playlists only (editorial team manages the list).

Redis keys follow the pattern: topk:{dimension}:{value}:{period}

topk:global::realtime          → [song1, song2, ..., song50]
topk:country:US:realtime       → [song1, song2, ..., song50]
topk:city:new_york:realtime    → [song1, song2, ..., song50]
topk:genre:pop:daily           → [song1, song2, ..., song50]
Section 5

Design Decisions & Trade-Offs

Approximate vs Exact Counting

Approximate (Hot Path)

  • 200 KB memory per dimension
  • Sub-second latency per event
  • No coordination between partitions
  • Mergeable across parallel instances
  • ≤ 0.03% over-count error

Exact (Cold Path)

  • ~800 MB for 100M song counters
  • Batch latency (hourly/daily)
  • Requires shuffle/reduce for aggregation
  • Guaranteed correct results
  • Required for official published charts

Why Both?

Approximate counting is perfect for “what’s trending right now?” — users don’t notice if the #47 song is actually #48. But for official weekly charts that determine royalty payouts, we need exact counts. The Lambda Architecture gives us both without compromise.

Lambda vs Kappa Architecture

Lambda (Our Choice)

  • Separate code paths for batch & stream
  • Batch corrects any stream approximation errors
  • Can use best tool for each path (Spark for batch, Samza for stream)
  • Tradeoff: two codebases to maintain

Kappa (Alternative)

  • Single stream processing path for everything
  • Replay Kafka log for historical recomputation
  • Simpler architecture, one codebase
  • Tradeoff: harder to guarantee exact counts; Kafka retention limits historical reprocessing

Why Lambda Wins Here

Top-K requires both approximate (fast) and exact (correct) results simultaneously. Kappa would need to maintain exact counts in the stream layer, which defeats the memory advantage of approximate algorithms. Lambda lets us use CMS in the hot path (200 KB) while Spark handles exact counts in batch (with disk-backed shuffles for 100M songs).

Time-Decay Weighting

Freshness vs Cumulative Popularity

A naive top-K counts all plays equally, so an old hit with millions of historical plays dominates forever. Time-decay weighting ensures charts reflect current trends:

weighted_count(song) = ∑ play_i × decay(now - timestamp_i)

// Exponential decay with half-life of 4 hours:
decay(age) = 2^(-age / half_life)

// A play from:
//   0 min ago  → weight 1.0
//   4 hrs ago  → weight 0.5
//   8 hrs ago  → weight 0.25
//   24 hrs ago → weight 0.016

Implementation: Instead of storing raw counts in the CMS, store weighted counts. On each event, increment by 1.0. Periodically (every hour), multiply all cells by the decay factor 2^(-1/half_life_hours). This is equivalent to exponential smoothing and prevents stale songs from lingering.

Windowed vs Decayed Counting

Approach Pros Cons
Tumbling Windows (e.g., last 1 hour) Simple; clear boundary; no drift Abrupt changes at window boundaries; a song can “fall off a cliff”
Sliding Windows (last 1 hour, updated every minute) Smoother transitions Expensive: must store per-minute counts and re-aggregate
Exponential Decay (our choice) Smooth; single counter per item; no window management Slightly harder to reason about; half-life tuning required

  • Samza Changelog Streams: CMS state is checkpointed to a compacted Kafka topic, enabling task migration and rebalancing without losing sketch data.
  • Venice Derived Data: Batch results are pushed to Venice using the Venice Push Job framework, supporting versioned snapshots and instant swapover between daily chart versions.

  • Pro-ML Feature Store: Play counts and chart positions are also exported as features for recommendation models, closing the feedback loop between popularity and discovery.

Common Interview Questions

Q1: How do you handle a new viral song that shoots up in 10 minutes?

The hot path (Samza + CMS) detects it within 30 seconds. The CMS count grows rapidly, pushing the song into the Min-Heap. Redis is updated on the next flush cycle (30s). Total detection latency: < 1 minute. The batch path will confirm the exact count in the next run.

Q2: What if a CMS collision causes a non-popular song to appear in Top 50?

This is the false-positive risk of CMS (it over-counts, never under-counts). Mitigation: (1) Use d=5 hash functions, which makes collision across all 5 rows astronomically unlikely for songs below 0.03% of total plays; (2) The batch path corrects any such errors within hours; (3) HeavyKeeper can replace CMS if collision rate is unacceptable for the hot path.

Q3: How do you scale to 200 countries × 5,000 cities?

Each dimension set is a separate CMS (~200 KB). For 200 countries = 40 MB; for 5,000 cities with reduced sketch width = 500 MB. This fits in a single Samza container (2–4 GB heap). For more dimensions, shard by dimension type across separate Samza jobs with their own Kafka consumer groups.

Q4: Why not just use a distributed counter (e.g., Redis ZINCRBY)?

At 50K events/sec peak, Redis ZINCRBY with 100M members in a sorted set would create enormous write amplification (O(log N) per update where N=100M). The CMS approach does O(d) = O(5) in-memory operations per event with no network I/O until the 30-second flush. Redis is only used for the final top-50 list, not per-event counting.

Q5: How do you prevent bot/fraud plays from inflating charts?

Three layers: (1) Dedup gate — same user + same song within 30 seconds is dropped; (2) Rate limiting — if a user generates > 100 play events/hour, flag and throttle; (3) Offline ML model — batch job identifies anomalous play patterns (e.g., 1000 accounts in the same IP range playing the same song) and removes fraudulent plays before official chart computation.

Q6: How do you handle multi-region deployment?

Each datacenter runs its own hot path (Samza + local CMS). Sketches from different datacenters are merged (CMS addition is commutative and associative) at the serving layer, or a single global Kafka cluster is used with regional consumers. For the cold path, all events are replicated to a single HDFS cluster for unified batch processing.
