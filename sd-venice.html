Section 01

What Is Venice?

Venice is LinkedIn's derived data platform that bridges the offline, nearline, and online worlds. It ingests precomputed datasets from batch pipelines (HDFS/Spark) and streaming sources (Kafka/Samza/Flink), stores them in RocksDB, and serves them with sub-millisecond latency to online applications.

Immutable Versions

Each batch push creates a new version. The entire dataset is atomically swapped from old to new — zero downtime, no partial states, instant rollback.

Dual Ingestion

Batch push from HDFS/Spark for full dataset refreshes. Streaming from Kafka for nearline updates. Hybrid stores combine both for freshness + completeness.

4 Client Types

Thin Client (remote, simple), Fast Client (1-hop, low latency), Da Vinci SSD (local, <1ms), Da Vinci RAM (in-process, <10µs for ML inference).

Key Properties

  • Immutable version swap: New versions are ingested in the background. Once ready, an atomic swap makes the new version live — zero-downtime with instant rollback to the previous version.
  • Batch + Streaming: Batch pushes provide the "ground truth" full dataset. Streaming (RT) writes overlay nearline updates on top. Hybrid stores merge both.
  • Multi-DC replication: Data is replicated across data centers via Kafka topics. Each DC has its own Venice cluster serving local reads.
  • Sub-millisecond reads: Da Vinci clients embed RocksDB directly in the application process — no network hops, no serialization overhead.
  • Schema evolution: Venice Controller runs a schema registry. Producers and consumers can evolve Avro schemas independently with compatibility checks.

Section 02

Core Components

Venice Controller

The Controller is the brain of Venice. It manages store lifecycle, version pushes, schema registry, and partition assignment. A Parent Controller coordinates Child Controllers across data centers for multi-DC replication.

Controller Responsibilities

  • Create/delete stores, manage store configurations and quotas
  • Orchestrate version pushes: create Kafka VT, track push status, trigger version swap
  • Schema registry: validate schema evolution, enforce compatibility rules
  • Parent Controller sends admin commands to Child Controllers via the Admin Topic

Venice Server (Storage Node)

Servers host RocksDB instances for their assigned partitions. Each partition is an independent RocksDB instance. A StoreIngestionTask thread consumes from the Kafka VT and RT topics and writes to the local RocksDB.

Server Internals

  • Helix assigns Leader/Follower pairs for each partition
  • Leader consumes from RT topic, Followers replicate from Leader or consume from VT
  • Each server hosts partitions from many stores — hundreds of RocksDB instances
  • Block Cache (16GB) is shared across all RocksDB instances on the server

Venice Router

The Router is a partition-aware read proxy for Thin Client requests. It looks up which server owns the requested partition, routes the request, and returns the result. For batch-get requests, it performs scatter-gather across multiple servers.

  • Routing metadata: Router maintains a mapping of store → partition → server via Helix External View
  • Scatter-gather: Multi-key batch-get requests are split by partition, sent in parallel, and merged
  • Retry & failover: If a server is slow or down, Router retries on the follower replica

Apache Kafka — Three Topic Types

Apache Helix + ZooKeeper

Venice uses Apache Helix for partition assignment and state management. Helix assigns each partition to a Leader and one or more Followers across servers. ZooKeeper stores partition metadata and provides leader election.

  • External View: Which servers own which partitions (used by Router for routing)
  • Customized View: Fine-grained partition state (LEADER, STANDBY, OFFLINE, etc.)
  • Metadata storage: Store configs, version metadata, push status in ZooKeeper znodes

Reads are local function calls — no network, no serialization, no Router. This is how LinkedIn serves ML features at <10µs latency for feed ranking models.
Section 03

Write Path — Batch Push vs Streaming

Venice has two completely different write paths. Understanding the difference is critical because they interact with RocksDB in fundamentally different ways.

Fig 2: Venice batch push (left) vs streaming write (right) paths, and how they interact with RocksDB

Path A: Batch Push (deferredWrite=true)

Batch push is Venice's flagship ingestion mode. It takes a sorted dataset from HDFS/Spark and bulk-loads it into RocksDB bypassing the MemTable and compaction entirely.

Batch Push Walkthrough

  1. Venice Push Job (VPJ) reads sorted key-value pairs from HDFS or Spark output.
  2. VPJ contacts Controller to create a new version (e.g., v4). Controller creates a new Kafka Version Topic venice_{store}_v4.
  3. VPJ writes sorted KV pairs to the Version Topic via VeniceWriter.
  4. Server's StoreIngestionTask consumes from the VT. Because data arrives sorted, it uses SstFileWriter to generate pre-sorted SST files directly — bypasses MemTable and compaction!
  5. Server calls ingestExternalFile() to bulk-load the SST files into RocksDB at the bottom level.
  6. For Hybrid stores: RT topic is replayed (from the configured rewind time) on top of the new version to apply recent streaming updates.
  7. Version swap: v4 becomes "current", v3 becomes "backup", v2 is deleted.
  8. Cleanup: Old version's RocksDB directory is deleted. Old Kafka VT is deleted after its retention period.
Why Batch Push Bypasses MemTable RocksDB's SstFileWriter creates SST files directly from sorted input without going through the write-ahead log or MemTable. Since VPJ data arrives pre-sorted, this avoids all write amplification from compaction. The SST files are ingested at the bottommost level via ingestExternalFile(), meaning they never need to be compacted down. This is dramatically faster than row-by-row put() calls.

Path B: Streaming / Real-Time

Streaming writes follow the traditional RocksDB write path through the MemTable and compaction pipeline.

Streaming Write Walkthrough

  1. Samza, Flink, or Online Producer writes key-value pairs to the store's RT topic in Kafka.
  2. StoreIngestionTask on the Leader server consumes from the RT topic.
  3. rocksDB.put(key, value) → writes to WAL (durability) + MemTable (32MB skiplist in RAM).
  4. MemTable full → becomes immutable → background flush thread writes it as an L0 SST file.
  5. Compaction merges L0 → L1 → L2 as levels fill up (standard leveled compaction).
  6. RT writes go to ALL existing versions — current, backup, and any future version if a push is in progress. This ensures version consistency on swap.

Section 04

Read Path

Venice reads follow RocksDB's LSM-tree lookup order. The read path short-circuits on the first match — once the key is found, the search stops immediately.

Read Order (First Match Wins)

Each layer is checked in order. The moment the key is found, RocksDB returns the value without checking deeper layers.

Step 1: Active MemTable (RAM)

The current write buffer. Contains the most recent writes. Fastest possible lookup — in-memory skiplist with O(log n) search. If the key was recently written, it's found here.

Step 2: Immutable MemTable(s) (RAM)

MemTables that are full and waiting to be flushed to disk. Still in RAM, still fast. Venice uses double-buffered MemTables (max 2), so at most 1 immutable MemTable exists.

Step 3: Block Cache (16GB Shared LRU)

An LRU cache shared across all RocksDB instances on the server. Caches recently-read SST blocks (16KB each). If the key's SST block was recently accessed by any store, it's in the cache. 16GB total across hundreds of RocksDB instances.

Step 4: Bloom Filter Check

Before reading any SST file from disk, RocksDB checks the file's Bloom filter. A Bloom filter can definitively say "key is NOT in this file" (no false negatives) but may say "key MIGHT be in this file" (possible false positive). Skips ~99% of SST files that don't contain the key.

Step 5: L0 SST Files (Disk)

L0 files are flushed MemTables. They may have overlapping key ranges, so RocksDB must check ALL L0 files. This is why too many L0 files hurts read performance. Venice's L0 compaction trigger limits this.

Step 6: L1 SST Files (Disk)

L1 files have non-overlapping key ranges. RocksDB uses binary search on file boundaries to find exactly 1 file that could contain the key. Only that file is read.

Step 7: L2...Ln SST Files (Disk)

Same as L1 — non-overlapping ranges, binary search, check at most 1 file per level. Each level is ~10x larger than the previous. Most data lives in the bottom levels.

Read Amplification Analysis

ScenarioLayers CheckedDisk I/ONotes
Key in MemTable 1 0 Best case — recent write, pure RAM lookup
Key in Block Cache 2-3 0 Recently read key, still in LRU cache
Key on disk (typical) 4-7 1 SST read Bloom filter skips most files, 1 file per level for L1+
Key doesn't exist All levels 0 (Bloom filter) Bloom filter says "not here" for every SST file
Section 05

RocksDB Compaction Deep Dive

Fig 3: RocksDB LSM-Tree architecture showing MemTable, L0-L2 levels, and the compaction merge process

LSM-Tree Architecture

RocksDB uses a Log-Structured Merge-Tree (LSM-Tree) that optimizes for write throughput by batching writes in memory and flushing them to disk as sorted files.

Write Flow Through the LSM-Tree

  1. put(key, value) → Write goes to the WAL (write-ahead log) for durability AND the MemTable (skiplist in RAM, 32MB).
  2. MemTable full (32MB) → Becomes an Immutable MemTable. A new active MemTable takes its place.
  3. Flush thread writes the Immutable MemTable to disk as a sorted L0 SST file.
  4. L0 accumulates files. L0 files can have overlapping key ranges because each is a snapshot of whatever was in memory.
  5. Compaction triggers when L0 has too many files (40 in Venice normal mode). Compaction merges L0 files with overlapping L1 files.
  6. Cascading compaction: If L1 grows beyond its size limit (2GB), excess files are compacted into L2. L2 overflows to L3, etc. Each level is ~10x larger.

Leveled Compaction (Default)

Venice uses Leveled Compaction, which is RocksDB's default and provides the best read performance.

How Leveled Compaction Works

  1. When L0 has too many files, pick L0 files + overlapping L1 files
  2. Sorted merge: combine all selected files, sort by key, keep latest version per key, drop tombstones (if at bottom level)
  3. Write new non-overlapping SST files to L1
  4. Delete the old input files
  5. If L1 exceeds its size target, repeat for L1 → L2, etc.

Key Invariant

L1 and above have non-overlapping key ranges within each level. This means a point lookup needs to check at most 1 file per level (after L0). This is why leveled compaction gives the best read amplification.

L0 is the exception — files can overlap, so all L0 files must be checked on reads.

Three Amplification Tradeoffs

Write Amplification (~10x)

Each byte written to RocksDB may be rewritten ~10x as it's compacted through levels. A 1GB MemTable flush can cause 10GB of total disk writes over time as data moves L0 → L1 → L2.

Read Amplification (1 file/level)

With leveled compaction, a point lookup checks at most 1 SST file per level (plus all L0 files). Bloom filters eliminate most negative lookups. Best-case: 1 disk read.

Space Amplification (~10%)

Obsolete versions of keys and tombstones consume space until compaction removes them. Leveled compaction keeps this tight — typically <10% overhead.

Other Compaction Styles

Universal (Size-Tiered) Merge all files of similar size together Better write amp, worse read/space amp Not used

Venice Batch Push = Zero Compaction Batch push uses SstFileWriter + ingestExternalFile() to load pre-sorted SST files directly at the bottom level. These files never go through the MemTable, never go through L0, and never need compaction. Write amplification is exactly 1x. This is why batch push is dramatically faster and cheaper than streaming writes for full dataset refreshes.

Backup Strategy

Fig 4: Venice backup flow — RocksDB checkpoint, SST hard-links, upload to GPFS and Ambry

How Venice Creates Backups

Backup Process

  1. Checkpoint.create(rocksDB).createCheckpoint(snapshotDir) — Creates hard links to all current SST files in a snapshot directory. This is instant (just filesystem metadata operations) and non-blocking (reads and writes continue normally).
  2. Record Kafka offset in OffsetRecord metadata — captures the exact position in VT/RT topics at the time of the snapshot.
  3. Upload SST files from the snapshot directory to GPFS (local fast storage) or Ambry (cross-DC blob store).

Combined Backup Cost For the complete Venice fleet of 4.64 PB: ~$149K/month provides both fast local recovery (GPFS, 5 versions) and cross-DC disaster recovery (Ambry, 1 snapshot). GPFS handles 99% of restores; Ambry is the insurance policy.

Incremental Backup

SST files in RocksDB are immutable — once written, they never change. This enables efficient incremental backups:

How Incremental Backup Works

  • Each backup records the set of SST file names and checksums
  • On the next backup, only new SST files (created since last backup) are uploaded
  • Old SST files are referenced by name — no re-upload needed
  • Backup metadata stores the file list + Kafka offset

Incremental Savings by Store Type

  • Batch-only stores: NO savings. Each version = 100% new SSTs (loaded via ingestExternalFile). The entire dataset is replaced.
  • Hybrid stores: 30-60% SST reuse between backups. Only compacted/flushed files since last backup are new.

BackupEngine vs Manual Backup

MethodProcessRestores FromData Completeness
BackupEngine API Flush MemTable first, then backup SST files only (no WAL needed) SST files alone Complete — MemTable was flushed before backup
Manual copy Copy SST files + WAL files SST + WAL replay Complete — WAL replayed on restore to recover MemTable data

Can You Restore from SST Files Alone?

NO — You Lose MemTable Data

SST files only contain data that has been flushed from the MemTable. Any writes still in the active MemTable are not in any SST file. To get complete data:

  • Formula: SST files + WAL = complete data
  • Or: Use BackupEngine which flushes the MemTable before backup, making the backup purely SST-based (no WAL needed)
  • Safe overlap: Data can exist in both WAL and SSTs. RocksDB uses sequence numbers to resolve — replaying WAL entries that are already in SSTs is a no-op.
Section 07

Data Deletes & Retention

Version Retention

Venice keeps a configurable number of versions per store. The default is numVersionsToPreserve = 2: one "current" version serving reads, and one "backup" version for instant rollback.

Version Lifecycle Example

  1. Before push: v3 = current (serving reads), v2 = backup
  2. New version v4 pushed: v4 is built in the background while v3 continues serving
  3. Push complete, version swap: v4 = current, v3 = backup, v2 = DELETE
  4. v2 deletion: rm -rf the entire RocksDB directory for v2 + delete the Kafka VT venice_{store}_v2 after its retention period

Data Deletes (GDPR / Compliance)

Streaming / Hybrid Stores

  • put(key, DELETE_MARKER) writes a tombstone to the RT topic
  • Tombstone is consumed by StoreIngestionTask and written to RocksDB
  • In RocksDB, tombstone persists until compaction merges it through all levels
  • At the bottommost level, compaction drops the tombstone and the original key — data is physically deleted
  • Timeline: depends on compaction frequency, could be hours to days

Batch-Only Stores

  • Next batch push replaces the entire version
  • Old keys simply don't exist in the new version — they're gone
  • No tombstones needed; deletion is implicit by omission
  • Old version's RocksDB directory is rm -rf'd on version retire
  • GDPR compliance: ensure the batch pipeline excludes deleted records
LinkedIn Data Deletion Workflow LinkedIn follows structured data deletion workflows per go/data-deletion. For Venice, the process depends on store type: batch-only stores are naturally purged on the next push, while hybrid stores require explicit delete markers that propagate through compaction.

Section 08

Restore from Backup

Restoring a Venice partition from backup involves downloading the snapshot, opening RocksDB, and catching up from Kafka to the present.

Step-by-Step Restore Process

  1. Download base snapshot (SST files + MANIFEST) from GPFS or Ambry.
    • GPFS: instant — files are on a local POSIX mount
    • Ambry: minutes to hours — depends on dataset size and network bandwidth
  2. Open RocksDB on the restored files. RocksDB reads the MANIFEST to understand the level structure and SST file layout.
  3. Read stored Kafka offset from backup metadata (OffsetRecord). This tells the server exactly where it left off in the VT and RT topics.
  4. Start consuming VT + RT from the recorded offset to the current latest offset in Kafka.
  5. Apply all Kafka messages (catch-up phase). Each message is written to RocksDB via put(). This replays everything that happened after the backup was taken.
  6. Once fully caught up (consumer offset = latest offset), mark the partition as READY and begin serving reads.

Example: Restoring a 4TB Store from 7-Day-Old Backup
  1. Download 4TB of SST files from Ambry (assuming 10 Gbps: ~53 minutes)
  2. Open RocksDB on downloaded files (~1 second)
  3. Replay the full Version Topic (4TB of sorted data, bulk-loaded via SstFileWriter)
  4. Replay 7 days of RT writes (volume depends on write rate)
  5. Total: 1-4 hours depending on network and write volume

Why Backup Freshness Matters

Fresh Backup (hours old)

Minimal Kafka catch-up needed. Restore completes in seconds (GPFS) or minutes (Ambry). The server quickly reaches READY state and begins serving.

Stale Backup (days old)

Must replay days of Kafka messages. For high-write stores, this can mean replaying terabytes of data. Restore can take hours, during which the partition is unavailable or served by other replicas.
