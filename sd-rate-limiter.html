Section 1

Overview & Requirements

What Is Rate Limiting?

A rate limiter controls the number of requests a client can send to a server within a defined time window. When the threshold is exceeded, excess requests are throttled — typically returning HTTP 429 Too Many Requests.

Why Rate Limiting Matters:
  • Prevent resource starvation — one abusive client can't starve others
  • Security — mitigate brute-force attacks, credential stuffing, DDoS
  • Cost control — prevent runaway API usage from inflating infra costs
  • Ensure SLA compliance — guarantee fair bandwidth across tenants
  • Stability — protect downstream services from cascading overloads

Functional & Non-Functional Requirements

Functional Requirements

  • Limit requests by client ID, IP, API key, or user ID
  • Support configurable limits per endpoint/service
  • Return clear error responses (429) with retry info
  • Support multiple rate tiers (free, premium, enterprise)
  • Allow soft limits (warn) vs hard limits (reject)

Non-Functional Requirements

  • Low latency — must not add significant overhead (< 1ms p99)
  • High availability — rate limiter failure should not block all requests
  • Distributed — consistent enforcement across multiple servers
  • Accuracy — minimal over-counting or under-counting
  • Fault tolerance — graceful degradation if Redis/store is unavailable

Where to Place the Rate Limiter

The rate limiter can live at different layers in the request path. Each has trade-offs:

Placement Pros Cons Best For
Client-side Reduces unnecessary network traffic Easily bypassed; unreliable Mobile SDK throttling
API Gateway Centralized; uniform enforcement; offloads services Single point of failure; coarse-grained Most production systems
Middleware / Sidecar Per-service granularity; service mesh integration Each service needs its own config Microservice architectures
Application Layer Fine-grained; business-logic-aware rules Duplicated logic across services Custom per-endpoint limits
LinkedIn Parallel

LinkedIn uses API gateway-level rate limiting (via the Rest.li framework and Frontdoor proxy) combined with per-service middleware for fine-grained control. The gateway handles global IP-based throttling, while individual services enforce user-tier-based limits.

Section 2

Rate Limiting Algorithms

Five major algorithms dominate rate limiter design. Each balances accuracy, memory, and implementation complexity differently.

Figure 1 — Visual comparison of the five rate limiting algorithms

Diagram Walkthrough

  1. Token Bucket — tokens are added at a fixed rate; each request consumes one token. When the bucket is empty, requests are rejected. Allows controlled bursts up to bucket capacity.
  2. Leaky Bucket — requests enter a FIFO queue and are processed at a fixed rate. Excess requests overflow and are dropped. Produces perfectly smooth output regardless of input burstiness.
  3. Fixed Window Counter — divides time into fixed intervals (e.g., 60s windows). A counter increments per request; resets at each window boundary. Simple but vulnerable to boundary spikes.
  4. Sliding Window Log — stores timestamps of every request. Counts requests within the trailing window. Most accurate but highest memory cost — O(n) per user.
  5. Sliding Window Counter — hybrid of fixed window + sliding window. Estimates current window count using weighted average of current and previous windows. Great accuracy-to-memory trade-off.

Token Bucket

The token bucket is the most widely used algorithm. Tokens are added at rate r tokens/second, up to a maximum bucket size b. Each request removes one token.

class TokenBucket:
    def __init__(self, capacity, refill_rate):
        self.capacity = capacity
        self.tokens = capacity
        self.refill_rate = refill_rate   # tokens per second
        self.last_refill = time.time()

    def allow_request(self):
        self._refill()
        if self.tokens >= 1:
            self.tokens -= 1
            return True
        return False

    def _refill(self):
        now = time.time()
        elapsed = now - self.last_refill
        self.tokens = min(self.capacity,
                          self.tokens + elapsed * self.refill_rate)
        self.last_refill = now

Memory: O(1) per bucket — only stores tokens and last_refill.

Used by: Amazon API Gateway, Stripe, LinkedIn Rest.li

Leaky Bucket

Functionally similar to a token bucket, but implemented as a FIFO queue. Requests are processed at a fixed rate. If the queue is full, new requests are dropped.

Fixed Window Counter

Divides time into fixed windows (e.g., every 60 seconds). A counter tracks requests in the current window and resets at each boundary.

The Boundary Problem: A user could send the maximum number of requests at the end of window N and again at the start of window N+1 — effectively doubling their rate across a short burst that spans the boundary.

# Example: limit = 100 req/min
# At 12:00:59 → user sends 100 requests (allowed, counter = 100)
# At 12:01:00 → counter resets to 0
# At 12:01:01 → user sends 100 requests (allowed, counter = 100)
# Result: 200 requests in 2 seconds!

Sliding Window Log

Maintains a sorted set of timestamps for each request. On every new request, remove all entries older than now - window_size, then check if count exceeds the limit.

# Redis implementation
def is_allowed(user_id, limit, window_secs):
    key = f"rate:{user_id}"
    now = time.time()
    cutoff = now - window_secs

    pipe = redis.pipeline()
    pipe.zremrangebyscore(key, 0, cutoff)  # Remove old entries
    pipe.zadd(key, {str(now): now})        # Add current request
    pipe.zcard(key)                        # Count entries
    pipe.expire(key, window_secs)
    _, _, count, _ = pipe.execute()

    return count <= limit

Accuracy: Perfect — no boundary issues. Memory: O(n) per user, where n = number of requests in the window. Expensive for high-volume users.

Sliding Window Counter

The best trade-off for most systems. Uses counters from the current and previous fixed windows, weighted by how far into the current window we are.

# Formula:
# estimated_count = prev_window_count × overlap_ratio + current_window_count
#
# Example: window = 60s, limit = 100
# Previous window (12:00-12:01): 84 requests
# Current window  (12:01-12:02): 36 requests
# Current time:    12:01:15 → 75% of prev window overlaps
#
# estimated = 84 × 0.75 + 36 = 63 + 36 = 99 → ALLOWED (under 100)

Memory: O(1) per user — only two counters. Accuracy: ~99.7% according to Cloudflare’s analysis.

Algorithm Comparison

Algorithm Memory Accuracy Burst Control Complexity
Token Bucket O(1) Good Allows controlled bursts Low
Leaky Bucket O(1) Good Smooths all bursts Low
Fixed Window O(1) Poor at boundaries None Very Low
Sliding Window Log O(n) Perfect Precise Medium
Sliding Window Counter O(1) ~99.7% Good Low
Section 3

Distributed Architecture

In a multi-server environment, a centralized data store (typically Redis) is required so that all API servers enforce a shared counter. Without this, each server would track limits independently, letting users exceed limits by distributing requests across servers.

Figure 2 — Distributed rate limiter architecture with Redis, rules engine, and response flow

Architecture Walkthrough

  1. Client Request — arrives at the API gateway or load balancer, which extracts the client identifier (API key, user ID, IP address).
  2. Rules Engine — looks up the rate limit configuration for this client/endpoint pair from a rules cache (backed by a config service or database). Rules define limits like “100 req/min for /api/search, 1000 req/min for /api/feed”.
  3. Redis Check — the rate limiter issues an atomic INCR + EXPIRE command to Redis. If the returned counter exceeds the limit, the request is rejected with 429.
  4. Allow / Deny — if under the limit, the request is forwarded to the backend service. If over, a 429 response is returned with rate limit headers.
  5. Async Logging — throttled requests are logged asynchronously for analytics, abuse detection, and alerting dashboards.

Redis Counters & Atomic Operations

Redis is the de facto choice for distributed rate limiting due to its single-threaded model and sub-millisecond latency. The core operation is atomic increment with expiry:

-- Lua script for atomic rate limiting in Redis
local key = KEYS[1]
local limit = tonumber(ARGV[1])
local window = tonumber(ARGV[2])

local current = redis.call('INCR', key)
if current == 1 then
    redis.call('EXPIRE', key, window)
end

if current > limit then
    return 0  -- rejected
else
    return 1  -- allowed
end

Using a Lua script ensures atomicity — INCR and EXPIRE execute as a single operation, preventing the key from existing without a TTL (which would cause a memory leak).

Race Conditions

The GET-then-SET Race

A naive implementation that reads the counter, checks the limit, and then writes is vulnerable to race conditions:

# WRONG - Race condition between GET and SET
count = redis.get(key)           # Thread A reads 99
                                  # Thread B reads 99
if count < limit:
    redis.incr(key)              # Thread A writes 100 → allowed
                                  # Thread B writes 101 → should have been denied!

Solutions:

  • Lua scripts (recommended) — execute check + increment atomically on the Redis server
  • Redis sorted sets — use ZADD/ZCARD for sliding window log with implicit atomicity
  • MULTI/EXEC transactions — pipeline commands, but doesn't provide conditional logic
  • Optimistic locking with WATCH — retry on conflict, adds complexity and latency under contention

Section 4

Design Decisions & Trade-offs

Gateway vs Service-Level Limiting

Gateway-Level (Centralized)

  • Single enforcement point — easier to manage
  • Protects all backend services uniformly
  • No code changes required in services
  • Can block requests before they consume backend resources

Risk: Single point of failure; coarse rules may over-throttle

Service-Level (Distributed)

  • Fine-grained per-endpoint control
  • Business-logic-aware (e.g., different limits for search vs write)
  • Each service owns its rate limit policy
  • No single point of failure

Risk: Inconsistent policies; harder to get global view

Recommendation: Use both. The gateway handles broad abuse prevention (IP-based, DDoS) while individual services enforce business-level limits (per-user, per-endpoint, per-tier).

Hard Limits vs Soft Limits

Type Behavior Use Case
Hard Limit Request is immediately rejected with 429 External APIs, untrusted clients, security-sensitive endpoints
Soft Limit Request is allowed but flagged; alert is triggered; may be degraded Internal services, gradual rollout, observability
Elastic Limit Temporary burst allowed up to N% over quota; hard stop after that Handling legitimate traffic spikes without user disruption

Graceful Degradation

What happens when Redis goes down?

If the rate limiter’s backing store is unavailable, you have two choices:

Fail Open (Allow All)

If Redis is down, allow all requests. Prioritize availability over protection.

Best for: internal services, non-critical APIs where downtime is worse than over-serving.

Fail Closed (Deny All)

If Redis is down, reject all requests. Prioritize safety over availability.

Best for: payment APIs, auth endpoints, security-critical paths.

Hybrid approach: Fall back to local in-memory counters (per server) with approximate limits. This provides degraded-but-functional rate limiting without a centralized store. Sync counters when Redis recovers.

Key Takeaway

No single algorithm wins everywhere. Stripe and LinkedIn pick token bucket for burst tolerance on payment/API platforms. GitHub uses fixed window for simplicity with large windows (1 hour). Cloudflare pioneered sliding window counter at edge scale where memory matters. Twitter uses sliding windows for tighter per-user fairness.

Section 6

Interview Questions

Q1: How would you implement distributed rate limiting across multiple data centers?

Key points:

  • Option A — Global Redis cluster: Single source of truth with cross-DC replication. Consistent but adds latency for remote DCs. Use Redis Cluster with read replicas in each DC.
  • Option B — Local counters with sync: Each DC maintains its own counter; periodically synchronize via gossip or event stream. Faster locally but may briefly over-serve during sync gaps.
  • Option C — Split quota: Divide the total limit across DCs proportionally (e.g., 3 DCs → each gets 1/3 of the quota). Simple but inflexible if traffic is unevenly distributed.
  • Best practice: Combine local enforcement with async global sync. Accept slight inaccuracy (e.g., 5% over-serve) in exchange for low latency. Alert if global aggregate exceeds hard limits.

Q2: How do you handle race conditions in a distributed rate limiter?

Key points:

  • Never use GET-then-SET — always use atomic operations
  • Redis Lua scripts execute atomically on the server side; this is the standard solution
  • For sorted-set-based sliding windows, use MULTI/EXEC pipelines
  • If using a database instead of Redis, use UPDATE ... SET count = count + 1 WHERE count < limit (row-level locking)
  • For extremely high throughput, consider probabilistic data structures (Count-Min Sketch) that trade perfect accuracy for zero contention

Q3: What happens if Redis fails? How does the rate limiter degrade?

Key points:

  • Fail-open vs fail-closed — depends on the criticality of the protected endpoint (see Section 4)
  • Local fallback: Each server keeps an in-memory token bucket scoped to that instance. Limits are divided by number of servers (e.g., 100/min global → 10/min per server with 10 servers)
  • Circuit breaker: Stop querying Redis after N consecutive failures; periodically probe to check recovery
  • Redis Sentinel / Cluster: Automatic failover to replica; most outages are short (< 30s) and handled by Sentinel promotion
  • Monitoring: Alert on Redis latency p99, connection pool exhaustion, and replication lag. These are leading indicators of rate limiter degradation.

Q4: How would you rate-limit by multiple dimensions simultaneously?

Key points:

  • Use composite keys: rate:{user_id}:{endpoint}:{window}
  • Check all applicable limits in a single Lua script or pipeline
  • A request is allowed only if ALL limits pass
  • Return the most restrictive remaining count in headers
  • Consider hierarchical limits: org → team → user → endpoint

Q5: Token Bucket vs Sliding Window Counter — when would you choose each?

Key points:

  • Token Bucket: When you want to allow controlled bursts (e.g., APIs where a client might batch requests). Two parameters (rate + capacity) provide intuitive tuning.
  • Sliding Window Counter: When you need smooth, consistent enforcement without burst tolerance. Better for user-facing rate limits where fairness matters more than burst flexibility.
  • Both are O(1) memory. Token bucket is simpler to implement; sliding window counter is more accurate near window boundaries.
  • For interviews, mention that most production systems use token bucket due to its simplicity and natural burst handling.

Q6: How would you design rate limiting for a multi-tenant SaaS platform?

Key points:

  • Define rate limit tiers in a configuration service (free: 100/hr, pro: 10K/hr, enterprise: custom)
  • Use tenant ID as the primary key in Redis, with per-endpoint sub-limits
  • Support dynamic tier upgrades without restart (config hot-reload via pub/sub)
  • Implement quota reservation for enterprise tenants — guaranteed minimum capacity
  • Provide a self-service dashboard showing usage vs limits with historical trends
  • Consider cost-based limiting for expensive operations (e.g., 1 search = 10 “points”, 1 read = 1 point)
