Online & Offline Delivery Flows

Online Delivery (recipient is connected)

  1. Sender's client encrypts the message with the recipient's session key and sends it over its WebSocket connection to Gateway A.
  2. Gateway A forwards the message to the Chat Service, which persists it to the Message Store (write-ahead) and returns a sent ✓ acknowledgment.
  3. Chat Service queries the Connection Registry to find the recipient's active gateway.
  4. Registry returns Gateway B. Chat Service pushes the message to Gateway B.
  5. Gateway B delivers the message over the recipient's WebSocket. Recipient sends a delivered ✓✓ ack back through the same path.
  6. Chat Service updates the delivery state and notifies the sender's gateway, which pushes the delivered ✓✓ tick to the sender.

Offline Delivery (recipient is disconnected)

  1. Sender's client sends the encrypted message to Gateway A.
  2. Gateway A forwards to the Chat Service. Message is persisted → sender gets sent ✓.
  3. Chat Service queries the Connection Registry — no entry found for recipient.
  4. Message stays in the Message Store in the recipient's offline queue.
  5. Chat Service fires a push notification (APNs / FCM) with a lightweight payload (sender name, no content due to E2E encryption).
  6. When the recipient reconnects, their gateway issues a sync request. Chat Service drains the offline queue and delivers all pending messages in order.
  7. Recipient acks each message → delivered ✓✓ flows back to sender.

Every message carries a unique message_id (UUID v7 with embedded timestamp) for idempotent delivery. If a client receives a duplicate, it silently discards it. The Chat Service also deduplicates within a sliding window using a Bloom filter backed by a hash set.

Offline Path & Sync

Offline Queue Design

  • Per-user queue in Message Store, ordered by message_id timestamp.
  • TTL — Messages expire after 30 days if undelivered.
  • Batch sync — On reconnect, client sends its last-seen message_id. Server streams all subsequent messages in order.
  • Push notification — Lightweight notification via APNs/FCM. Cannot include message body (E2E encrypted). Contains sender ID and conversation ID only.
  • Multi-device — Each device maintains its own sync cursor. Message is only deleted from server after all linked devices ack.

Group Messaging

Fan-Out Strategies

When a user sends a message to a group of N members, the Chat Service must deliver the message to all N−1 other members. The fan-out strategy varies by group size:

Small Groups (≤ 256 members)

  • In-memory fan-out — Group Service keeps membership list in cache
  • Chat Service iterates members, queries Connection Registry for each
  • Online members get immediate delivery; offline members get queued
  • Latency: < 50ms for all online members
  • Used for ~95% of all groups

Large Groups (> 256 members)

  • Queued fan-out — Message placed in a fan-out queue
  • Worker pool processes queue, delivering in batches of 50
  • Prevents a single large group from starving other messages
  • Latency: < 500ms p99 for all members
  • Back-pressure and rate limiting per group

Message Ordering

Maintaining consistent message order across distributed servers is challenging. WhatsApp uses a combination of techniques:

TechniquePurposeDetails
Lamport Timestamps Logical ordering across servers Each Chat Service node maintains a logical clock. On send, timestamp = max(local, received) + 1. Provides causal ordering without synchronized physical clocks.
Sender Sequence Number Per-sender FIFO Each client attaches a monotonically increasing sequence number per conversation. Receiver buffers and reorders if messages arrive out of sequence.
Server Timestamp Tie-breaking Chat Service stamps each message with a server-side NTP-synchronized timestamp for display ordering when logical timestamps are equal.

Why Lamport Timestamps?

  1. Physical clock synchronization across thousands of servers is impractical at sub-millisecond precision.
  2. Lamport timestamps guarantee causal ordering: if message A causally precedes B, then L(A) < L(B).
  3. They are lightweight — a single integer incremented on each event.
  4. Combined with sender sequence numbers, they provide total ordering within a conversation with minimal coordination overhead.

End-to-End Encryption

Signal Protocol Overview

WhatsApp uses the Signal Protocol (formerly TextSecure Protocol) to provide end-to-end encryption for all messages. The server never has access to plaintext content or encryption keys.

Key Principle: Server is Untrusted

The server stores and forwards only ciphertext. Even if the server is compromised, message content remains confidential. Encryption and decryption happen exclusively on client devices. The server facilitates key exchange but never sees the shared secret.

PrimitiveUsageAlgorithm

Key Exchange & Ratcheting

Extended Triple Diffie-Hellman (X3DH) Key Exchange

  1. Registration: Each device generates an Identity Key pair, a Signed Pre-Key pair, and a batch of One-Time Pre-Key pairs. Public keys are uploaded to the server.
  2. Session initiation: Alice wants to message Bob. She fetches Bob's key bundle (Identity Key, Signed Pre-Key, one One-Time Pre-Key) from the server.
  3. X3DH computation: Alice performs three (or four) Diffie-Hellman computations using her keys and Bob's bundle to derive a shared secret S.
  4. Root key: The shared secret S is fed into HKDF to produce the initial Root Key and Chain Key.
  5. First message: Alice encrypts her first message using a Message Key derived from the Chain Key. She attaches her Identity Key and ephemeral key so Bob can perform the same X3DH computation.

Double Ratchet Algorithm

After the initial key exchange, the Double Ratchet provides forward secrecy and break-in recovery:

  • Symmetric ratchet (Chain Key) — After each message, the Chain Key is ratcheted: CK_new = HMAC(CK, 0x02). The Message Key is derived: MK = HMAC(CK, 0x01). Old Chain Keys are deleted.
  • DH ratchet — On each reply, the sender generates a new ephemeral DH key pair and includes the public key. Both parties perform a new DH exchange to derive a new Root Key and Chain Key. This provides break-in recovery: even if current keys are compromised, future messages are secure.
  • Forward secrecy — Past Message Keys cannot be derived from current state because the ratchet is one-way (HMAC). Compromising the current state does not reveal past messages.

Group Encryption

Groups use the Sender Keys variant of the Signal Protocol:

  • Each member generates a Sender Key and distributes it to all group members via pairwise E2E-encrypted channels.
  • Messages are encrypted once with the Sender Key (symmetric), rather than N times for each member.
  • When a member leaves, all remaining members generate new Sender Keys to ensure the departing member cannot decrypt future messages.
  • Trade-off: more efficient than pairwise encryption (O(1) vs O(N) encryptions per message) but requires key redistribution on membership changes.

System Design Interview Questions

Group Scaling

Q: How would you scale group messaging to support groups with 100,000+ members?

  • Use topic-based fan-out via a message broker (Kafka). Each large group maps to a topic partition.
  • Consumers pull messages at their own rate, preventing thundering herd on the Chat Service.
  • Introduce a fan-out service that processes group messages asynchronously with configurable parallelism.
  • For read-heavy large groups (e.g., announcement channels), switch to fan-out on read: store a single copy and let each member's client pull from the group timeline.
  • Shard the Connection Registry by user ID range so lookups during fan-out can be parallelized.

End-to-End Encryption Challenges

Q: What are the trade-offs of E2E encryption at scale?

  • Server-side search is impossible — The server cannot index or search message content. Search must be client-side only.
  • Spam/abuse detection is harder — Content moderation cannot inspect ciphertext. Must rely on metadata analysis, user reports, and on-device classifiers.
  • Multi-device sync complexity — Each device needs its own encryption session. Adding a new device requires re-establishing keys with all contacts.
  • Key management at scale — With 2B users, the key server must handle billions of key bundle fetches and one-time pre-key replenishments.
  • Group key rotation — When a member leaves, all sender keys must rotate. For large groups this creates O(N²) encrypted messages for key distribution.

Message Ordering

Q: How do you guarantee message ordering in a distributed system?

  • Per-sender ordering is achievable with sender-side monotonic sequence numbers. The receiver buffers and reorders.
  • Global ordering is expensive (requires consensus like Raft/Paxos). WhatsApp avoids it — only per-conversation ordering is guaranteed.
  • Lamport timestamps provide causal ordering without clock synchronization. If A → B (A causally precedes B), then L(A) < L(B).
  • Vector clocks are more precise but have O(N) space overhead per message (N = number of participants). Impractical for large groups.
  • In practice, the client uses (Lamport timestamp, server timestamp, sender sequence number) as a composite sort key for display ordering.
