Section 01

Overview & Scale Requirements

A realtime monitoring system observes infrastructure and application health by collecting metrics, storing them in a time-series database, evaluating alerting rules, and rendering dashboards. The four pillars are:

Four Pillars of Monitoring
  1. Metrics Collection — Agents on every host scrape or receive metrics (CPU, memory, request latency, error rates) and forward them to a central pipeline.
  2. Time-Series Storage — A purpose-built database that stores (metric, labels, timestamp, value) tuples with extreme write throughput and compression.
  3. Alerting Engine — Continuously evaluates rules against live data, fires alerts, de-duplicates, groups, and routes them to the right on-call engineer.
  4. Dashboard & Query — Interactive dashboards let engineers visualize trends, drill down into anomalies, and correlate signals across services.

Section 02

High-Level Architecture

The system is organized as a pipeline: data sources emit metrics, agents collect and forward them through a Kafka buffer, stream processors fan out to the time-series database, alerting engine, and dashboard query layer.

Fig 1 — End-to-end architecture: from data sources to dashboards and alerts

Component Walkthrough

  1. Data Sources: Application servers, databases, Kubernetes pods, network devices, and custom instrumented services emit raw metrics via StatsD, OpenTelemetry, or Prometheus exposition format.
  2. Metrics Agents: Lightweight per-host agents (e.g., Telegraf, node_exporter, or custom agents) scrape local metrics, add host-level labels (hostname, datacenter, pool), batch samples, and push to Kafka.
  3. Kafka Buffer: A Kafka cluster acts as the durable, ordered, partitioned buffer between producers and consumers. Partitioning by metric name or hash ensures write locality. Kafka absorbs ingest spikes without backpressuring agents.
  4. Stream Processor: Samza/Flink consumers read from Kafka, perform pre-aggregation (e.g., per-minute rollups), label enrichment, and fan-out writes to the TSDB and the alerting engine.
  5. Time-Series DB (TSDB): Purpose-built storage (e.g., InfluxDB, Thanos, Cortex, or custom) optimized for append-only writes, time-range scans, and heavy compression. Sharded by metric hash across nodes.
  6. Dashboard / Query Layer: A query frontend (like Grafana) translates user queries into TSDB range reads, applies aggregation functions (rate, avg, percentile), and renders charts with sub-second latency.
  7. Alert Engine: A separate stateful service evaluates alerting rules against the latest data, manages alert state machines (pending → firing → resolved), and dispatches notifications through configured channels.

Push vs Pull Collection

Pull Model (Prometheus-style)

  • Central server scrapes /metrics endpoints
  • Server controls scrape interval and targets
  • Easy to detect if a target is down (scrape fails)
  • Simpler security: server initiates connections
  • Challenge: doesn't work well behind firewalls or with short-lived jobs

Push Model (StatsD-style)

  • Agents push metrics to a central gateway / Kafka
  • Works with ephemeral / serverless / batch jobs
  • Easier to scale: adding sources doesn't burden a scraper
  • Better for high-cardinality, event-driven metrics
  • Challenge: harder to detect silent failures (no push = dead or idle?)
Hybrid Approach: Most production systems use both. Pull for long-lived services (easy health detection), push for ephemeral jobs and edge devices. A Pushgateway bridges the two: short-lived jobs push to it, the monitoring server pulls from it.
Section 03

Data Model

Metric Format & Label Cardinality

Metrics follow the metric_name{label=value, ...} timestamp value format. Each unique combination of metric name + label set creates a distinct time series.

http_requests_total{method="GET", handler="/api/users", status="200"}  1681234567  42871
http_requests_total{method="POST", handler="/api/users", status="201"} 1681234567  1203
node_cpu_seconds_total{cpu="0", mode="idle", instance="host-42"}       1681234567  98123.45

The Cardinality Explosion Problem

Cardinality = product of all label values. A metric with 5 labels, each having 100 unique values, creates 1005 = 10 billion potential time series. Real-world culprits:

Bad LabelWhy It's DangerousFix
user_idMillions of unique users = millions of seriesUse histograms/logs instead
request_idUnique per request → unboundedNever use as a metric label
emailPII + unbounded cardinalityUse aggregated cohorts
url_path (raw)/users/123 vs /users/456 → explosionUse route templates: /users/:id

Rule of thumb: Keep total active series under 10M per TSDB node. Monitor prometheus_tsdb_head_series as a meta-metric.

Metric Types

TypeBehaviorExampleQuery Pattern
Counter Monotonically increasing; resets on restart http_requests_total rate(http_requests_total[5m]) → req/sec
Gauge Goes up and down; snapshot of current state node_memory_available_bytes Direct value or avg_over_time(...[1h])
Histogram Client-side bucketed distribution; cumulative counters per bucket http_request_duration_seconds_bucket{le="0.5"} histogram_quantile(0.99, rate(...))
Summary Client-side calculated quantiles; not aggregatable across instances rpc_duration_seconds{quantile="0.99"} Direct read; cannot be re-aggregated

Histogram vs Summary: When to Use What

Prefer Histograms in almost all cases. Histograms are aggregatable across instances (you can compute a global p99 from multiple servers). Summaries are pre-computed on the client and cannot be combined — the p99 of p99s is not the global p99.

Use Summaries only when you need exact quantiles from a single instance and cannot afford the bucket explosion (e.g., a batch job reporting its own latency percentiles).

Section 04

Time-Series Storage Engine

Write-Optimized Append-Only Design

Time-series data has a unique access pattern: write-once, read-rarely, always in order. This allows extreme optimization compared to general-purpose databases.

LSM-Tree Inspired Architecture

  • Head Block (in-memory): Newest data (last 2 hours) lives in RAM. Samples are appended to per-series chunks. A WAL (Write-Ahead Log) ensures durability.
  • Compacted Blocks (on-disk): Every 2 hours, the head block is flushed to an immutable, compressed block on disk. Background compaction merges small blocks into larger ones.
  • Inverted Index: A posting list maps each label pair to the set of series IDs containing it, enabling fast label-based filtering (e.g., job="apiserver").
  • Chunk Files: Each series stores samples in compressed chunks (~120 samples). Chunks are memory-mapped for fast reads.

Compression: Delta-of-Delta & XOR

Facebook's Gorilla paper (2015) showed that time-series data compresses to 1.37 bytes per sample using two techniques:

Timestamps: Delta-of-Delta

Metrics arrive at regular intervals (e.g., every 15s). Instead of storing the full 64-bit timestamp:

t₁ = 1681234500
t₂ = 1681234515  → Δ = 15
t₃ = 1681234530  → Δ = 15  → ΔΔ = 0
t₄ = 1681234545  → Δ = 15  → ΔΔ = 0
t₅ = 1681234561  → Δ = 16  → ΔΔ = 1

Most ΔΔ values are 0 (1 bit) or very small (few bits). Compresses 64-bit timestamps to ~1–4 bits each.

Values: XOR Encoding

Consecutive metric values tend to be similar (e.g., CPU at 42.3%, 42.1%, 42.4%). XOR of consecutive IEEE 754 floats yields mostly zeros:

v₁ = 42.3  (IEEE 754 bits)
v₂ = 42.1  → XOR(v₁,v₂) = 00...0101
v₃ = 42.4  → XOR(v₂,v₃) = 00...0011

Leading and trailing zeros are run-length encoded. Average: ~0.37 bytes per value.

EncodingRaw SizeCompressedRatio
Timestamp (64-bit)8 bytes~0.5 bytes (ΔΔ)16:1
Value (64-bit float)8 bytes~0.87 bytes (XOR)9:1
Total per sample16 bytes~1.37 bytes12:1
At Scale: 10M samples/sec × 1.37 bytes = ~13.7 MB/s = ~1.2 TB/day = ~8.2 TB/week for raw data. Without compression, this would be ~138 MB/s = ~12 TB/day. Compression reduces storage costs by 10×.

Downsampling & Retention

Raw data cannot be stored forever. Downsampling progressively reduces resolution while keeping long-term trends queryable.

TierResolutionRetentionStorage Estimate
Raw15-second samples7 days~8 TB
1-minute rollupmin/max/avg/count per minute30 days~5 TB
5-minute rollupmin/max/avg/count per 5 min1 year~10 TB
1-hour rollupmin/max/avg/count per hour5 years~4 TB

How Downsampling Works

A background compaction job reads raw chunks and produces rollup chunks. For each window (e.g., 1 minute), it computes:

  • min — lowest value in the window (for threshold alerting)
  • max — highest value (for peak detection)
  • sum — total (for rate computation)
  • count — number of samples (for accurate averaging)

Storing all four aggregates (not just avg) is critical. avg(avg(x)) is mathematically incorrect for unequal group sizes; storing sum + count allows correct re-aggregation.

Section 05

Alerting Pipeline

Threshold Rules & Anomaly Detection

Static Threshold Rules

alert: HighErrorRate
expr:  rate(http_errors_total[5m])
         / rate(http_requests_total[5m])
         > 0.05
for:   5m
labels:
  severity: critical
annotations:
  summary: "Error rate above 5%"
  • Simple, predictable, easy to understand
  • Prone to false positives during traffic spikes
  • Requires manual tuning per service

Anomaly Detection

  • Seasonal decomposition: Learn daily/weekly patterns, alert on deviation from expected baseline
  • Z-score / IQR: Statistical outlier detection over sliding windows
  • Prophet / Holt-Winters: Forecasting models that predict expected values and flag anomalies
  • ML-based: Autoencoders, isolation forests on multi-dimensional metric vectors

More adaptive but harder to debug. Engineers distrust "the model said so" without explainability.

Alert De-duplication & Escalation

Alert State Machine

                    ┌───────────┐
  rule matches ───→│  PENDING   │──── "for" duration elapsed ────→ FIRING
                    └───────────┘                                     │
                          ↑                                           │
                     rule stops ◄─── resolved ◄──── condition clears ─┘
                     matching

The for clause prevents flapping: a condition must persist for N minutes before transitioning from PENDING to FIRING.

De-duplication & Grouping

  1. Group by: Alerts with the same alertname and cluster labels are grouped into a single notification. Instead of 500 "PodCrashLooping" alerts, the on-call sees "PodCrashLooping: 500 instances in cluster prod-ltx1-k8s-7".
  2. Inhibition: If a higher-severity alert is firing (e.g., "NodeDown"), suppress related lower-severity alerts (e.g., "PodNotReady" on the same node).
  3. Silencing: Temporary mutes during maintenance windows. Engineers create time-bounded silences matching label selectors.
  4. Repeat interval: Re-notify every N hours if still firing, but don't spam. Typical: re-page after 4 hours.

Alert Fatigue Prevention

The #1 Operational Risk

Alert fatigue kills monitoring effectiveness. When on-call engineers receive 200+ alerts/day, they start ignoring all of them. Prevention strategies:

StrategyHow It Helps
Actionable alerts onlyEvery alert must have a clear runbook or action. If no one can act on it, delete the rule.
SLO-based alertingAlert on error budget burn rate, not individual failures. A single 500 error isn't critical; burning 10% of monthly error budget in 1 hour is.
Symptom vs causeAlert on user-facing symptoms ("p99 latency > 500ms") not causes ("CPU > 80%"). One symptom alert replaces dozens of cause alerts.
Regular alert reviewsMonthly review: delete alerts that fired but required no action. Track signal-to-noise ratio.
Tiered severityOnly critical pages. warning goes to Slack. info goes to dashboard only.

LinkedIn Mapping & Interview Questions

LinkedIn's Monitoring Stack

LinkedIn operates one of the largest internal monitoring systems in the industry. Here's how the generic components map to LinkedIn's infrastructure:

Generic ComponentLinkedIn ImplementationNotes
Metrics Agent InGraphs Agent Runs on every host; collects JMX, system, and custom metrics; pushes to Kafka
Message Buffer Kafka LinkedIn invented Kafka — used as the durable transport layer for all metrics
Stream Processor Samza Pre-aggregation, label enrichment, fan-out to TSDB and alerting
Time-Series Database InGraphs TSDB Custom-built time-series store optimized for LinkedIn's scale; handles 100M+ active series
Alerting Engine IRIS Threshold-based alerting with escalation plans, on-call rotation integration, and PagerDuty/Slack routing
Dashboards InGraphs UI Internal dashboard platform for visualizing metrics, creating charts, and sharing views across teams

LinkedIn Data Flow

  1. InGraphs Agent on each host collects metrics (JMX from Java services, system stats, custom counters) every 10–60 seconds.
  2. Agent pushes metric batches to Kafka topics partitioned by metric namespace. Kafka provides backpressure absorption and durability.
  3. Samza jobs consume from Kafka, perform per-minute rollups, enrich with fabric/pool labels, and write to InGraphs TSDB.
  4. InGraphs TSDB stores data across multiple retention tiers. Engineers query via the InGraphs UI or API.
  5. IRIS evaluates alerting rules against near-real-time data, groups alerts, and routes to on-call via PagerDuty, Slack (#oncall channels), or email escalation chains.

Why LinkedIn Built Custom (InGraphs) Instead of Using Prometheus

  • Scale: Prometheus was designed for single-cluster monitoring. LinkedIn needed a globally federated system across dozens of data centers.
  • Integration: Deep integration with LinkedIn's deployment system, Kafka, Samza, and IRIS alerting was easier to build natively.
  • History: InGraphs predates Prometheus (LinkedIn started in ~2011, Prometheus was open-sourced in 2015).
  • Push model: LinkedIn's Java-heavy microservices naturally push JMX metrics; Prometheus's pull model requires additional adapters.

Common Interview Questions

Design Questions

  1. How would you design a system to ingest 10M metrics per second? What are the bottlenecks?
  2. Compare pull-based (Prometheus) vs push-based (StatsD/InGraphs) collection. When would you choose each?
  3. How does a time-series database achieve 12:1 compression? Walk through delta-of-delta and XOR encoding.
  4. Design a downsampling pipeline. Why must you store sum + count instead of just average?
  5. How would you handle label cardinality explosion? A developer added user_id as a metric label — what happens?

Alerting & Operational Questions

  1. Design an alerting pipeline with de-duplication, grouping, and escalation. How do you prevent alert storms?
  2. What is alert fatigue and how do you combat it? What metrics would you track for your alerting system itself?
  3. Explain SLO-based alerting. How does error budget burn rate work?
  4. How would you alert on anomalies without generating false positives? Compare statistical vs ML approaches.
  5. A team complains their dashboard is slow. How would you optimize query performance on a TSDB with 100M active series?

Quick-Fire Answers
