Section 2.1

The Core Operating System: The Kernel

The term operating system has two meanings: the entire software package (kernel + tools + GUI), or more narrowly, the kernel — the central software that manages and allocates computer resources (CPU, RAM, devices). This book focuses on the kernel.

The kernel simplifies programming by providing a software layer to manage limited resources. The kernel executable lives at /boot/vmlinuz (the z means it's compressed).

Figure 1: The Linux kernel sits between user-space applications and hardware, providing 6 core services.

Tasks Performed by the Kernel

TaskDescription
Process SchedulingLinux is a preemptive multitasking OS. Multiple processes reside in memory; the kernel scheduler decides which gets CPU time and for how long (not the processes themselves).
Memory ManagementVirtual memory isolates processes from each other and the kernel. Only needed pages stay in RAM; rest goes to swap. More processes can run simultaneously.
File SystemProvides a hierarchical file system on disk. Create, read, update, delete files and directories.
Process Creation/TerminationLoads programs into memory, allocates resources, and frees them when processes finish.
Device AccessProvides a standardized interface to all devices (keyboards, disks, network cards), arbitrating access among multiple processes.
NetworkingTransmits and receives network packets on behalf of user processes, including routing.
System Call APIProcesses request kernel services via system calls — the kernel entry points.

User Mode vs Kernel Mode

Modern CPUs operate in at least two modes. User mode: can only access user-space memory, cannot execute privileged instructions. Kernel mode: can access all memory and execute any instruction (halt, I/O, MMU). The CPU switches modes during system calls, interrupts, and exceptions.

Figure 2: Hardware enforces the separation between user mode and kernel mode. User-mode programs cannot access kernel memory.

Process vs Kernel View

Process View

  • Doesn't know when it will be scheduled
  • Doesn't know where it is in physical RAM
  • Doesn't know disk locations of its files
  • Cannot directly communicate with other processes
  • Cannot create/terminate itself directly
  • Cannot talk to hardware directly

Kernel View

  • Knows and controls everything
  • Decides CPU scheduling order and duration
  • Maintains all process data structures
  • Maps virtual memory to physical RAM
  • Mediates all IPC
  • Handles all device I/O

The Shell

A shell is a user-space program that reads commands and executes programs. Unlike Windows where the command interpreter is part of the kernel, UNIX shells are ordinary user processes. Different users can run different shells simultaneously.

Major Shells

ShellBinaryKey Features
Bourne shellshThe original. I/O redirection, pipelines, variables, functions, background execution. Written by Steve Bourne.
C shellcshC-like syntax. Added command history, line editing, job control, aliases. By Bill Joy (BSD). Not backward-compatible with sh.
Korn shellkshBackward-compatible with sh. Added C shell interactive features. By David Korn (AT&T).
BashbashGNU reimplementation of Bourne shell. Most widely used on Linux. Combines sh compatibility with csh/ksh interactive features. POSIX conformant.
ZshzshExtended Bourne shell with many improvements. Default on macOS. Powerful completion, theming.

Shells serve two purposes: interactive use (typing commands) and script interpretation (executing shell scripts — text files containing commands).

Section 2.3

Users and Groups

Every user has a unique login name (username) and numeric user ID (UID). Users can belong to one or more groups, each with a group ID (GID). The superuser (root, UID 0) bypasses all permission checks.

Key Files

FileContains
/etc/passwdUser accounts: username, UID, GID, home directory, login shell. (Password field usually x — actual hash in shadow file.)
/etc/shadowEncrypted passwords. Readable only by root.
/etc/groupGroups: group name, GID, comma-separated list of member usernames.

Superuser and Capabilities

The superuser (UID 0, root) can do anything. Since kernel 2.2, Linux divides superuser privileges into capabilities (e.g., CAP_KILL, CAP_NET_ADMIN) so a process can have a subset of root powers.

Single Directory Hierarchy, Directories, Links, and Files

Linux has a single hierarchical directory structure rooted at /. Unlike Windows, there are no drive letters — all disks are mounted into this one tree.

Figure 3: The Linux filesystem is a single tree rooted at /. All files, devices, and even virtual filesystems (/proc) are part of this hierarchy.

File Types

TypeSymbolDescription
Regular file-Ordinary data files (text, binary, images)
DirectorydContains filename-to-inode mappings (links)
Symbolic linklPoints to another filename (can dangle if target deleted)
Block devicebDisk drives (/dev/sda)
Character devicecTerminals, serial ports (/dev/tty)
Named pipe (FIFO)pIPC mechanism with a filesystem name
SocketsIPC endpoint (/var/run/docker.sock)

Hard Links vs Symbolic Links

Hard Link

  • Direct pointer to the same inode
  • File is deleted only when ALL hard links are removed
  • Cannot cross filesystem boundaries
  • Cannot link to directories (usually)
  • ln file hardlink

Symbolic (Soft) Link

  • Contains the pathname of another file
  • Can dangle if target is deleted
  • Can cross filesystems
  • Can link to directories
  • ln -s file symlink

File Permissions

Three categories: owner (u), group (g), other (o). Three permissions each: read (r=4), write (w=2), execute (x=1).

Section 2.5

File I/O Model

UNIX's defining I/O feature is universality of I/O: the same system calls (open(), read(), write(), close()) work on ALL file types — regular files, devices, pipes, sockets. The kernel translates these into the appropriate operations.

Figure 4: File descriptors are small integers referencing open files. fd 0/1/2 are stdin/stdout/stderr. The VFS layer in the kernel routes I/O to the correct subsystem.

File Descriptors

FDNameC streamPurpose
0stdinstdinStandard input (keyboard by default)
1stdoutstdoutStandard output (terminal by default)
2stderrstderrStandard error (terminal by default)

The stdio library (fopen, fprintf, fgets) is layered on top of system calls (open, write, read), adding buffering for efficiency.

Everything Is a File — Same write() for Everything

Whether you're writing to a text file, sending data over a network, or piping to another process — your code uses the same calls: open(), read(), write(), close().

C
// Writing to a REGULAR FILE
int fd = open("/home/user/data.txt", O_WRONLY);  // fd = 3
write(fd, "hello", 5);         // 5 = number of bytes ("hello" = 5 chars)
close(fd);

// Writing to a NETWORK SOCKET
int fd = socket(AF_INET, SOCK_STREAM, 0);        // fd = 4
connect(fd, server_addr, ...);
write(fd, "hello", 5);         // SAME write() call!
close(fd);

// Writing to a PIPE
int pipefd[2];
pipe(pipefd);                  // pipefd[0]=read end, pipefd[1]=write end
write(pipefd[1], "hello", 5); // SAME write() call!
close(pipefd[1]);
write(fd, "hello", 5) — what does the 5 mean? It's the number of bytes to write. "hello" = h, e, l, l, o = 5 characters = 5 bytes.
write(fd, "hello", 5) → where=fd, what="hello", how many bytes=5.

VFS: The Kernel Router for I/O

When you call write(fd, ...), the kernel's Virtual File System (VFS) looks up what type of resource that fd points to and routes the data to the correct subsystem.

Figure 4b: The VFS is a router inside the kernel. Your program uses one simple interface (read/write), the kernel handles the complexity of where data actually goes.

Real Example: Opening Multiple Resources

Every time you open() something, the kernel assigns the next available fd number:

C
// Program starts with fd 0, 1, 2 already open (stdin/stdout/stderr)

int fd3 = open("/desktop/kiran.txt", O_WRONLY | O_CREAT);   // fd 3
write(fd3, "my notes", 8);

int fd4 = open("/music/song.mp3", O_RDONLY);                // fd 4
read(fd4, buffer, 4096);

int fd5 = open("/var/log/app.log", O_WRONLY | O_APPEND);    // fd 5
write(fd5, "error occurred\n", 15);

int fd6 = socket(AF_INET, SOCK_STREAM, 0);                 // fd 6
connect(fd6, db_address, ...);
write(fd6, "SELECT * FROM users", 19);                      // database = socket!

unlink("/desktop/old_file.txt");   // delete a file — no fd needed, just a syscall

Resulting fd table for this process:

  fd 0 → stdin           (keyboard)
  fd 1 → stdout          (terminal)
  fd 2 → stderr          (terminal)
  fd 3 → kiran.txt       (regular file)
  fd 4 → song.mp3        (regular file)
  fd 5 → app.log         (regular file)
  fd 6 → DB connection   (socket)

File Descriptors Are Per-Process, Not Per-Machine

Each process has its own independent fd table. fd 3 in Process A has nothing to do with fd 3 in Process B.

Figure 4c: Each process has its own fd table. fd 3 means different things in different processes. Limits exist at per-process and machine-wide levels.

File Descriptor Limits: ulimit -n

ulimit -n is per process, not per machine.

FD Limits Explained
$ ulimit -n
1024          # each SINGLE process can open max 1024 fds

# So if you have 3 processes running:
# Process A → can open up to 1024 fds
# Process B → can open up to 1024 fds
# Process C → can open up to 1024 fds
# Total possible = 3 × 1024 = 3072 fds across machine

# But the MACHINE-WIDE cap is separate:
$ cat /proc/sys/fs/file-max
100000        # total fds ALL processes on the machine can share

# Even if each process is allowed 1024, if 100 processes
# together hit 100000, no more fds for anyone.

Changing the Limits

Increase FD Limits
# Increase per-process limit (temporary, current shell only)
$ ulimit -n 65535

# Increase per-process limit (permanent)
# edit /etc/security/limits.conf:
kiran  hard  nofile  65535
kiran  soft  nofile  65535

# Increase machine-wide limit
$ sudo sysctl -w fs.file-max=200000
Why does this matter? A web server like Nginx handles thousands of connections. Each connection = 1 socket = 1 fd. Default 1024 is way too low — production servers bump ulimit -n to 65535 or higher.

File Descriptor vs Inode

An fd is a temporary handle your process holds. An inode is the permanent identity of a file on disk. Multiple fds (from different processes or even the same process) can point to the same inode.

Figure 4d: fds are temporary per-process handles. Inodes are permanent per-file identities on disk. Multiple fds can point to the same inode, each with independent read/write positions.
PropertyFile Descriptor (fd)Inode
LifetimeTemporary — dies when process exits or calls close()Permanent — lives while file exists on disk
ScopePer-process (each process has its own table)System-wide (one inode per file on the filesystem)
What is it?A small integer (3, 4, 5...)Metadata: size, owner, permissions, disk block locations
RelationshipMany fds can point to one inodeOne inode per file, multiple hard links share it
Read/write positionEach fd has its own independent offsetN/A — inode stores file data, not cursor position
Example
# Same file opened twice in one process — different fds, same inode:
fd 3 = open("kiran.txt")   → inode 12345, position=0
fd 4 = open("kiran.txt")   → inode 12345, position=0 (independent)

# Two processes opening the same file:
Process A: fd 3 ──┐
                   ├──→ inode 12345 → /home/kiran/data.txt (on disk)
Process B: fd 5 ──┘

# close(fd) tells the kernel "I'm done with this number, reuse it."
# So if you close fd 3, the next open() might give you fd 3 again.

Sockets: Local vs Network — Do You Need a Port?

Local (UNIX Domain Socket)

  • No port needed — just a file path
  • e.g., /var/run/docker.sock
  • Data goes through kernel directly
  • No network stack involved
  • Fastest IPC between processes on same machine
  • socket(AF_UNIX, ...)

Cross-Machine (TCP/IP Socket)

  • Port required + IP address
  • e.g., 10.0.0.2:8080
  • Goes through full network stack
  • All 7 OSI layers involved
  • Works across the internet
  • socket(AF_INET, ...)
Figure 4e: UNIX domain sockets skip the entire network stack — just a file path, no port. TCP/IP sockets need an IP address + port and traverse every network layer.

The OSI 7-Layer Model: Full Network Path

When two programs on different machines communicate via a socket, data passes through all 7 OSI layers. The Socket API is the door your code uses to enter the stack.

Figure 4f: The complete OSI model. Layer 7 (Application) is closest to you. Layer 1 (Physical) is closest to the wire. Each layer communicates logically with its peer on the other machine.
#LayerWhat It DoesExamples
7ApplicationYour app's protocol — closest to YOUHTTP, FTP, DNS, SSH, SMTP
6PresentationEncryption, encoding, compressionSSL/TLS, JPEG, ASCII, gzip
5SessionEstablish, manage, terminate connectionsNetBIOS, RPC, session tokens
4TransportEnd-to-end delivery, adds port numbersTCP (reliable), UDP (fast)
3NetworkRouting, adds IP addressesIP, ICMP, routing protocols
2Data LinkFraming, adds MAC addresses, error checkEthernet, WiFi, ARP
1PhysicalBits on the wire — closest to the WIRECable, fiber, radio, voltage

Easy to remember: Layer 1 is the floor (physical stuff you can touch). Layer 7 is the ceiling (apps you interact with).

Data Encapsulation: Wrapping and Unwrapping

When data goes down the stack on Machine A, each layer wraps it with a header. When it arrives on Machine B and goes up, each layer strips its header.

Figure 4g: Each layer adds its own header as data travels down. The packet grows like Russian nesting dolls. On the receiving side, each layer peels off its header to deliver the original data.
Encapsulation Step by Step
Sender (going DOWN):
  App data:                    [Hello]
  Transport adds port:         [TCP:8080 | Hello]
  Network adds IP:             [IP:10.0.0.1→10.0.0.2 | TCP | Hello]
  Data Link adds MAC:          [ETH:MAC_A→MAC_B | IP | TCP | Hello | CRC]
  Physical:                    0101010110110...

Receiver (going UP):
  Physical:                    receives bits
  Data Link:                   strips ETH header, checks MAC → [IP | TCP | Hello]
  Network:                     strips IP header, checks dest IP → [TCP | Hello]
  Transport:                   strips TCP header, delivers to port 8080 → [Hello]
  Application:                 receives "Hello"
Locally, you skip all this! UNIX domain sockets (AF_UNIX) talk through the kernel directly — no TCP, no IP, no Ethernet, no physical layer. That's why local sockets are much faster.

Section 2.7

Processes

A process is an instance of an executing program. The kernel loads code into virtual memory, allocates resources, and tracks the process via kernel data structures. From the kernel's view, processes are entities among which resources must be shared.

Figure 5: fork() creates a child as a copy of the parent. exec() replaces the child's program with a new one. Together they create new processes running new programs.

Process Memory Layout

A process has four logical segments: Text (code), Data (static variables), Heap (dynamic allocation), Stack (local variables, call frames).

Process Creation: fork() + exec()

  • fork() creates a child that is a copy of the parent (text shared, data/heap/stack copied)
  • The child typically calls exec() to load a new program, replacing all segments
  • fork() returns child's PID to parent, returns 0 to child

Process Termination

  • exit(status) or _exit(status) — voluntary termination
  • Killed by a signal (e.g., SIGKILL) — involuntary
  • Status 0 = success, nonzero = error
  • Parent collects status with wait()
  • Check last command's exit status: echo $?

Process Credentials (UIDs/GIDs)

IDPurpose
Real UID/GIDWho you really are. Inherited from parent. Login shell gets these from /etc/passwd.
Effective UID/GIDUsed for permission checks. Usually same as real. Changed by set-user-ID programs.
Supplementary GIDsAdditional groups the process belongs to.

Special Processes

ProcessDetails
init (PID 1)Parent of all processes. Created by kernel at boot from /sbin/init. Runs as root. Cannot be killed. Adopts orphaned processes.
DaemonsLong-lived background processes. No controlling terminal. Examples: sshd, httpd, syslogd, cron.

Resource Limits

Each process has soft and hard limits on resources (open files, memory, CPU time). Set with setrlimit() system call or ulimit shell command. Inherited by child processes via fork().

Java Memory Architecture — Deep Dive

This section maps every Java component to its exact location in the Linux process memory layout. We trace how ZooKeeper and Espresso use Text, Data, BSS, Heap, Stack, and Kernel memory.

Java Process Memory Layout

A running Java program is a regular Linux process. Its virtual address space has the same segments as any C program — but the JVM carves up the heap into specialized sub-regions and adds large off-heap native allocations.

Figure: Complete JVM process memory layout showing Text, Data, Heap (young/old gen), Native/Off-heap, Thread Stacks, and Kernel Space

TEXT Machine Code — Read-Only, Shared

  • JVM binary code: libjvm.so, the core JVM runtime (interpreter loop, GC algorithms, class loader)
  • JIT-compiled methods: After the JVM detects a “hot” method (called thousands of times), the C1/C2 compiler compiles it from bytecode to native machine instructions, stored in the CodeCache
  • Interpreter dispatch loop: The bytecode interpreter itself is compiled C++ code in the text segment
  • This region is read-only and can be shared between processes running the same JVM binary

DATA Initialized Global/Static Variables

  • JVM version strings, default configuration constants
  • Signal handler function pointers
  • Static final fields from loaded Java classes (after class loading)
  • Values are read from the executable file when the JVM binary is loaded

BSS Uninitialized Global/Static Variables

  • Large static arrays before initialization
  • JVM internal counters that start at zero
  • The binary doesn’t store these bytes — the kernel records the size and fills with zeros at startup

HEAP Dynamic Memory — The Largest Region

In a Java process, “heap” has two meanings:

Java Heap (-Xmx)Native Heap (malloc)
Eden: Where new objects are born — short-lived
Survivor S0/S1: Survived 1+ minor GC cycles
Old Gen (Tenured): Long-lived objects (DataNode tree, HashMap, SessionTracker) 		Metaspace: Class metadata, method vtables
DirectByteBuffer: Netty I/O buffers via sun.misc.Unsafe
JIT CodeCache: Compiled native methods
Thread-local storage for native code
Managed by GC (G1, ZGC, etc.) Outside GC control. malloc/free or Netty ref-counting

STACK Per-Thread, Auto-Managed

  • Each Java thread gets its own stack, typically 512KB–1MB (-Xss)
  • Contains method call frames with local variables, return addresses, operand values
  • Primitives (int, long, boolean) are stored directly on the stack
  • Object references are on the stack, but the actual objects are on the heap
  • 200 threads × 1MB = 200MB of stack memory!
  • If recursion goes too deep: StackOverflowError

KERNEL SPACE Not Accessible to JVM

  • Socket buffers (sk_buff) for TCP connections
  • Page cache (cached disk I/O)
  • File descriptor table, epoll interest lists
  • The JVM interacts with kernel space only through system calls
Total Process Memory = Heap (-Xmx) + Metaspace + Direct Buffers + Code Cache + Thread Stacks × N + Native libs + Kernel overhead
Example: 4G + 256M + 512M + 240M + 200×1M + 100M ≈ 5.3GB

# Detailed memory map
$ cat /proc/$(pgrep -f ZooKeeper)/maps | head -40

# Java-specific: heap + non-heap
$ jcmd $(pgrep -f ZooKeeper) VM.native_memory summary

Stack vs Heap — What Goes Where in Java

Figure: Stack holds primitives and references per-thread; Heap holds actual objects shared by all threads

Stack: Per-Thread, Automatic

When ZooKeeper’s RequestProcessor chain calls PrepRequestProcessor → SyncRequestProcessor → FinalRequestProcessor, each call pushes a frame:

┌─────────────────────────────────────┐  ← Stack Pointer (top)
│ serialize() frame                   │
│   int offset = 0    (primitive)     │
│   ByteBuffer buf    (ref → off-heap)│
│   return address                    │
├─────────────────────────────────────┤
│ processRequest() frame              │
│   Request req       (ref → heap)    │
│   int txnType = 1   (primitive)     │
│   byte[] data       (ref → heap)    │
│   return address                    │
├─────────────────────────────────────┤
│ main() frame                        │
│   String[] args     (ref → heap)    │
│   int port = 2181   (primitive)     │
│   return address                    │
└─────────────────────────────────────┘  ← Stack Bottom

When serialize() returns, its frame is popped — all its locals vanish instantly. No GC needed.

fork() in Java — The Dangerous Copy

Figure: fork() duplicates the address space but only 1 thread survives — mutexes from dead threads stay locked forever

What Happens When a Java Process Forks

Java apps almost never call fork() directly. But Runtime.exec("command") internally calls fork() + exec():

  1. fork() duplicates the entire virtual address space: 4GB heap, 1GB Netty direct buffers, all thread stacks. With Copy-on-Write (COW), physical pages aren’t copied until modified, but page table entries are all duplicated.
  2. Only ONE thread survives: If your ZooKeeper has 200 threads, the child gets only the calling thread. The other 199 vanish. Mutexes they held are permanently locked in the child (JVM internal locks, GC locks, classloader locks).
  3. exec() must follow immediately: It replaces the entire address space with a new program, cleaning up the dangerous half-forked state.
OOM Kill Risk: Redis and large JVMs can trigger OOM killer during fork(). Even with COW, the kernel accounts for potential memory. The vm.overcommit_memory sysctl controls this. Set to 1 for large JVM workloads. Modern Java (9+) uses posix_spawn() via ProcessBuilder to avoid full fork().

Netty I/O — Two Worlds of Memory

GC can move heap objects during compaction. The kernel needs a stable address for DMA. Direct buffers are pinned at a fixed native address.

Netty’s Memory Management

  • PooledByteBufAllocator: Pre-allocated arenas of direct memory, sliced into buffers as needed

Memory Mappings

The mmap() system call creates a new memory mapping in a process's virtual address space. There are two types: file mappings (backed by a file on disk) and anonymous mappings (no backing file, pages zeroed).

Figure 6: File mappings let you access file contents directly through memory pointers. Anonymous mappings provide zero-filled memory (used by malloc for large allocations).

Private vs Shared Mappings

Private Mapping

  • Changes NOT visible to other processes
  • Changes NOT written to underlying file
  • Copy-on-write semantics
  • Used for: loading program text, private data

Shared Mapping

  • Changes visible to all processes sharing the mapping
  • Changes written through to the underlying file
  • Used for: memory-mapped I/O, shared memory IPC

Section 2.11

Signals

Signals are software interrupts that notify a process of an event. They can be sent by the kernel, another process, or the process itself. A process can ignore a signal, be killed by it, be suspended, or handle it with a custom function.

Figure 9: Signals flow from sources (kernel, other processes, self) through pending/blocked states to one of four actions: default, ignore, custom handler, or block.

Signal Lifecycle

  1. Generated — kernel, another process, or self sends the signal
  2. Pending — signal is queued, waiting to be delivered
  3. Blocked (optional) — added to process's signal mask, stays pending until unblocked
  4. Delivered — process takes action (default, ignore, or handler)
SIGKILL (9) and SIGSTOP cannot be caught, blocked, or ignored! These are the kernel's guaranteed way to kill or stop any process.

ProcessesRunning program instance. Created with fork()+exec(). Has PID, PPID, credentials (UIDs/GIDs). init (PID 1) is ancestor of all.
Memory Mappingsmmap() maps files or anonymous memory into process address space. Private or shared. Used for loading programs, large mallocs, IPC.
LibrariesStatic (.a): copied into executable. Shared (.so): loaded at runtime, shared in memory. Dynamic linker resolves at runtime.
IPC7 mechanisms: signals, pipes, FIFOs, sockets, message queues, semaphores, shared memory. Sockets work across network.
SignalsSoftware interrupts. Generated by kernel/process/self. Actions: default, ignore, handler, block. SIGKILL/SIGSTOP cannot be caught.
ThreadsShare memory within a process. Own stack only. Lighter than processes. Need mutexes for synchronization.
Sessions & JobsSession = collection of process groups. Foreground job gets terminal. Ctrl+C/Z send signals to foreground group.
PseudoterminalsMaster-slave virtual device pair. Slave acts like a terminal. Used by terminal emulators, SSH, screen/tmux.
TimeReal time (wall clock, Epoch-based) vs CPU time (user + system). Measured with time command.
Client-ServerClient sends requests, server responds. Can be on same or different hosts. Communicate via IPC/sockets.
RealtimeGuaranteed response deadlines. POSIX.1b extensions. Linux PREEMPT_RT patch for hard realtime.
/procVirtual filesystem exposing kernel data. Per-process info at /proc/PID/. System tunables at /proc/sys/. Human-readable text.

Master Troubleshooting Cheat Sheet

Complete Linux Troubleshooting Commands
# ═══════════════ SYSTEM INFO ═══════════════
$ uname -a                     # Kernel & system info
$ cat /etc/os-release           # Distribution info
$ uptime                       # Uptime & load averages
$ dmesg | tail                 # Kernel messages
$ journalctl -xe               # System logs

# ═══════════════ PROCESSES ═══════════════
$ ps aux --sort=-%mem | head   # Top memory consumers
$ ps aux --sort=-%cpu | head   # Top CPU consumers
$ top / htop                   # Interactive process monitor
$ pstree -p                    # Process tree
$ strace -p PID                # Trace system calls of running process
$ ltrace -p PID                # Trace library calls

# ═══════════════ MEMORY ═══════════════
$ free -h                      # RAM and swap usage
$ cat /proc/meminfo            # Detailed memory stats
$ vmstat 1 5                   # Virtual memory stats every 1s
$ pmap PID                     # Memory map of a process
$ slabtop                      # Kernel slab cache usage

# ═══════════════ DISK & FILESYSTEM ═══════════════
$ df -h                        # Filesystem disk space
$ du -sh /var/*                # Directory sizes
$ iostat -x 1                  # Disk I/O stats
$ iotop                        # I/O usage per process
$ lsblk                        # Block device layout
$ mount | column -t            # Mounted filesystems

# ═══════════════ NETWORK ═══════════════
$ ss -tuln                     # Listening sockets
$ ss -tun                      # Established connections
$ ip addr show                 # Network interfaces
$ ip route                     # Routing table
$ netstat -s                   # Network statistics
$ tcpdump -i eth0              # Packet capture

# ═══════════════ USERS & SECURITY ═══════════════
$ w                            # Who is logged in
$ last                         # Login history
$ id                           # Current user/group info
$ getfacl file                 # Access control lists
$ getcap /usr/bin/*            # File capabilities

# ═══════════════ SIGNALS & IPC ═══════════════
$ kill -l                      # List all signals
$ ipcs                         # System V IPC status
$ lslocks                      # Active file locks

# ═══════════════ /proc DEEP DIVE ═══════════════
$ cat /proc/PID/status         # Process overview
$ cat /proc/PID/maps           # Memory map
$ ls -la /proc/PID/fd/         # Open files
$ cat /proc/PID/io             # I/O counters
$ cat /proc/PID/limits         # Resource limits
$ sysctl -a                    # All kernel parameters
