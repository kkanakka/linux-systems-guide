Section 01

Overview & Requirements

Pastebin is a text-sharing service that lets users upload snippets of text (code, logs, config) and share them via a unique URL. Think of it as a simple, write-once, read-many content store with optional expiration and syntax highlighting.

Back-of-Envelope: 10M pastes/day × 10 KB avg = 100 GB/day of new content. Over a year that’s ~36.5 TB of raw content. With metadata overhead and replication, plan for ~100 TB/year. Read QPS: 100M / 86400 ≈ 1,150 reads/sec. Write QPS: 10M / 86400 ≈ 115 writes/sec.

Functional Requirements

  • Create Paste — Upload text content with optional title, language, expiration, and visibility setting
  • Read Paste — Retrieve paste content by unique short URL/ID
  • Syntax Highlighting — Render pastes with language-specific highlighting
  • Expiration — Pastes auto-delete after configurable TTL (10 min, 1 hour, 1 day, 1 week, 1 month, never)
  • Visibility — Public (listed), unlisted (link-only), or private (owner-only)
  • Raw View — Serve plain-text content without HTML rendering
  • Delete Paste — Owner can delete their own pastes

Non-Functional Requirements

  • Low Latency — p99 read latency < 200ms; write latency < 500ms
  • High Availability — 99.9% uptime; reads must continue even if write path degrades
  • Data Durability — Zero data loss for non-expired pastes; replicated across AZs
  • Consistency — Eventual consistency acceptable (paste visible within seconds of creation)
  • Abuse Prevention — Rate limiting, spam detection, content size caps

Architecture Overview

Figure 1 — High-level architecture of a Pastebin-style service

Architecture Walkthrough

  1. Client — Browser or API consumer sends create/read requests through a load balancer.
  2. Load Balancer — Distributes traffic across API server instances; handles TLS termination and rate limiting at the edge.
  3. API Servers — Stateless application tier that validates input, generates unique paste IDs, and orchestrates reads/writes.
  4. Key Generation Service (KGS) — Pre-generates unique short keys (Base62-encoded) to avoid collisions. Keys are stored in a two-table scheme: unused_keys and used_keys.
  5. Metadata DB — Stores paste metadata (title, language, expiration, visibility, owner, timestamps). Structured, indexed, and queryable.
  6. Content Store (Blob Storage) — Stores the actual paste content as immutable blobs keyed by content hash (SHA-256). Enables deduplication.
  7. Cache (Redis/Memcached) — Caches hot paste metadata and content to absorb the 10:1 read-heavy workload.
  8. CDN — Serves raw paste content and syntax-highlighted HTML for popular pastes at the edge.
  9. Cleanup Service — Background worker that scans for expired pastes and deletes them from both metadata DB and content store (if no other paste references the same content hash).

Section 02

Data Model & Storage Strategy

Why Separate Metadata from Content?

Separation Benefits

  • Independent scaling — Metadata is tiny (~200 bytes), content is large (up to 10 MB). Different storage engines optimize for each.
  • Content deduplication — Multiple pastes with identical content reference the same blob via SHA-256 hash.
  • Cache efficiency — Metadata fits entirely in cache; content cached separately by popularity.
  • Faster queries — Listing, searching, and filtering operate on small metadata records only.

Tradeoffs

  • Two-phase writes — Must write content first, then metadata. Requires idempotency and error handling.
  • Referential integrity — Orphaned blobs possible if metadata write fails; cleanup service handles this.
  • Extra hop on read — Read metadata to get content hash, then fetch content. Mitigated by caching.

Content Deduplication via SHA-256

Before storing content, compute SHA-256(content). Use the hash as the blob key. If the hash already exists in the content store, skip the write and point the new paste metadata at the existing blob. This saves significant storage for commonly pasted snippets (e.g., “Hello World” programs, default configs).

Dedup Write Flow

  1. Client submits paste content to API server.
  2. API server computes content_hash = SHA-256(content).
  3. Check content store: does content_hash already exist?
  4. If yes: skip content write, increment reference count.
  5. If no: write content blob with key = content_hash, set ref_count = 1.
  6. Generate unique paste_id from KGS. Write metadata record pointing to content_hash.
  7. Return paste URL to client.

Figure 2 — Data model relationships: Paste Metadata → Content Blob (via content_hash), Paste Metadata → User (via owner_id)

Data Model Walkthrough

  1. Paste Metadata is the primary record. Each paste gets a unique paste_id from KGS and stores a pointer (content_hash) to the actual content.
  2. Content Blob is keyed by SHA-256 hash. Multiple paste metadata records can point to the same blob — this is how deduplication works.
  3. ref_count tracks how many pastes reference a blob. When it reaches zero (all referencing pastes deleted or expired), the cleanup service garbage-collects the blob.
  4. User Table is optional — anonymous pastes set owner_id = NULL. Registered users get API keys and daily quotas.
  5. Expiration is stored as an absolute timestamp. The cleanup service queries WHERE expires_at < NOW() in batches.
LinkedIn Mapping: Metadata lives in Espresso (document store with secondary indexes for listing by owner, expiration scans). Content blobs live in Ambry (immutable blob store optimized for large objects, with built-in replication and content-addressed storage).

Delete Paste — DELETE /api/v1/pastes/{paste_id}

Response — 204 No Content

HTTP/1.1 204 No Content
Delete Flow: Verify ownership via API key or session token. Decrement ref_count on the content blob. Soft-delete the metadata record. If ref_count reaches zero, the cleanup service will garbage-collect the blob.

CDN-Friendly: Raw content responses include Cache-Control: public, max-age=3600 headers. Popular pastes are served directly from CDN edge nodes without hitting the origin.
Section 04

Key Design Decisions

Content Deduplication

Pros

  • Significant storage savings — common snippets stored once
  • Content-addressed storage aligns with blob store best practices
  • SHA-256 collision probability is astronomically low (~10-60)
  • Faster writes when content already exists (skip blob write)

Cons

  • Requires ref_count management and garbage collection
  • Hash computation adds CPU cost on every write
  • Concurrent dedup writes need atomic check-and-set
  • Deletion is more complex (decrement ref, not direct delete)

Paste Expiration

Lazy Deletion (Chosen)

  • On read: check expires_at; return 404 if expired
  • Background cleanup service purges in batches every 5 minutes
  • Simple, no TTL support needed at storage layer
  • Cleanup service is eventually consistent — acceptable for this use case

Active Deletion (Alternative)

  • Use storage-level TTL (e.g., Cassandra TTL, Redis EXPIREAT)
  • Automatic, no cleanup service needed
  • Harder to implement dedup (can’t set TTL on shared blobs)
  • Less control over deletion timing and batch sizes

Syntax Highlighting

Client-Side (Chosen for Default)

  • Use highlight.js or Prism.js in the browser
  • Zero server CPU cost for highlighting
  • Supports 200+ languages without server changes
  • Slightly slower first render on large pastes

Server-Side (Chosen for Popular Pastes)

  • Pre-render HTML for pastes with > N views
  • Cache highlighted HTML in CDN
  • Instant render for popular pastes
  • Higher server cost, but only for hot content

Rate Limiting

Multi-Layer Rate Limiting

Layer Strategy Limit
Edge / LB IP-based token bucket 60 creates/min per IP
API Key Sliding window counter 500 creates/day per API key
Anonymous IP + fingerprint 10 creates/hour, max 50 KB each
Content Size cap 10 MB max per paste
Section 05

Scaling & Performance

Read Path Optimization

Three-Tier Caching

  1. CDN Edge — Cache raw content and pre-rendered HTML for popular pastes. TTL = 1 hour. Absorbs 60-70% of all reads.
  2. Redis / Memcached — Cache paste metadata and hot content blobs. LRU eviction. Absorbs 25-30% of remaining reads.
  3. Read Replicas — Metadata DB read replicas in each datacenter for cache misses. Only 5-10% of reads hit the primary.

Write Path Optimization

  • Content-first write — Write blob to content store before metadata. If metadata write fails, orphaned blob is cleaned up.
  • Async indexing — Syntax highlighting pre-render and search indexing happen asynchronously via message queue.
  • KGS key pre-generation — Keys are pre-generated in bulk and cached. API servers pull from a local pool, refilling from KGS in background.

Blob Storage Sharding

Content blobs are sharded by the first 2 characters of the SHA-256 hash, distributing across 256 shards (16 × 16). Each shard is replicated 3× across availability zones. This provides uniform distribution (SHA-256 output is uniformly random) and avoids hot-shard problems.

Section 06

Interview Questions

Q: How do you handle very large pastes (e.g., 10 MB)?

A: Large pastes are uploaded via chunked transfer encoding. The API server streams content directly to the blob store without buffering the full body in memory. On the read path, large pastes are served with Content-Length and Accept-Ranges headers to enable partial downloads. The CDN caches them at the edge, so repeat reads don’t hit origin. For syntax highlighting, large pastes use client-side rendering with virtual scrolling (only highlight visible lines) to avoid freezing the browser.

Q: How does paste expiration work at scale?

A: Expiration uses a two-layer approach. Read-time check: every read compares expires_at against the current time; expired pastes return 404 immediately. Background cleanup: a scheduled job runs every 5 minutes, querying WHERE expires_at < NOW() LIMIT 1000 in batches. It soft-deletes metadata, decrements blob ref_counts, and garbage-collects blobs with ref_count = 0. The batch size and frequency are tunable. At 10M pastes/day with average 1-day TTL, the cleanup service processes ~7,000 expirations/minute, well within a single worker’s capacity. Multiple workers can shard by paste_id range for parallelism.

Q: How do you prevent abuse (spam, malware, illegal content)?

A: Multiple layers: (1) Rate limiting at edge, API key, and IP level (see Section 4). (2) Content scanning — async pipeline checks pastes against known malware hashes and blocklists. (3) Size limits — 10 MB cap per paste, 50 KB for anonymous users. (4) CAPTCHA for anonymous users creating more than 3 pastes/hour. (5) Reporting mechanism — users can flag pastes for review. (6) Account reputation — new accounts start with lower quotas. Flagged content is hidden immediately and reviewed within 24 hours.

Q: How does content deduplication interact with expiration and deletion?

A: Each content blob maintains a ref_count tracking how many paste metadata records reference it. When a paste expires or is deleted, its metadata is removed and the blob’s ref_count is atomically decremented. The blob is only garbage-collected when ref_count reaches zero. This means if User A creates paste “Hello World” (expires in 1 hour) and User B creates the same paste (never expires), the content blob persists because User B’s reference keeps ref_count > 0. The atomic decrement uses UPDATE SET ref_count = ref_count - 1 WHERE content_hash = ? AND ref_count > 0 to prevent underflow. A periodic consistency checker scans for blobs with ref_count = 0 and no referencing metadata as a safety net.
