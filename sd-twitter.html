Overview & Scale Estimation

Twitter is a global microblogging platform where users post short messages (tweets), follow other users, and consume a personalized timeline. The core challenge is delivering a real-time, ranked feed to hundreds of millions of users while ingesting an enormous write load.

Core Functional Requirements

  • Post tweet — text (280 chars), images, videos, links
  • Follow/unfollow — asymmetric social graph
  • Home timeline — chronological + ranked feed of followed users' tweets
  • Search — full-text, hashtag, and user search
  • Trending topics — real-time detection of popular hashtags and keywords
  • Like, retweet, reply — engagement actions on tweets

Non-Functional Requirements

  • Availability > 99.99% — users expect Twitter to always be reachable
  • Latency — timeline load < 200ms p99, tweet post < 500ms
  • Consistency — eventual consistency is acceptable for timeline; strong consistency for tweet creation
  • Partition tolerance — must tolerate datacenter failures

Tweet Creation & Fan-Out

The central architectural challenge of Twitter is the fan-out problem: when a user tweets, that tweet must appear in the timeline of every follower. With some users having 100M+ followers, naive approaches break down. There are three fundamental strategies.

Figure 1 — High-level architecture showing tweet ingestion, fan-out service, timeline cache, and read path

Fan-Out on Write (Push Model)

How It Works

When a user posts a tweet, the system immediately pushes the tweet ID into the timeline cache (Redis sorted set) of every follower. The timeline is pre-computed at write time.

Step-by-Step: Fan-Out on Write

  1. Tweet arrives — user posts a tweet via the API gateway. The tweet is written to the Tweets table (primary storage) and gets a unique tweet_id (Snowflake ID).
  2. Follower lookup — the Fan-Out Service queries the follows table to retrieve the list of all follower IDs for the author.
  3. Timeline insertion — for each follower, the service does ZADD timeline:{user_id} {timestamp} {tweet_id} in Redis, inserting the tweet into every follower's pre-computed timeline.
  4. Async via message queue — the fan-out is done asynchronously through a message queue (Kafka) so the tweet post returns immediately to the author.
  5. Timeline read — when a follower opens their home feed, they simply do ZREVRANGE timeline:{user_id} 0 49 to get the latest 50 tweet IDs, already sorted. Extremely fast: O(log N + K).

Pros

  • Timeline read is O(1) — pre-computed, just a Redis lookup
  • Read latency is extremely low (< 5ms from cache)
  • Simple read path — no joins, no aggregation at read time

Cons

  • Celebrities with 100M followers → 100M Redis writes per tweet
  • Huge write amplification and potential minutes of lag
  • Wasted work for inactive followers who never read their timeline
  • High storage cost for per-user timeline caches

Fan-Out on Read (Pull Model)

How It Works

No pre-computation at write time. When a user opens their timeline, the system fetches tweets at read time by querying the followed users' tweet lists and merging them.

Step-by-Step: Fan-Out on Read

  1. Tweet arrives — user posts a tweet. It is written only to the Tweets table indexed by author_id. No fan-out happens.
  2. Timeline request — when a follower opens their home feed, the Timeline Service fetches their following list (e.g., 500 users).
  3. Parallel fetch — the service queries the Tweets table for the latest N tweets from each of the 500 followed users, in parallel.
  4. K-way merge — results are merged using a min-heap (priority queue) to produce a sorted timeline of the top 50 tweets by timestamp.
  5. Return — the merged result is returned to the client. Optionally cached briefly for repeat loads.

Pros

  • No write amplification — tweet is stored once
  • Celebrity tweets are not a problem at write time
  • No wasted computation for inactive users

Cons

  • Read latency is high — must query 500+ users' tweet lists and merge
  • Read-heavy system (1000:1 ratio) means this is expensive at scale
  • Difficult to meet <200ms SLA with large following lists

Hybrid Approach (Twitter's Actual Solution)

The Winning Architecture

Twitter uses a hybrid approach: Fan-Out on Write for regular users, Fan-Out on Read for celebrities. A user is classified as a "celebrity" if their follower count exceeds a threshold (e.g., > 500K followers).

Step-by-Step: Hybrid Fan-Out

  1. Regular user tweets — fan-out on write. The tweet ID is pushed into all followers' Redis timeline caches (same as push model).
  2. Celebrity tweets — no fan-out at write time. The tweet is stored in the Tweets table and in a separate celebrity_tweets cache indexed by author.
  3. Timeline read — the Timeline Service fetches the user's pre-computed timeline from Redis (covers regular users' tweets) AND fetches latest tweets from each celebrity the user follows.
  4. Merge — the two result sets are merged by timestamp and ranked. The merged timeline is returned to the client.
  5. Threshold tuning — the celebrity threshold is tuned dynamically based on system load. Users near the boundary might be re-classified periodically.

The Celebrity Problem

Why Celebrities Break Fan-Out on Write

Consider a celebrity with 50 million followers who tweets. Under pure fan-out on write:

  • 50M Redis ZADD operations must complete
  • At 100K ops/sec per Redis shard, that's 500 seconds (8+ minutes) of fan-out
  • Followers see the tweet with minutes of delay, not seconds
  • A burst of celebrity tweets (e.g., during a live event) can overwhelm the entire fan-out pipeline
  • Most of those 50M followers are inactive — the work is wasted

Hybrid Solution to the Celebrity Problem

  • Celebrity tweets skip the fan-out pipeline entirely
  • Only ~10–50 celebrity accounts need fan-out on read per user (most people follow few celebrities)
  • Merging 10–50 celebrity tweet lists at read time adds only ~20–50ms
  • Net result: tweet posting is fast for everyone, timeline reads stay under 200ms
Section 3

Timeline Generation

The home timeline is the core product surface. It must feel instant, be personalized, and reflect near-real-time activity. The architecture combines pre-computed caches, a hydration pipeline, and cursor-based pagination.

Redis Sorted Sets for Timeline Storage

Data Structure

Each user's timeline is stored as a Redis Sorted Set keyed by timeline:{user_id}. The score is the tweet's Snowflake timestamp, and the member is the tweet_id.

Key:    timeline:12345
Type:   Sorted Set (ZSET)

Score (timestamp)       Member (tweet_id)
───────────────────     ─────────────────
1712345678901           tweet_98765
1712345679234           tweet_98770
1712345680100           tweet_98773
1712345681456           tweet_98780
...

Commands:
  ZADD  timeline:12345 1712345681456 tweet_98780    # Insert tweet
  ZREVRANGE timeline:12345 0 49                     # Get latest 50
  ZREMRANGEBYRANK timeline:12345 0 -801             # Trim to 800 max
  ZCARD timeline:12345                              # Count entries

Why Sorted Sets?

  1. O(log N) insert — adding a tweet to a sorted set is logarithmic, fast even with 800 entries per user.
  2. O(log N + K) range query — fetching the top K tweets by score (timestamp) is efficient.
  3. Automatic ordering — tweets are always sorted by time; no client-side sorting needed.
  4. Built-in dedup — a tweet_id can only appear once per set, preventing duplicates.
  5. Trim support — ZREMRANGEBYRANK keeps the set bounded (e.g., max 800 entries), preventing unbounded memory growth.

Hydration Pipeline

The timeline cache stores only tweet_ids for compactness. Before returning to the client, those IDs must be hydrated into full tweet objects with author info and media URLs.

Hydration Flow: tweet_ids → Full Response

Timeline Cache (Redis)
    │
    ▼
[tweet_id_1, tweet_id_2, ..., tweet_id_50]
    │
    ├──▶ Tweet Service    ──▶ { text, created_at, media_ids, author_id, ... }
    │        (batch GET from Tweets table / cache)
    │
    ├──▶ User Service     ──▶ { username, display_name, avatar_url, verified }
    │        (batch GET from Users table / cache)
    │
    └──▶ Media Service    ──▶ { image_url, video_url, thumbnail }
             (resolve media_ids to CDN URLs)
    │
    ▼
Merge & return hydrated timeline JSON to client

Hydration Pipeline Steps

  1. Fetch tweet_ids — ZREVRANGE timeline:{user_id} 0 49 returns the latest 50 tweet IDs.
  2. Batch tweet lookup — multiget against the Tweet Service (backed by a cache + database). Returns tweet text, author_id, media_ids, timestamp, reply/retweet metadata.
  3. Batch user lookup — extract unique author_ids, multiget against User Service. Returns display names, avatars, verification badges.
  4. Batch media lookup — extract media_ids, resolve to CDN URLs via Media Service. Images are served from edge CDN.
  5. Merge & serialize — combine tweet data + user data + media URLs into the final JSON response. Apply any ranking adjustments.
  6. Client cache headers — set Cache-Control for short-lived client-side caching (e.g., 30s).

Cursor-Based Pagination

Why Not Offset-Based?

Offset pagination (LIMIT 50 OFFSET 100) breaks in real-time feeds because new tweets shift the offset window, causing duplicates or missed tweets. Cursor-based pagination uses a stable reference point.

GET /api/timeline?count=50
Response: { tweets: [...], next_cursor: "1712345678901" }

GET /api/timeline?count=50&cursor=1712345678901
  → ZREVRANGEBYSCORE timeline:{uid} (1712345678901 +inf 0 49
Response: { tweets: [...], next_cursor: "1712345670500" }

# Cursor = Snowflake timestamp of the LAST tweet in previous page
# Each page fetches tweets OLDER than the cursor
# Immune to new tweets arriving between page loads

Cursor Pagination Benefits

  1. No duplicates — new tweets inserted after the cursor don't shift results.
  2. No gaps — deleted tweets between pages don't cause skips.
  3. Efficient — Redis ZREVRANGEBYSCORE with score boundaries is O(log N + K).
  4. Stateless server — the cursor is a timestamp; the server doesn't need to track session state.
Section 4

Search & Trending

Twitter Search and Trending Topics are two distinct but interconnected systems. Search requires a full-text inverted index over hundreds of billions of tweets, while trending requires real-time stream processing to detect velocity spikes in hashtags and keywords.

Inverted Index for Hashtags

How Tweet Search Works

Every tweet is tokenized and indexed into an inverted index that maps terms (words, hashtags, mentions) to lists of tweet IDs. This enables sub-second full-text search across the entire tweet corpus.

Inverted Index Structure:
─────────────────────────
Term              → Posting List (sorted by timestamp desc)
─────────────────────────
#worldcup         → [tweet_999, tweet_987, tweet_950, tweet_912, ...]
#breakingnews     → [tweet_998, tweet_975, tweet_960, ...]
"earthquake"      → [tweet_997, tweet_990, tweet_985, ...]
@elonmusk         → [tweet_996, tweet_980, ...]

Query: "#worldcup goal"
  1. Fetch posting list for #worldcup
  2. Fetch posting list for "goal"
  3. Intersect lists (AND) or union (OR) depending on query type
  4. Rank by recency + engagement score
  5. Return top K results

Index Architecture

  1. Early bird index — Twitter uses an in-memory, real-time index called Earlybird. New tweets are indexed within seconds of posting, enabling real-time search.
  2. Sharding — the index is hash-partitioned by tweet ID across thousands of index shards. Each shard holds a subset of the inverted index.
  3. Scatter-gather — a search query is broadcast to all shards (scatter), each returns top-K results, and a root aggregator merges them (gather).
  4. Tiered storage — recent tweets (last 7 days) are in hot in-memory index; older tweets are in on-disk index with longer query latency.
  5. Hashtag tokenization — hashtags are treated as single tokens. #WorldCup is indexed as-is (case-normalized) for exact hashtag search.

Trending Detection

Detecting Trends in Real Time

Trending isn't about absolute volume — it's about velocity. A hashtag that goes from 100 to 10,000 tweets in 5 minutes is trending, even if another hashtag has 50,000 steady tweets. The system detects anomalous acceleration.

Sliding Window + Count-Min Sketch

  1. Stream ingestion — every tweet flows through a Kafka topic. A stream processor (like Heron/Storm) extracts hashtags and keywords from each tweet.
  2. Count-Min Sketch — a probabilistic data structure that approximates the frequency of each term using sub-linear memory. Instead of exact counts for millions of terms, it uses a 2D array of counters with multiple hash functions.
  3. Sliding window — counts are maintained over rolling time windows (e.g., 1-min, 5-min, 1-hour). Old windows are decayed or dropped.
  4. Velocity calculation — for each term, compute velocity = count(current_window) / count(previous_window). A velocity > threshold (e.g., 5x) flags the term as trending.
  5. Filtering — remove spam, NSFW, and known-noisy terms. Apply geo-filtering for location-specific trends.
  6. Ranking — trending topics are ranked by a combination of velocity, total volume, and topic diversity. Personalized trends layer in user interests.

Key Interview Questions

Fan-Out Strategy

Q: How would you design the tweet delivery system? Why not use a single approach?

A: The key insight is that no single approach works for all users. Fan-out on write gives O(1) read latency for regular users but breaks for celebrities with millions of followers (write amplification). Fan-out on read avoids write amplification but is too slow for the read-heavy workload (1000:1 ratio). The hybrid approach classifies users by follower count: regular users (<500K followers) use push, celebrities use pull. At read time, merge the pre-computed timeline with a small number of celebrity tweet fetches. This keeps timeline load under 200ms while bounding write amplification.

Celebrity Problem Deep Dive

Q: What happens when a celebrity with 50M followers tweets during the Super Bowl?

A: Under pure fan-out on write, this single tweet triggers 50M Redis writes, taking 8+ minutes and saturating the fan-out pipeline. Meanwhile, other users' tweets queue behind it. The hybrid model solves this: the celebrity tweet is stored once and fetched at read time. Additionally, implement rate limiting on the fan-out pipeline, priority queues (celebrity tweets get lower fan-out priority), and circuit breakers that automatically switch high-traffic accounts to pull mode. For burst events like the Super Bowl, pre-warm caches and scale the read-path infrastructure horizontally.

Real-Time Search

Q: How do tweets become searchable within seconds of posting?

A: Twitter's Earlybird uses an in-memory inverted index optimized for write-heavy workloads. New tweets are tokenized and indexed into the in-memory segment within seconds (not minutes like batch-indexed systems). The index is sharded by tweet ID and uses a scatter-gather pattern: a query is broadcast to all shards, each returns its top-K matches, and a root node merges results. For hashtag trending detection, a parallel stream processing pipeline (Count-Min Sketch with sliding windows) identifies velocity spikes. At LinkedIn, Galene serves a similar role with near-real-time indexing, while Samza handles the streaming analytics equivalent of Twitter's trending pipeline.

Timeline Consistency & Edge Cases

Q: A user posts a tweet and immediately checks their profile. The tweet isn't there yet. How do you handle this?

A: This is the read-after-write consistency problem. Solutions: (1) For the author's own timeline, read from the primary database (not the cache) for a short window after posting. (2) Remember the timestamp of the user's last write in a client-side cookie; if the cached timeline is older, bypass cache and read from the primary. (3) Use async_write_ack: the API returns success only after the tweet is confirmed in the primary store, but fan-out is still async. (4) For the author's profile page specifically, always include a "your recent tweets" section that reads directly from the Tweets table with a WHERE author_id = ? query.

Snowflake ID Generation

Q: How does Twitter generate globally unique, time-ordered IDs at 300K tweets/sec?

A: Twitter's Snowflake generates 64-bit IDs composed of: timestamp (41 bits, ~69 years), datacenter ID (5 bits), worker ID (5 bits), and sequence number (12 bits, 4096 per ms per worker). This gives time-ordering without coordination between ID generators. Each worker generates IDs independently, avoiding a single point of failure. The timestamp prefix enables efficient range queries (WHERE tweet_id > ? is equivalent to WHERE created_at > ?) and enables cursor-based pagination using the tweet_id directly as the cursor.
