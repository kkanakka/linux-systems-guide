Section 01

Overview & Scale

LeetCode is an online coding platform where engineers practice algorithmic problems, participate in weekly contests, and prepare for technical interviews. The system must accept code in dozens of programming languages, execute it securely against hidden test cases, and return verdicts within seconds.

Core Requirements

  • Functional: Submit code → compile → run against test cases → return verdict (Accepted, Wrong Answer, TLE, MLE, Runtime Error, Compilation Error)
  • Non-Functional: Secure sandboxed execution, horizontal scalability of workers, low-latency results, fair contest judging
  • Availability: 99.9% uptime — submissions must never be silently lost
  • Consistency: Identical code + test case = identical verdict every time

Section 02

High-Level Architecture

Core Services

Problem Service

Manages the problem catalog — descriptions, difficulty tags, hints, editorial solutions, and test cases. Serves problem pages with Markdown rendering and supports CRUD for problem setters.

Submission Service

Receives user code submissions, validates them, persists to the database, and enqueues them for judging. Tracks submission status and streams verdict updates to the client via WebSocket.

Judge Worker Pool

A horizontally scalable pool of workers that dequeue submissions, spin up sandboxed containers, compile and execute code against test cases, and report verdicts back. The most compute-intensive component.

User Service

Authentication, profiles, submission history, streak tracking, premium subscriptions, and progress analytics. Maintains solved-problem bitmaps for O(1) lookups.

Contest Service

Manages contest lifecycle — registration, problem reveal at start time, real-time leaderboard computation, and final ranking with penalty calculation.

API Gateway

Rate limiting, authentication, request routing, and anti-cheat throttling. Limits submission frequency to prevent abuse (e.g., max 10 submissions/minute per user).

Submission Flow — 6 Steps

  1. Submit: User writes code in the browser IDE and clicks "Submit". The client sends the source code, language, and problem ID to the API Gateway.
  2. Validate & Persist: The Submission Service validates the request (language supported, code size < 64KB, rate limit not exceeded), generates a submission ID, persists it with status QUEUED, and publishes it to the submission queue.
  3. Dequeue & Prepare: A Judge Worker picks the submission from the queue, fetches test cases from the Problem Service, and prepares the sandbox environment (pulls the language-specific Docker image if not cached).
  4. Compile & Execute: Inside the sandbox container, the worker compiles the code (if applicable), then runs it against each test case sequentially. stdin is piped in, stdout is captured, and resource usage is monitored by cgroups.
  5. Judge & Verdict: The worker compares actual output against expected output for each test case. It determines the final verdict (AC, WA, TLE, MLE, RE, CE) along with runtime and memory statistics.
  6. Report Back: The worker updates the submission record in the database, publishes the verdict to a notification topic, and the Submission Service pushes the result to the user's browser via WebSocket.

Why a Queue?

The submission queue (Kafka / SQS) decouples ingestion from execution. During contest spikes, thousands of submissions arrive within seconds. The queue absorbs the burst while workers process at a steady rate, preventing overload and ensuring no submission is lost.

Section 03

Code Execution Sandbox

Executing arbitrary user code is the most security-critical component. Each submission runs in a disposable, isolated container with strict resource constraints.

Container Isolation

Isolation Strategy: Docker + seccomp + cgroups

Each submission gets its own short-lived Docker container. The container is created from a pre-built language-specific image, runs the code, and is destroyed immediately after. No state persists between submissions.

  • Namespace isolation: PID, network, mount, and user namespaces are all separated
  • seccomp profile: Blocks dangerous syscalls (fork bombs, socket creation, file system writes outside /tmp)
  • Read-only rootfs: Container filesystem is mounted read-only; only /tmp is writable
  • No network access: Network namespace is completely empty — no loopback, no external connectivity
  • Non-root user: Code executes as an unprivileged user inside the container

Resource Limits

ResourceLimitEnforcement
CPU2 corescgroup cpu.cfs_quota_us / cpu.cfs_period_us
Memory256 MBcgroup memory.limit_in_bytes — OOM killer terminates on breach
Wall-Clock Time5 secondsExternal watchdog timer kills container after timeout
Disk50 MBtmpfs mount with size=50m on /tmp
NetworkNoneEmpty network namespace — no interfaces created
Processes64cgroup pids.max prevents fork bombs
Open Files256RLIMIT_NOFILE set via setrlimit

Sandbox Lifecycle

  1. Image Pull (cached): Language images (e.g., judge-python:3.11, judge-cpp:gcc12) are pre-pulled on every worker node. Warm cache ensures <100ms container creation.
  2. Container Create: Docker API creates the container with all cgroup limits, seccomp profile, read-only rootfs, and empty network namespace.
  3. Copy Code: Source code and input files are bind-mounted into /submission (read-only).
  4. Compile: For compiled languages, run the compiler inside the container. If compilation fails, return CE verdict immediately.
  5. Execute Per Test Case: For each test case, pipe stdin, capture stdout/stderr, measure wall time and peak memory via cgroup stats.
  6. Teardown: Container is forcefully removed. All resources are reclaimed.

Time Limit Multipliers

Interpreted languages are inherently slower. The judge applies per-language multipliers to the base time limit: C/C++ = 1x, Java = 2x, Python = 5x, JavaScript = 3x. This ensures fair comparison of algorithmic complexity across languages.

Section 04

Contest System

Weekly contests see 20,000+ concurrent participants submitting code simultaneously. The contest system must handle burst traffic, maintain a real-time leaderboard, and ensure fair ranking.

Real-Time Leaderboard

Redis Sorted Sets

The leaderboard is powered by Redis sorted sets (ZADD / ZREVRANGE). Each participant's score is stored as a composite value encoding both solved count and penalty time, enabling O(log N) updates and O(log N + K) range queries.

Score = (problems_solved * 1,000,000) - total_penalty_minutes

ZADD contest:weekly-350 <score> <user_id>
ZREVRANGE contest:weekly-350 0 49 WITHSCORES   -- top 50

Leaderboard Update Flow

  1. User submits solution → judge returns Accepted verdict
  2. Contest Service calculates new score: solved_count * 1M - (finish_time + penalty * wrong_attempts)
  3. Atomic ZADD updates the sorted set
  4. Leaderboard UI polls every 5 seconds or receives SSE push for top-100 changes
  5. User's rank is computed via ZREVRANK — O(log N)

Concurrent Submissions

Handling Contest Spikes

During contests, submission rate spikes 10–50x above baseline. The system handles this through:

  • Priority Queue: Contest submissions get higher priority than practice submissions in the judge queue
  • Auto-scaling Workers: K8s HPA scales judge worker pods based on queue depth metrics
  • Warm Pool: Pre-provisioned worker pods are kept warm 10 minutes before contest start
  • Deduplication: Identical code resubmitted within 30 seconds is rejected at the API Gateway

Ranking Algorithm

FactorWeightDescription
Problems SolvedPrimaryMore solved = higher rank (always dominant factor)
Finish TimeSecondaryTime from contest start to last AC submission
Wrong Attempt Penalty+5 min eachEach wrong submission on a solved problem adds 5 minutes penalty
Unsolved PenaltiesNoneWrong attempts on unsolved problems carry no penalty

Rating System (Elo-like)

After each contest, user ratings are updated using a system similar to Codeforces' Elo variant. Expected rank is computed from current rating, and the delta is proportional to (expected_rank - actual_rank). New users have higher volatility (larger rating changes) that decreases over time.

new_rating = old_rating + K * (expected_rank - actual_rank) / N
K = base_factor * volatility_multiplier
volatility = max(0.5, 1.0 - contests_participated * 0.02)

Indexing Strategy

  • submissions: Composite index on (user_id, problem_id, created_at DESC) for "my submissions" queries
  • submissions: Index on (problem_id, status, runtime_ms) for percentile ranking ("faster than X%")
  • submissions: Index on (contest_id, user_id) for contest submission lookups
  • problems: Full-text index on title + description_md for search
  • test_cases: Index on (problem_id, ordinal) for ordered retrieval

Common Interview Questions

Q1: How do you sandbox user code securely?

Key points:

  • Containers provide process-level isolation via Linux namespaces (PID, network, mount, user)
  • seccomp profiles whitelist safe syscalls — block fork (beyond limit), socket, mount, ptrace
  • cgroups enforce hard CPU, memory, and PID limits — kernel OOM killer handles breaches
  • No network access eliminates data exfiltration and calling external services
  • Read-only rootfs prevents persistent modifications to the execution environment
  • For defense-in-depth: run workers on dedicated node pools isolated from production services

Q2: How do you scale the judge worker pool?

Key points:

  • Horizontal Pod Autoscaler (HPA): Scale on Kafka consumer lag or queue depth metric
  • Pre-warm before contests: Scheduled scale-up 10 min before contest start based on registration count
  • Priority classes: Contest submissions get PriorityClass: contest-critical; practice gets best-effort
  • Node auto-scaling: Cluster autoscaler adds nodes when pod scheduling fails due to resource pressure
  • Worker efficiency: Each worker processes submissions serially but multiple workers run in parallel. Workers reuse warm containers for the same language.
  • Capacity planning: 100K submissions/day ÷ 86400s ≈ 1.2 sub/s average, but contest peaks hit 100+ sub/s for 5–10 minutes

Q3: How do you handle Time Limit Exceeded (TLE)?

Key points:

  • External watchdog: A timer process outside the container kills it via docker kill after the time limit
  • Why not alarm(): User code could catch/ignore SIGALRM; external enforcement is mandatory
  • Wall time vs CPU time: Measure wall-clock time (not CPU time) because sleep-based evasion should still be caught
  • Per-language multipliers: Python gets 5x the base limit vs C++ to account for interpreter overhead
  • Graceful degradation: On timeout, capture partial output for debugging, mark as TLE, teardown container
  • Container cleanup: Set --stop-timeout=1 so Docker sends SIGKILL 1 second after SIGTERM

Q4: How do you ensure verdict consistency?

Key points:

  • Pinned compiler/runtime versions in Docker images (never use :latest)
  • Deterministic resource limits via cgroups (same CPU quota on every worker)
  • Workers run on homogeneous hardware (same CPU model) to avoid timing variance
  • For borderline TLE cases: re-judge on a fresh worker and take the better result
  • Floating-point comparison uses epsilon tolerance (|a - b| < 1e-6)

Q5: How would you design the "Run Code" (test run) vs "Submit" feature?

Key points:

  • Run Code: Executes against only sample test cases (2–3 cases); user sees actual vs expected output; lower priority in queue
  • Submit: Executes against all test cases (including hidden); user sees only verdict and stats; higher priority
  • Both share the same sandbox infrastructure but different queue priorities
  • Run Code results are not persisted long-term; Submit results are stored permanently
  • Rate limit Run Code more aggressively (e.g., 20/min) to prevent it from starving Submit operations
