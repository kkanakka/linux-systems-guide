Section 01

Overview & Scale

TicketMaster is the world's largest event ticketing platform, handling everything from small local venues to stadium concerts with 80,000+ seats. The core challenge: selling a finite, non-fungible inventory (every seat is unique) under extreme burst traffic while guaranteeing zero overbooking.

Key Constraints: Unlike e-commerce where you can restock, a concert seat sold twice means two people show up to the same chair. There is no recovery — overbooking is an absolute hard failure.

Functional Requirements

  • Users can browse events by category, location, date, and artist
  • Users can view a seat map and select specific seats or best-available
  • Selected seats are temporarily held while the user completes checkout
  • Payment processing with atomic seat assignment — seat is yours only after payment succeeds
  • Users receive confirmation & e-tickets via email/push notification
  • Sellers can create events, define venue layouts, set pricing tiers

Non-Functional Requirements

Consistency over Availability

We choose strong consistency for the booking path. A brief period of unavailability is acceptable; selling the same seat twice is not. This is the opposite trade-off from most social media systems.

Read-Heavy, Write-Spiky

Event browsing and seat-map views are 100:1 reads-to-writes normally, but during a flash sale the write ratio spikes dramatically. The system must handle both gracefully.

Booking Flow Walkthrough

End-to-End Booking Flow

  1. User searches for an event → Search Service returns matching events from the inverted index
  2. User selects event → Event Service returns event details; Inventory Service returns the seat map with real-time availability
  3. User picks seats → Booking Service calls Inventory Service to place a temporary hold (Redis TTL 5–10 min). Optimistic lock check ensures no double-booking
  4. User enters payment → Booking Service initiates payment via a third-party gateway. If payment fails, the hold is released immediately
  5. Payment confirmed → Booking Service atomically updates the seat status from HELD to BOOKED with the user's booking ID
  6. Confirmation → Notification Service sends e-ticket via email & push notification. A Kafka event is published for downstream analytics
  7. Hold expires (no payment) → Redis TTL fires, a callback releases the seat back to AVAILABLE automatically
Why a Saga, not a 2PC? Distributed two-phase commit is too slow and fragile for a user-facing checkout flow. Instead, the Booking Service uses a saga with compensating transactions: if payment fails after seats are held, the compensating action releases the hold. Each step is idempotent so retries are safe.
Section 03

Seat Reservation & Double-Booking Prevention

This is the most critical part of the system. Two users must never successfully book the same seat. We use a combination of optimistic locking in the database and temporary holds in Redis to achieve this.

Optimistic Locking with Version Field

Every seat row has a version column. When a service reads a seat, it notes the version. When it writes, it includes the version in a WHERE clause. If another transaction modified the row in between, the version won't match and the UPDATE affects zero rows — the application detects this and retries or rejects.

Atomic UPDATE Pattern

UPDATE seats
SET    status    = 'HELD',
       held_by   = :user_id,
       held_at   = NOW(),
       version   = version + 1
WHERE  seat_id   = :seat_id
  AND  event_id  = :event_id
  AND  status    = 'AVAILABLE'
  AND  version   = :expected_version;

-- If affected_rows == 0 → someone else got it first
-- If affected_rows == 1 → hold acquired successfully

Why Optimistic over Pessimistic Locking?

  1. No lock contention: Pessimistic locks (SELECT FOR UPDATE) hold row-level locks, causing connection pool exhaustion under 100K concurrent users
  2. Retry-friendly: The losing user simply sees "seat taken" and picks another — no deadlock scenarios
  3. High throughput: Most seat selections target different seats, so conflicts are rare. Optimistic locking shines when collision rate is low
  4. Deterministic winner: The first transaction to commit wins — simple and fair

Temporary Hold with Redis TTL

After the optimistic lock succeeds, we don't mark the seat as BOOKED immediately — the user still needs to enter payment. Instead, we create a temporary hold with a Redis key that expires after 5–10 minutes.

-- Redis commands for temporary hold
SET  seat_hold:{event_id}:{seat_id}  {user_id}  EX 600   NX

-- EX 600  → expires in 10 minutes (600 seconds)
-- NX      → only set if key does NOT exist (prevents race condition)

-- Check: if SET returns OK → hold acquired
--         if SET returns nil → seat already held by another user
Defense in Depth: Both the database version check and the Redis NX flag prevent double-booking. Even if one layer has a bug, the other catches it. The database is the ultimate source of truth; Redis is a fast coordination layer.

Auto-Release on Expiry

When the Redis TTL expires (user abandoned checkout), a release mechanism kicks in:

  • Option A — Keyspace Notifications: Redis publishes an expiry event on __keyevent@0__:expired. A consumer service listens, verifies the seat is still HELD (not BOOKED), and sets it back to AVAILABLE.
  • Option B — Polling Sweep: A background worker scans for seats where status = 'HELD' and held_at < NOW() - INTERVAL 10 MINUTE. Simpler but slightly delayed.
  • Option C — Hybrid: Use keyspace notifications as primary, with the polling sweep as a safety net to catch any missed expirations.

Hold Too Short (e.g., 2 min)

Users lose seats mid-checkout. Increases abandoned cart rate and frustrates users entering payment details on slow connections.

Hold Too Long (e.g., 30 min)

Scalpers hold seats without buying, reducing effective inventory. Popular events appear sold out while seats are locked by idle sessions.

Section 04

Virtual Waiting Room

When Taylor Swift tickets go on sale, 10 million users hit the buy button simultaneously. Without a waiting room, the system either crashes or randomly rejects users with 503 errors. A virtual waiting room absorbs the spike and meters users into the booking flow at a sustainable rate.

The Thundering Herd Problem: If all 10M users are allowed to hit the Inventory Service at once, even a well-provisioned backend collapses. Database connections exhaust, lock contention skyrockets, and latency spirals to timeouts — nobody gets a ticket.

Queue & Position Assignment

How the Waiting Room Works

  1. User clicks "Buy Tickets" → instead of hitting the booking API directly, the request goes to the Waiting Room Service
  2. Position assigned: The service assigns a queue position using INCR waitingroom:{event_id}:counter in Redis. This is atomic and handles millions of concurrent increments
  3. User sees position: The client polls or receives a WebSocket update showing "You are #34,521 in line. Estimated wait: 8 minutes"
  4. Batch admission: A scheduler releases users in batches (e.g., 500 users every 10 seconds) based on current system capacity
  5. Token issued: When it's the user's turn, they receive a signed admission token (JWT with expiry). The booking API only accepts requests with a valid token
  6. Token expires: If the user doesn't start checkout within 2 minutes of receiving the token, it expires and the slot is recycled to the next person in queue

Batch Admission & Thundering Herd Prevention

Technique How It Helps
Rate-limited admission Release N users per interval, keeping backend load predictable. N is dynamically adjusted based on p99 latency of the booking service
Jittered polling Clients poll for queue position with randomized intervals (base + jitter) to avoid synchronized request waves
CDN-served waiting page The waiting room page is static HTML served from CDN. Only the position update is a lightweight API call, reducing origin load by 99%
Connection pooling WebSocket connections for position updates are multiplexed through a gateway, not directly to backend services
Backpressure signal If the booking service reports high latency, the admission scheduler slows down the release rate automatically

Fairness Guarantees

FIFO ordering: The Redis INCR counter guarantees strict ordering — the user who clicked first gets the lowest number and enters first. No favoritism, no randomness.

Anti-bot measures: Before entering the queue, users must pass a CAPTCHA or proof-of-work challenge. The waiting room also enforces:

  • One queue position per authenticated user (deduplicated by user ID)
  • IP-based rate limiting to prevent automated queue flooding
  • Device fingerprinting to detect headless browsers
  • Admission tokens are bound to the user's session — non-transferable

With Waiting Room

Controlled throughput, fair ordering, stable backend. Users get a clear expectation of wait time. System sells all tickets without crashing.

Without Waiting Room

Random 503 errors, database meltdown, retry storms, unfair advantage to automated scripts, potential complete outage with zero tickets sold.

Sharding Strategy

  1. Seats table: Sharded by event_id. All seats for a single event are co-located, enabling single-shard transactions for multi-seat bookings
  2. Bookings table: Sharded by user_id for efficient "my bookings" queries. Cross-shard joins with seats are avoided by denormalizing event/seat info into the booking record
  3. Events table: Small enough to replicate across all shards or serve from a single primary with read replicas
  4. Hot event isolation: Extremely popular events (e.g., Taylor Swift) can be assigned to dedicated database instances to prevent noisy-neighbor effects

Interview Deep-Dive Questions

Q1: How do you prevent overbooking without pessimistic locks?

Answer: Optimistic locking with a version column. The UPDATE includes WHERE version = :expected. If the row was modified by another transaction, the update affects zero rows and the application retries or rejects. Combined with Redis SET ... NX for the hold phase, you get two independent guards against double-booking. The database is the final arbiter — even if Redis has a split-brain, the DB rejects the duplicate.

Q2: How do you handle flash sales where 10M users want 50K seats?

Answer: Virtual waiting room with a FIFO queue. Users get a position number via atomic INCR. A scheduler admits users in batches calibrated to backend capacity (e.g., 500/10s). The waiting page is served from CDN, so 10M users hitting "refresh" costs almost nothing at the origin. Admission tokens (JWTs with short TTL) ensure only admitted users can call the booking API. Backpressure from the booking service dynamically throttles admission rate.

Q3: How do you render a real-time seat map for 80K seats?

Answer: The seat map is a pre-rendered SVG/canvas with the venue layout baked in (sections, rows, positions). Only the status overlay (available/held/booked) is dynamic. This status is fetched as a compact bitfield or compressed JSON — 80K seats × 2 bits = 20KB. For real-time updates during a flash sale, use Server-Sent Events (SSE) to push delta updates (seat X changed from AVAILABLE to HELD) rather than polling the full map. Stale reads are acceptable for the map — the hold attempt is the consistency boundary.

Q4: What happens if payment takes 30 seconds but the hold is 10 minutes?

Answer: The hold timer is generous (5–10 min) to accommodate slow payment flows. If the user abandons, the TTL auto-releases. If payment succeeds, the seat transitions atomically from HELD to BOOKED. If payment fails, the Booking Service immediately releases the hold (compensating transaction in the saga). The idempotency key on the booking prevents duplicate charges on retries.

Q5: How do you prevent scalper bots from buying all tickets?

Answer: Multi-layer defense: (1) CAPTCHA/proof-of-work before entering the waiting room, (2) one queue position per authenticated user ID, (3) IP rate limiting and device fingerprinting, (4) admission tokens bound to user session (non-transferable), (5) purchase limits per user per event (e.g., max 4 tickets), (6) post-purchase velocity checks — if a user buys tickets across 50 events in a day, flag for review.

Q6: How would you handle a partial failure during multi-seat booking?

Answer: If a user selects 4 seats, all 4 must be held atomically. Since seats are sharded by event_id, all seats for one event are on the same shard, enabling a single-shard transaction. If any seat fails the optimistic lock check, the entire batch is rejected — no partial holds. This is an all-or-nothing operation at the database level. The user is shown which seats were taken and can reselect.
