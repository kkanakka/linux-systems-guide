Section 01

Introduction & Requirements

Comment systems appear deceptively simple: accept text, store it, render a list. In practice, they hide deep complexity in data modeling (representing trees efficiently), query optimization (rendering nested threads without N+1 queries), real-time delivery (pushing new comments to thousands of viewers instantly), moderation (filtering spam and toxicity at scale), and abuse prevention (rate limiting, shadow banning, trust scoring). This guide covers each of these dimensions in production-grade detail.

Functional Requirements

  • Threaded nested replies with configurable max depth (default 10 levels)
  • Real-time delivery via WebSocket — new comments appear without page refresh
  • Upvoting / downvoting with denormalized score aggregation
  • Rich-text comments with sanitized HTML rendering (Markdown input, pre-rendered HTML output)
  • @mention notifications — tag users in comments, trigger push/email/in-app notification
  • Cursor-based pagination with multiple sort modes (Top, Newest, Oldest)
  • Full-text search across comment bodies within a thread or globally
  • Moderation dashboard — ML-assisted spam/toxicity scoring, community flagging, human review queue

Non-Functional Requirements

  • Sub-100ms p99 read latency for cached thread rendering
  • 50K concurrent WebSocket connections per gateway node
  • 10K comments/sec write throughput sustained across the cluster
  • 99.95% availability — no single point of failure
  • Horizontal scalability — adding nodes increases capacity linearly

Architecture Overview

Figure 1 — Full system architecture showing all 5 layers: Client, Gateway, Service, Event Bus, and Data.

Architecture Walkthrough — 5 Layers

  1. Client Layer — Web/mobile clients connect via HTTPS for REST API calls and WSS for real-time WebSocket subscriptions. Clients maintain a local comment cache and use cursor tokens for pagination.
  2. Gateway Layer — API Gateway handles authentication, rate limiting (token bucket per user), request routing, and TLS termination. WebSocket Gateway maintains persistent connections and subscribes to Redis Pub/Sub channels per entity.
  3. Service Layer — Stateless Comment Service handles CRUD, tree assembly, and business logic. Moderation Service runs ML classifiers synchronously (fast pre-filter) and asynchronously (deep analysis). Notification Service fans out @mention and reply alerts.
  4. Event Bus Layer — Kafka topics (comment.created, comment.voted, comment.flagged, comment.moderated) decouple the write path from downstream consumers. Samza/Flink consumers process events for moderation, counters, search indexing, and real-time delivery.
  5. Data Layer — Primary DB (sharded by entity_id) stores comments and closure table. Redis cluster serves as L2 cache (serialized trees) and pub/sub backbone. Search index (Elasticsearch/Galene) powers full-text comment search.

Write Flow — 6 Arrows

  1. Client → API Gateway — POST /v1/threads/{id}/comments with body, parent_id, and auth token.
  2. API Gateway → Comment Service — Rate limit check passes, request forwarded with authenticated user context.
  3. Comment Service → Moderation Service — Synchronous pre-filter call (<10ms): regex, blocklist, rate check. Returns allow/block/flag.
  4. Comment Service → Primary DB — Serializable transaction: INSERT comment + INSERT closure entries + INCREMENT parent reply_count.
  5. Comment Service → Kafka — Publish comment.created event with full payload.
  6. Comment Service → Client — HTTP 201 with created comment including server-generated fields (id, created_at, body_html).

Read Flow — 3-Tier Cache

  1. L1 In-Process LRU (30s TTL, 100MB per instance, ~0ms) — Hot threads served from application memory. Highest hit rate for viral content.
  2. L2 Redis Cluster (5min TTL, serialized comment trees, ~5ms) — Shared across all service instances. Populated on L1 miss.
  3. L3 Database (closure table join, cursor pagination, DataLoader batching, ~50ms) — Full tree reconstruction from DB. Result populates both L2 and L1.
Section 02

Threading Models

The single most consequential architectural decision in a comment system is how to represent and query tree-structured data. The threading model determines write performance, read efficiency, move/reparent cost, and query complexity. There are four classical approaches, each with fundamentally different trade-offs. Getting this wrong means either rebuilding the data layer later or living with permanent performance limitations.

2.1 Adjacency List

Each comment stores a parent_id foreign key pointing to its direct parent. Top-level comments have parent_id = NULL. This is the simplest model and the most natural mapping of a tree to a relational table.

Strengths

  • O(1) insert — just set parent_id on the new row
  • O(1) move — update a single parent_id to reparent an entire subtree
  • No extra storage overhead beyond the FK column
  • Standard B-tree index on parent_id, no exotic index types needed

Weaknesses

  • O(n) subtree retrieval — requires recursive CTE or multiple round-trips
  • Ancestor queries also require recursion
  • Depth computation requires traversal
-- Fetch top-level comments
SELECT * FROM comments
WHERE thread_id = ? AND parent_id IS NULL
ORDER BY created_at DESC LIMIT 20;

-- Fetch direct replies
SELECT * FROM comments
WHERE parent_id = ? ORDER BY created_at ASC;

-- Recursive CTE to get full subtree (PostgreSQL)
WITH RECURSIVE comment_tree AS (
  SELECT id, parent_id, body_html, depth, created_at
  FROM comments WHERE id = :root_id
  UNION ALL
  SELECT c.id, c.parent_id, c.body_html, c.depth, c.created_at
  FROM comments c
  JOIN comment_tree ct ON c.parent_id = ct.id
)
SELECT * FROM comment_tree ORDER BY depth, created_at;

2.2 Materialized Path

Each comment stores its full ancestry as a path string, e.g., /A/B/D/. Subtree queries become prefix scans. Depth is computed by counting delimiters.

Strengths

  • O(1) insert — concatenate parent's path + own ID
  • O(log n) subtree via LIKE prefix scan or GIN trigram index
  • O(1) ancestor retrieval — split the path string
  • Depth = count slashes in the path

Weaknesses

  • O(subtree) move — must rewrite path for every descendant
  • Path column size limits maximum depth
  • LIKE prefix queries use partial index only — not as efficient as B-tree equality
  • Wide rows in high-depth trees
-- Schema addition
ALTER TABLE comments ADD COLUMN path VARCHAR(500);
-- e.g., path = '/A/B/D/'

-- Insert: parent path = '/A/B/', new comment id = 'D'
INSERT INTO comments (id, thread_id, parent_id, path, ...)
VALUES ('D', :thread, 'B', '/A/B/D/', ...);

-- Fetch all descendants of comment B
SELECT * FROM comments
WHERE thread_id = ? AND path LIKE '/A/B/%'
ORDER BY path, created_at;

-- GIN trigram index for fast prefix matching
CREATE INDEX idx_comments_path_trgm
ON comments USING gin (path gin_trgm_ops);

2.3 Nested Sets

Each node stores lft and rgt boundary values. A node's descendants have lft values between the parent's lft and rgt. Reads are extremely fast (single range scan), but writes are catastrophically expensive.

Strengths

  • O(1) subtree read — single range scan on (lft, rgt)
  • Ancestors and depth computable from boundary values
  • Entire tree ordered by a single lft sort

Weaknesses

  • O(n) insert — every insert requires renumbering all nodes with lft or rgt > insertion point
  • Heavy write locking — renumbering touches the entire table, causing lock contention
  • Impractical for high-write systems — at 10K writes/sec, renumbering is a non-starter
-- Schema additions
ALTER TABLE comments ADD COLUMN lft INT, ADD COLUMN rgt INT;

-- Fetch all descendants of a comment
SELECT child.* FROM comments child, comments parent
WHERE child.thread_id = ?
  AND parent.id = :parent_id
  AND child.lft > parent.lft
  AND child.rgt < parent.rgt
ORDER BY child.lft;

-- Insert requires gap creation (O(n) updates!)
UPDATE comments SET rgt = rgt + 2
WHERE thread_id = ? AND rgt >= :insertion_point;
UPDATE comments SET lft = lft + 2
WHERE thread_id = ? AND lft > :insertion_point;
INSERT INTO comments (lft, rgt, ...) VALUES (:insertion_point, :insertion_point + 1, ...);
Why Nested Sets Fail at Scale: Inserting a single comment into a thread with 10,000 existing comments requires updating up to 10,000 rows (all nodes with lft/rgt greater than the insertion point). At 10K writes/sec, this generates 100M row updates/sec of overhead — clearly impractical.

2.4 Closure Table (Recommended)

The Optimal Choice for Comment Systems

A Closure Table uses a separate junction table (comment_closure) that stores every ancestor-descendant pair with its depth. This denormalizes the tree structure into a flat, indexed lookup table.

  • Insert is O(d) where d = depth of the new node. Since depth is typically capped at 10–15, this is effectively O(1).
  • Subtree retrieval is O(1) — a single indexed join on ancestor_id.
  • Ancestor retrieval is O(1) — a single indexed join on descendant_id.
  • Depth-aware queries are trivial — filter on the depth column.
  • Storage trade-off: O(n×d) rows in the closure table, where n = comments and d = avg depth.

Closure Table Schema

CREATE TABLE comment_closure (
  ancestor_id   UUID NOT NULL REFERENCES comments(id),
  descendant_id UUID NOT NULL REFERENCES comments(id),
  depth         INT  NOT NULL DEFAULT 0,
  PRIMARY KEY (ancestor_id, descendant_id)
);

CREATE INDEX idx_closure_descendant ON comment_closure(descendant_id, depth);
CREATE INDEX idx_closure_ancestor   ON comment_closure(ancestor_id, depth);

Closure Table Data Example

For a tree A → B → D (A is root, B is child of A, D is child of B), the closure table contains:

ancestor_iddescendant_iddepth
AA0
AB1
AD2
BB0
BD1
DD0

Every node has a self-referencing entry at depth 0. Adding node E under B would insert: (B,E,1), (A,E,2), (E,E,0) — exactly depth + 1 rows.

Insertion SQL

-- Insert a new comment C_new under parent P
-- Step 1: Insert the comment row
INSERT INTO comments (id, thread_id, parent_id, body_raw, body_html, depth, ...)
VALUES (:new_id, :thread, :parent_id, :body_raw, :body_html, :depth, ...);

-- Step 2: Insert closure entries
-- Copy all ancestors of the parent, incrementing depth by 1
INSERT INTO comment_closure (ancestor_id, descendant_id, depth)
SELECT ancestor_id, :new_id, depth + 1
FROM comment_closure
WHERE descendant_id = :parent_id;

-- Step 3: Insert self-reference
INSERT INTO comment_closure (ancestor_id, descendant_id, depth)
VALUES (:new_id, :new_id, 0);

Query Examples

-- Get full subtree of comment A (all descendants)
SELECT c.* FROM comments c
JOIN comment_closure cc ON c.id = cc.descendant_id
WHERE cc.ancestor_id = :comment_a_id AND cc.depth > 0
ORDER BY c.depth, c.created_at;

-- Get direct children only (depth = 1)
SELECT c.* FROM comments c
JOIN comment_closure cc ON c.id = cc.descendant_id
WHERE cc.ancestor_id = :parent_id AND cc.depth = 1
ORDER BY c.created_at;

-- Get all ancestors of a comment (breadcrumb)
SELECT c.* FROM comments c
JOIN comment_closure cc ON c.id = cc.ancestor_id
WHERE cc.descendant_id = :comment_id AND cc.depth > 0
ORDER BY cc.depth DESC;

Why Closure Table Wins for Comment Systems

Adjacency List wins on pure insert speed and storage, but Closure Table wins on the operations that dominate a comment system’s workload: subtree retrieval, ancestor lookup, and depth-aware queries. Since depth is capped (≤10–15), the O(d) insert overhead is negligible. The O(n×d) storage is a modest trade-off: a 10M-comment system with avg depth 5 produces ~50M closure rows — well within a single shard’s capacity. The closure table also pairs naturally with cursor-based pagination (Section 6) and DataLoader batching (Section 4).

Diagram Walkthrough

  1. Adjacency List (top-left) — Each node stores only parent_id. Arrows show the FK relationships. To get A's subtree, you start at A and recursively follow parent_id references — 3 queries for 3 levels.
  2. Materialized Path (top-right) — Each node stores its full ancestry: A stores "/A/", B stores "/A/B/", D stores "/A/B/D/". Subtree of A = WHERE path LIKE '/A/%'. The path string grows with depth.
  3. Nested Sets (bottom-left) — A has lft=1, rgt=8. B has lft=2, rgt=5. D has lft=3, rgt=4. Subtree of A = WHERE lft BETWEEN 1 AND 8. Inserting a new node under C requires incrementing lft/rgt for all nodes to the right.
  4. Closure Table (bottom-right) — A separate table stores every ancestor–descendant pair. The 6 closure rows for the 3-node tree A→B→D are shown. Subtree of A = JOIN closure WHERE ancestor = A. One indexed join, no recursion.

Schema Design Decisions

Raw + Rendered Body

Storing both body_raw (Markdown) and body_html (sanitized HTML) avoids re-rendering on every read. The HTML is generated once on write (or edit) through a sanitization pipeline that strips XSS vectors while preserving safe formatting. Edits update both columns atomically.

Denormalized Counters

vote_count and reply_count are denormalized on the comment row to avoid COUNT(*) aggregations on every read. Updated atomically via SET vote_count = vote_count + 1 inside the vote/reply transaction. Eventually consistent (5s max lag) for display, but the source-of-truth is always the votes/comments tables.

mod_score on Comment Row

Storing the ML moderation score directly on the comment enables fast filtering (WHERE mod_score < 0.7) without joining to a moderation table. Updated asynchronously as ML analysis completes. Supports threshold-based auto-actions without additional queries.

UUID v7 as Primary Key

Time-ordered: monotonically increasing within a millisecond, preserving B-tree insert locality. No enumeration: unlike auto-increment, attackers can’t guess comment IDs. Distributed generation: no central ID server needed. B-tree friendly: sequential inserts avoid random page splits.

Index Strategy

Critical Indexes

-- Primary read path: thread listing with sort
CREATE INDEX idx_comments_thread_created
ON comments (thread_id, created_at DESC, id);

-- Top-sort variant
CREATE INDEX idx_comments_thread_votes
ON comments (thread_id, vote_count DESC, created_at DESC, id);

-- Reply fetch
CREATE INDEX idx_comments_parent_created
ON comments (parent_id, created_at ASC, id)
WHERE parent_id IS NOT NULL;

-- Closure table indexes (already shown above)
-- idx_closure_desc ON comment_closure(descendant_id, depth)
-- idx_closure_anc  ON comment_closure(ancestor_id, depth)

-- User's comment history
CREATE INDEX idx_comments_user
ON comments (user_id, created_at DESC);

The partial index on parent_id WHERE NOT NULL excludes root comments, shrinking the index and speeding reply lookups. The composite (thread_id, created_at DESC, id) supports cursor-based pagination with no sort spill.

Write Path & Read Path

Figure 3 — Detailed write path (left) and read path (right) with cache tiers and consistency model.

Diagram Walkthrough

  2. Right side: Read Path shows the 3-tier cache hierarchy (L1 in-process → L2 Redis → L3 DB) with TTLs and latencies at each tier.
  3. Center: Kafka bus connects the write path to async consumers (moderation, real-time delivery, search indexing, counter updates).
  4. Bottom: Consistency annotations show which parts are strongly consistent (DB transaction) vs. eventually consistent (cached counters, real-time delivery).

Write Path — 6 Stages

Stage 1: Input Validation + HTML Sanitization

Validate body length (1–10,000 chars), depth check (parent depth < max), thread existence and lock status. Parse Markdown to HTML using a whitelist-based sanitizer (e.g., DOMPurify server-side) that strips <script>, onclick, javascript: URIs, and other XSS vectors while preserving <b>, <i>, <a>, <code>, <blockquote>.

Stage 2: Rate Limit Check

Redis sliding-window rate limiter: max 5 comments per minute per user, max 20 per hour. Implemented as a sorted set with timestamps as scores. ZRANGEBYSCORE counts recent entries; ZADD records the new request. Returns 429 Too Many Requests if exceeded.

Stage 3: Synchronous Moderation Pre-Filter

Fast pre-filter (<10ms): regex banned-word list, URL reputation check (DNS blocklist), content bounds check. If confidence > 0.95, reject immediately with 400. If < 0.3, proceed. Between 0.3–0.95, publish and flag for async deep analysis. This prevents obvious spam from ever reaching the database.

Stage 4: Database Transaction (Serializable)

BEGIN;
  -- Insert comment row
  INSERT INTO comments (id, thread_id, parent_id, user_id,
    body_raw, body_html, depth, mod_score, status, created_at, updated_at)
  VALUES (:id, :thread, :parent, :user, :raw, :html,
    :depth, :mod_score, 'ACTIVE', NOW(), NOW());

  -- Insert closure entries (copy parent's ancestors + self)
  INSERT INTO comment_closure (ancestor_id, descendant_id, depth)
  SELECT ancestor_id, :id, depth + 1
  FROM comment_closure WHERE descendant_id = :parent;

  INSERT INTO comment_closure (ancestor_id, descendant_id, depth)
  VALUES (:id, :id, 0);

  -- Increment parent's reply_count
  UPDATE comments SET reply_count = reply_count + 1
  WHERE id = :parent;

  -- Increment thread's comment_count
  UPDATE threads SET comment_count = comment_count + 1
  WHERE id = :thread;
COMMIT;

Stage 5: Kafka Event Publish

After transaction commits, publish comment.created event to Kafka with full comment payload, user metadata, and moderation scores. Kafka key = thread_id ensures ordering within a thread. If Kafka is unavailable, event is written to a local WAL buffer and replayed on recovery.

Stage 6: HTTP 201 Response

Return the created comment with all server-generated fields: id, created_at, body_html, vote_count: 0, reply_count: 0, depth. Client can optimistically render without waiting for WebSocket confirmation.

Read Path — 3-Tier Cache

TierStorageTTLCapacityLatency

L2: Redis Cluster Serialized comment trees (MessagePack) 5 minutes Shared cluster ~5ms

Stampede Protection

When a cache entry expires on a viral thread, thousands of concurrent requests would all miss cache and hit the DB simultaneously. Distributed lock (Redis SET ... NX EX): only the first cache miss acquires the lock and populates. All other requests wait on the lock (with a short timeout) and then read the freshly populated cache. Fallback: serve stale cache if lock wait exceeds 100ms.

Consistency Model

Strong Consistency

  • Comment content and threading (DB transaction)
  • Vote uniqueness (unique constraint)
  • Comment status changes (ACTIVE → DELETED)

Eventual Consistency

  • Denormalized counters (vote_count, reply_count) — 5s max lag
  • Cache content — 30s L1, 5min L2 staleness
  • Real-time delivery — at-most-once with gap-sync on reconnect
Section 05

Real-Time Comment Delivery

Figure 4 — Real-time comment delivery: Comment Service → Kafka → Samza → Redis Pub/Sub → WebSocket Gateway → Clients.

Delivery Flow Walkthrough

  1. Comment Service publishes a comment.created event to Kafka after the DB transaction commits.
  2. Samza consumer (Real-Time Service) reads the event, enriches it with user display info (avatar, name), and determines which entity/thread the comment belongs to.
  3. Redis Pub/Sub publish — the Samza consumer publishes the enriched comment to a Redis channel keyed by entity:{entity_id}.
  4. WebSocket Gateway subscription — each gateway node subscribes to Redis channels for the entities its connected clients are viewing. When a message arrives on a channel, the gateway fans it out to all local WebSocket connections watching that entity.
  5. Client receives the new comment via WebSocket frame and renders it in the UI with an animation. The client merges server events with its local optimistic update (from its own POST), deduplicating by comment ID.
  6. Disconnect handling — when a client disconnects, the gateway unsubscribes from channels that have no remaining viewers. Channel cleanup prevents Redis memory leaks.

Connection Lifecycle

  1. Connect — Client opens WSS connection to the WebSocket Gateway with auth token. Gateway validates JWT, assigns connection to a connection pool.
  2. Subscribe — Client sends {"action": "subscribe", "entity_id": "X"}. Gateway subscribes to entity:X Redis channel if not already subscribed (refcount tracks viewers per channel).
  3. Receive Events — Gateway pushes comment.created, comment.voted, comment.deleted events as they arrive on the channel.
  4. Disconnect — Client closes connection or network drops. Gateway decrements refcount; if zero, unsubscribes from Redis channel.

Gap-Sync Protocol

WebSocket delivery is at-most-once: if a client is disconnected when a comment arrives, it misses that event. To handle this:

  • Each event carries a monotonically increasing sequence_number per entity (backed by a Redis counter).
  • On reconnect, the client sends its last_seen_sequence_number.
  • The gateway queries a short-lived event log (Redis list, 1-hour TTL) for all events with sequence > last_seen and replays them.
  • If the gap is too large (>1000 events), the client falls back to a full REST API fetch of the latest page.

Scale: 50K Connections Per Node

Each WebSocket Gateway node handles 50,000 concurrent connections using non-blocking I/O (epoll/kqueue). Memory footprint: ~20KB per connection (buffers + metadata) = ~1GB per node. A cluster of 20 gateway nodes supports 1M concurrent viewers. Horizontal scaling: add nodes and let the load balancer distribute new connections via least-connections algorithm.

Section 06

Cursor-Based Pagination

Two-Level Pagination

Comment threads use a two-level pagination scheme:

  • Level 1 — Root comments: Paginated by cursor across all top-level comments (parent_id IS NULL) in a thread.
  • Level 2 — Replies: Each root comment’s replies are lazy-loaded on demand with a separate cursor scoped to that parent.

This avoids loading the entire tree upfront. A thread with 10K comments might show 20 root comments initially, each with a “Show 12 replies” button that triggers a Level 2 fetch.

Cursor Definitions

Pagination LevelCursor FieldsUse Case
Root (Newest/Oldest) (created_at, id) Chronological root comment listing
Root (Top) (vote_count, created_at, id) Popularity-sorted root comments
Replies (parent_id, created_at, id) Replies under a specific parent, always chronological

Sort Modes

ModeORDER BY ClauseUse Case
Top vote_count DESC, created_at DESC, id DESC Default — surfaces the highest-scored comments
Newest created_at DESC, id DESC Real-time discussions, live events, breaking news
Oldest created_at ASC, id ASC Reading threads chronologically from the beginning

Why Not Offset-Based Pagination?

Three Fatal Flaws

  • Drift: If 5 new comments are inserted while a user paginates, OFFSET 20 skips 5 unseen comments and shows 5 duplicates. In a real-time system with 10K writes/sec, every page request is inconsistent.
  • Performance: OFFSET 10000 forces the database to scan and discard 10,000 rows before returning the next 20. At scale, this causes query timeouts and full-table scans.
  • Inconsistency: Concurrent deletes can cause a page to have fewer items than LIMIT or skip items entirely, confusing client rendering logic.

Cursor Approach

Range Query + Opaque Token + Limit+1 Trick

-- Newest sort: cursor = (last_created_at, last_id)
SELECT * FROM comments
WHERE thread_id = :thread
  AND parent_id IS NULL
  AND (created_at, id) < (:cursor_ts, :cursor_id)
ORDER BY created_at DESC, id DESC
LIMIT 21;  -- fetch 21, return 20, use 21st to determine has_more

-- Top sort: cursor = (last_vote_count, last_created_at, last_id)
SELECT * FROM comments
WHERE thread_id = :thread
  AND parent_id IS NULL
  AND (vote_count, created_at, id) < (:cursor_votes, :cursor_ts, :cursor_id)
ORDER BY vote_count DESC, created_at DESC, id DESC
LIMIT 21;

The cursor is encoded as an opaque base64 token: base64(vote_count:created_at:id). The client treats it as a black box, allowing the server to change the underlying sort strategy without breaking API compatibility.

Section 07

Spam Detection & Content Moderation

Figure 5 — Multi-layer moderation pipeline: Pre-Filters → ML Classifiers → External APIs → Score-Based Routing → Human Review Queue.

Moderation Pipeline Walkthrough

  1. Comment arrives at the Comment Service after passing input validation.
  2. Layer 1 Pre-Filters run synchronously (<10ms): rate limiting check, content length bounds, regex banned-word list, URL reputation check against DNS blocklist. Fast reject for obvious violations.
  3. Layer 2 ML Classifiers run in parallel: toxicity model (Perspective API or self-hosted transformer), spam classifier (trained on historical moderation decisions), LLM semantic analysis for nuanced cases.
  4. Layer 3 External APIs (optional): Akismet for spam detection, CleanSpeak for profanity and contextual analysis. Results merged with ML scores via weighted average.
  5. Score-Based Routing determines the comment’s fate based on the combined moderation score. Thresholds are tunable per-tenant.
  6. Human Review Queue receives borderline cases. Moderators see the comment, ML scores, user history, and surrounding context. Their decisions feed back into ML training data.

Layer 1: Pre-Filters (<10ms)

  • Rate limiting — Redis sliding window: 5 comments/min, 20/hour per user
  • Content bounds — Min 1 char, max 10,000 chars, reject empty or excessively long
  • Regex banned words — Compiled regex matching against a curated blocklist (updated daily). Case-insensitive, Unicode-aware, handles leetspeak substitutions
  • URL reputation — Extract URLs, check against DNS blocklist (Spamhaus, SURBL). Known-malicious domains = instant reject

Layer 2: ML Classifiers

  • Toxicity Model — Perspective API or self-hosted DistilBERT fine-tuned on toxic comment datasets. Returns scores for: toxicity, severe_toxicity, identity_attack, insult, profanity, threat. Inference latency: ~50ms.
  • Spam Classifier — Gradient-boosted tree (XGBoost) trained on historical moderation decisions. Features: user account age, comment frequency, text entropy, link density, duplicate detection, user trust score. Inference latency: ~10ms.
  • LLM Semantic Analysis — For borderline cases (score 0.3–0.7), a lightweight LLM analyzes context: is the comment sarcasm? Is a flagged word used in a non-harmful context? Inference latency: ~200ms. Only triggered for ambiguous cases.

Score-Based Routing

Score RangeActionVisibility
< 0.3 Auto-approve Immediately visible to all users
0.3 – 0.7 Publish + Flag for review Visible but queued for human review
> 0.7 Hold for review Hidden until moderator approves

Thresholds are tunable per-tenant: a children’s platform might set auto-approve at <0.1, while a news discussion forum might use <0.4.

Human Review Queue

Priority Formula

priority = report_count × ML_boundary_proximity × (1 / user_trust_score)

Comments with many reports, borderline ML scores, and low-trust authors are reviewed first.

Moderator Actions

  • Approve — Mark as clean, increment user trust score
  • Reject with reason — Remove comment, decrement user trust, log reason code for ML training
  • Edit — Remove offending content while preserving the rest (with "[edited by moderator]" notice)
  • Shadow-ban — User’s future comments are visible only to themselves; prevents ban evasion and angry retaliation
  • Escalate — Send to senior moderator or legal team for complex cases

Feedback loop: Every approve/reject decision is logged as a labeled training example. The ML models are retrained weekly on the latest moderation decisions, continuously improving accuracy.

Shadow Banning

Shadow-banned users can still post comments, but their comments are only visible to themselves (the API returns them normally for the banned user’s session but excludes them for everyone else). This is more effective than hard bans because:

  • The banned user doesn’t know they’re banned, so they don’t create new accounts
  • No angry responses or ban-evasion behavior
  • Implemented by adding a WHERE user_id NOT IN (shadow_banned_set) filter on reads, using a Redis set for O(1) membership check

  • Low-trust users (<0.3) always go through human review regardless of ML score
  • Trust score is bounded [0.0, 1.0] and decays toward 0.5 over time without activity

Escalation Thresholds

  • 10 flags → escalate to priority review queue
  • Trusted user flags (trust > 0.8) → weighted 2x (a single trusted flag counts as two)
  • Flag abuse detection — if a user flags >20 comments in an hour, their flags are temporarily de-weighted and their account is reviewed for coordinated flagging
Section 08

Scaling Strategy

Figure 6 — Scaling architecture: database sharding, multi-tier caching, CDN edge, and connection pooling.

Scaling Walkthrough

  1. Database sharding by hash(entity_id) % N distributes threads across shards. All comments for a single thread land on the same shard, ensuring single-shard tree reconstruction.
  2. Read replicas (2 per shard) handle read traffic. Writes go to primary only. Replication lag <1s for strong consistency.
  3. L1/L2/L3 cache tiers absorb read traffic before it reaches the DB. Write-through invalidation: on new comment, invalidate L2 (L1 expires naturally via TTL).
  4. CDN edge caching for public threads (10s TTL, stale-while-revalidate). Surrogate-Key headers enable targeted purge when comments change.
  5. PgBouncer connection pooling (200 connections per shard) prevents connection exhaustion under load spikes.

Database Sharding

  • Shard key: hash(entity_id) % N where N = number of shards
  • Co-location: All comments, closure entries, votes, and flags for a thread live on the same shard
  • Primary + 2 read replicas per shard
  • Cross-entity queries (e.g., "all comments by user X") require scatter-gather across all shards, served by a secondary index or a dedicated user-comments lookup table

Multi-Tier Cache Details

Write-Through Invalidation

On comment creation: immediately invalidate the L2 Redis cache for the affected thread. L1 cache expires naturally (30s TTL). This ensures new comments appear within 30s even without WebSocket delivery.

Stampede Protection

Distributed lock (SETNX comment_lock:{thread_id} {instance_id} EX 5): only one cache miss populates L2. Others wait up to 100ms on the lock, then fall through to stale cache or direct DB query.

CDN Edge Caching

  • Public threads only (private/restricted threads bypass CDN)
  • 10s TTL with stale-while-revalidate: 30 — serve stale while refreshing in background
  • Surrogate-Key headers (Surrogate-Key: thread:{id}) enable targeted purge on comment creation without flushing the entire CDN cache
  • CDN absorbs 60–80% of read traffic for popular threads

Performance Optimizations

  • DataLoader batching — Prevents N+1 queries when loading user avatars and vote states for a page of comments. Batches all user_ids into a single WHERE user_id IN (...) query.
  • Denormalized counters — vote_count and reply_count on the comment row avoid COUNT(*) per-comment on reads.
  • Pagination limits — Max depth: 10 levels. Max page size: 50 comments. Prevents unbounded tree traversal and memory allocation.
  • PgBouncer connection pooling — 200 connections per shard, transaction-level pooling. Prevents connection exhaustion during traffic spikes.

Section 10

Failure Modes & Resilience

DB Shard Failure

  • Auto-failover in 30 seconds — Orchestrator promotes a read replica to primary
  • Writes return 503 + Retry-After header during failover window
  • Reads served from remaining replica with eventual consistency (replication lag <1s pre-failure)
  • Client exponential backoff with jitter: base 100ms, max 30s, jitter ±50%

Redis Failure

  • Cache miss fallback — All reads go directly to DB. Latency increases from ~5ms to ~50ms but system remains available
  • Rate limiting fallback — Switches to in-process token bucket (less accurate but functional). Per-instance limits approximated from global config.
  • Pub/Sub fallback — Real-time delivery degrades: gateway nodes fall back to gossip protocol (peer-to-peer event sharing). Higher latency (~2s) but eventual delivery.

Kafka Unavailability

  • Synchronous fanout — Comment Service directly calls Moderation Service and Real-Time Service (circuit breaker protected)
  • Local WAL buffer — Events written to a local Write-Ahead Log (bounded to 1GB). When Kafka recovers, events are replayed in order.
  • Idempotent consumers — All Kafka consumers are idempotent (dedup by comment_id), so replayed events don’t cause duplicates

Thundering Herd on Viral Threads

When a thread goes viral (e.g., celebrity post, breaking news), a single entity can receive 100K+ reads/sec:

  • Stampede lock — Only one cache miss triggers a DB query; others wait on the lock
  • CDN stale-while-revalidate — CDN serves stale content while refreshing in background
  • Request coalescing — Multiple identical requests within a 50ms window are collapsed into a single DB query; all requestors receive the same response
  • Pre-warming — For predictable events (scheduled live streams), pre-populate cache before the event starts

Why Espresso for Comments?

Espresso supports document-level partitioning by entity_id, keeping all comments for a post co-located. It provides:

  • Automatic horizontal scaling with partition rebalancing
  • Secondary indexes for parent_id and user_id queries
  • Change-capture streams that integrate natively with Kafka
  • Strong consistency within a partition — critical for closure table transactions and counter increments

Interview Questions — Design & Architecture

  1. "How would you handle a viral post with 1M comments?"
    • Hot partition mitigation: CDN edge caching, stampede locks, request coalescing
    • Pre-computed top-N snapshots refreshed every 30s
    • Read replicas absorb read traffic; writes rate-limited to prevent shard overload
  2. "How do you ensure real-time comment updates?"
    • WebSocket connections via gateway cluster, Kafka → Samza → Redis Pub/Sub fanout
    • Gap-sync protocol handles reconnection without missing events
    • Fallback: long-polling every 5s for clients that don't support WebSocket
  3. "Why not store the entire comment tree as a single document?"
    • Document size limits (16MB in MongoDB, similar in Espresso)
    • Write contention: every new comment requires updating the root document
    • No partial updates: must read-modify-write the entire tree
    • Cannot paginate within a document
  4. "Compare Closure Table vs. Adjacency List for your comment system."
    • Closure Table: O(1) subtree reads, O(d) inserts, O(n×d) storage
    • Adjacency List: O(1) inserts, O(n) subtree reads via recursive CTE
    • Closure Table wins when read-to-write ratio is high (50:1 in our case)
  5. "How do you handle @mentions in comments?"
    • Parse Markdown for @username patterns during write, resolve to user_ids
    • Store mentions in a separate comment_mentions table
    • Kafka event triggers notification service to send in-app/push/email alerts

Interview Questions — Data Model & Scaling

  1. "How would you handle like counts without race conditions?"
    • Atomic increment: UPDATE comments SET vote_count = vote_count + 1 WHERE id = ?
    • Idempotent: unique constraint on (user_id, comment_id) in votes table
    • Toggle: vote endpoint upserts — if exists, delete (undo vote) and decrement
  2. "What happens when a user deletes a comment with replies?"
    • Soft delete: set status = DELETED, render as "[deleted]" in UI
    • Preserve tree structure for reply context
    • Hard delete only when all descendants are also deleted (garbage collection job)
  3. "How do you partition comments across databases?"
    • Hash(entity_id) ensures all comments for one entity co-locate
    • Cross-entity queries (user comment history) = scatter-gather or secondary index
    • Resharding: consistent hashing with virtual nodes to minimize data movement
  4. "How does cursor pagination work when vote_count changes?"
    • Cursor encodes values at read time; a comment’s rank can shift between pages
    • No duplicates within a session (each page starts after the cursor point)
    • Trade-off: some comments may be missed or re-ranked; acceptable for non-critical UX
  5. "How would you add a ‘pin comment’ feature?"
    • Add pinned BOOLEAN and pinned_at TIMESTAMPTZ to comments table
    • Fetch pinned comments separately, prepend to results before pagination
    • Max 1–3 pinned comments per thread (enforced by the Comment Service)

Interview Questions — Moderation & Edge Cases

  1. "How do you prevent coordinated spam attacks?"
    • IP-based throttling at API gateway (token bucket)
    • Samza consumer detects burst patterns: same content from multiple accounts = botnet
    • CAPTCHA triggered after rate limit threshold
    • Device fingerprinting to correlate accounts
  2. "What is shadow banning and when would you use it?"
    • User’s comments visible only to themselves; they don’t know they’re banned
    • Prevents ban evasion (creating new accounts out of anger)
    • Used for repeat low-severity offenders who don’t warrant a full ban
  3. "How do you handle edit history?"
    • Append-only comment_edits table with (comment_id, body_raw, body_html, edited_at)
    • Show "edited" badge in UI; click to view diff
    • Moderation re-scores on edit (edited comment goes through ML pipeline again)
  4. "How would you implement a trust-based moderation system?"
    • Per-user trust score: approved comments increase trust, rejections decrease it
    • High-trust users (>0.8) skip ML analysis; low-trust (<0.3) always require human review
    • Feedback loop: moderation decisions continuously train the ML models
  5. "What about internationalization and emoji?"
    • UTF-8mb4 storage (4-byte UTF-8 for full emoji support)
    • Content-length limits in characters, not bytes
    • Language-specific ML models for moderation (toxicity varies by language/culture)
    • RTL text rendering support in the comment UI
