Section 01

Overview & Scale Estimation

A comment system powers user interactions across posts, videos, articles, news feeds, and product listings. Users can post top-level comments, reply in threads, like/dislike, and see real-time updates. The system must handle massive read-heavy traffic with low-latency rendering of threaded comment trees.

Functional Requirements

  • Create comments on any entity (post, video, article, product)
  • Threaded replies — nested conversations with configurable max depth
  • Like / react to comments
  • Sort by Top, Newest, or Oldest
  • Real-time — new comments appear without page refresh
  • Moderation — spam detection, flagging, admin review
  • Pagination — efficient loading of large comment threads

Section 02

Architecture & Data Model

The system is built around a Comment Service that coordinates reads and writes, backed by a partitioned database for persistence, a distributed cache for hot threads, and an event bus for asynchronous fanout.

Core Components

Comment Service

Stateless REST/gRPC service that handles CRUD operations, builds comment trees, and enforces business rules (max depth, rate limiting, content validation).

Redis Cache

Caches serialized comment trees per entity. Invalidated on writes. Sorted sets for paginated top-level comment lists. TTL-based eviction for cold threads.

Espresso DB (Primary Store)

Partitioned by entity_id so all comments for one entity live on the same shard. Supports secondary indexes on parent_id and user_id.

Kafka (Event Bus)

Publishes comment.created, comment.liked, comment.flagged events. Consumers update counters, trigger moderation, push real-time notifications.

Architecture Diagram

Figure 1 — Comment System high-level architecture: write path, read path, and async event processing.

Write Path Walkthrough

  1. Client sends POST /comments with entity_id, content, and optional parent_id.
  2. Comment Service validates input (depth check, rate limit, content length), generates a Snowflake ID, computes depth from parent.
  3. Espresso DB — comment row is inserted; reply_count on parent is incremented atomically.
  4. Cache Invalidation — the cached comment tree for this entity_id is invalidated in Redis.
  5. Kafka Event — a comment.created event is published with full comment payload.
  6. Downstream Consumers — notification service alerts the entity owner and parent comment author; moderation pipeline scores the comment for spam.

Read Path Walkthrough

  1. Client requests GET /comments?entity_id=X&sort=top&cursor=abc
  2. Comment Service checks Redis for a cached page. On cache hit, returns immediately (< 5ms).
  3. On cache miss, queries Espresso for top-level comments (parent_id IS NULL) sorted by the requested order, with cursor-based pagination.
  4. For each top-level comment, fetches up to N replies (fan-out limited to depth ≤ 3). Tree is assembled in-memory.
  5. Result is cached in Redis with a TTL (30s for hot entities, 5min for cold) and returned to client.
Section 03

Threading Approaches

The core challenge of a comment system is representing and querying tree-structured data efficiently. There are three classical approaches, each with distinct trade-offs.

Adjacency List

Each comment stores a parent_id pointing to its parent. Top-level comments have parent_id = NULL. The tree is reconstructed by the application via recursive queries or in-memory assembly.

-- Fetch top-level comments
SELECT * FROM comments
WHERE entity_id = ? AND parent_id IS NULL
ORDER BY created_at DESC LIMIT 20;

-- Fetch replies for a comment
SELECT * FROM comments
WHERE parent_id = ? ORDER BY created_at ASC;

Materialized Path

Each comment stores its full ancestry as a path string, e.g., /root_id/parent_id/self_id. Subtree queries become prefix matches.

-- Schema addition
path VARCHAR(500)  -- e.g., "/100/205/310"

-- Fetch all descendants of comment 205
SELECT * FROM comments
WHERE entity_id = ? AND path LIKE '/100/205/%'
ORDER BY path, created_at;

Nested Sets

Each node stores lft and rgt values. A node's descendants have lft between the parent's lft and rgt. Reads are fast but writes require renumbering.

-- Schema additions
lft INT, rgt INT

-- Fetch all descendants of a comment
SELECT * FROM comments
WHERE entity_id = ? AND lft > parent.lft AND rgt < parent.rgt
ORDER BY lft;

Comparison

Criteria Adjacency List Materialized Path Nested Sets
Write complexity O(1) — single insert O(1) — concat path O(n) — renumber siblings
Read subtree Recursive / multi-query Single LIKE prefix query Single range query
Move subtree Update one parent_id Update all descendant paths Full renumbering
Depth limit Unlimited (but cap in practice) Limited by path column size Unlimited
Index friendliness Standard B-tree index LIKE prefix — partial index use Range scan on (lft, rgt)
Concurrency No contention No contention Lock contention on renumber

Recommendation

Adjacency List + Denormalized Counts

For a high-write, real-time comment system, Adjacency List is the best choice:

  • O(1) writes with no locking or renumbering — critical at 115 writes/sec.
  • Denormalized reply_count and like_count avoid expensive aggregation queries on reads.
  • Depth capped at 3 means tree reconstruction never requires deep recursion — at most 4 queries (one per level) or a single query with application-side grouping.
  • Cache the assembled tree in Redis. The 50:1 read ratio means most reads hit cache, making the multi-query reconstruction cost negligible.

Materialized Path is a strong alternative if you need subtree queries without depth limits, but the path column becomes a liability at scale (wide rows, partial index usage).

Sorting & Pagination

Sort Modes

Mode ORDER BY Use Case
Top like_count DESC, created_at DESC Default — surfaces the most-liked comments first
Newest created_at DESC Real-time discussions, live events
Oldest created_at ASC Reading threads chronologically

Cursor-Based Pagination (Not Offset)

Why Not Offset?

Offset-based pagination (LIMIT 20 OFFSET 100) breaks in real-time comment feeds:

  • Drift — if 5 new comments are inserted while a user paginates, OFFSET 20 skips 5 comments the user hasn't seen and shows 5 duplicates.
  • Performance — the database must scan and discard all OFFSET rows. At OFFSET 10000 this is a full scan.
  • Inconsistency — concurrent deletes can cause pages to have fewer items than expected.

Cursor-Based Approach

The cursor encodes the last seen sort key, making each page request a simple range query with no skipping:

-- Newest sort: cursor = last_seen_created_at
SELECT * FROM comments
WHERE entity_id = ?
  AND parent_id IS NULL
  AND created_at < :cursor_created_at
ORDER BY created_at DESC
LIMIT 20;

-- Top sort: cursor = (last_like_count, last_created_at)
SELECT * FROM comments
WHERE entity_id = ?
  AND parent_id IS NULL
  AND (like_count, created_at) < (:cursor_likes, :cursor_ts)
ORDER BY like_count DESC, created_at DESC
LIMIT 20;

The API returns a next_cursor token (base64-encoded sort values) that the client passes on the next request.

Offset Pagination

  • Simple to implement
  • Supports "jump to page N"
  • Duplicates/skips on concurrent writes
  • O(offset) DB scan cost
  • Breaks in real-time feeds

Cursor Pagination

  • Stable under concurrent writes
  • O(1) seek via index
  • Consistent real-time experience
  • Cannot jump to arbitrary page
  • Slightly more complex API contract
Interview Tip: Always mention cursor-based pagination in any system with real-time writes. It shows you understand the consistency vs. convenience trade-off. Mention that the cursor should be opaque to the client (base64-encoded) so the server can change the underlying sort strategy without breaking clients.
Section 05

Moderation Pipeline

At 10M comments/day, manual moderation is impossible. The system uses a multi-stage pipeline combining ML scoring, community flagging, and human review for edge cases.

Pipeline Architecture

Moderation Flow

  1. Inline Pre-Screen (synchronous) — on comment submission, a lightweight regex + blocklist filter catches obvious spam (URLs, repeated characters, known slurs). Runs in < 10ms; rejects immediately if confidence > 0.95.
  2. Kafka Event — comment.created event published to the moderation topic.
  3. ML Spam Classifier (async) — a Samza/Flink consumer runs the comment through a trained model that scores for spam, toxicity, and manipulation. Features include user history, comment text embeddings, entity context, and temporal patterns.
  4. Auto-Action — if ML confidence > 0.9, the comment status is updated to SPAM automatically. If between 0.5–0.9, status set to FLAGGED for human review.
  5. Community Flagging — users can report comments. After N flags (threshold varies by entity size), comment is moved to FLAGGED and added to the admin review queue.
  6. Admin Review Queue — human reviewers see flagged comments with ML scores, user history, and context. They approve, delete, or escalate. Decisions feed back into ML training data.

Edge Case — Spam Bursts: A botnet can submit thousands of comments in seconds. Mitigations: per-user rate limiting (e.g., 5 comments/minute), CAPTCHA after threshold, IP-based throttling via a token bucket at the API gateway, and a Kafka consumer that detects burst patterns and bulk-flags.

Why Espresso for Comments?

Espresso supports document-level partitioning by entity_id, keeping all comments for a post co-located. It provides:

  • Automatic horizontal scaling with partition rebalancing
  • Secondary indexes for parent_id and user_id queries
  • Change-capture streams that integrate natively with Kafka
  • Strong consistency within a partition — important for counter increments

Common Interview Questions

Design & Architecture

  1. "How would you handle a viral post with 1M comments?" — Discuss hot partition mitigation: separate cache layer (Couchbase/Redis), pre-computed top-N comment snapshots, read replicas, and rate-limited writes with a queue.
  2. "How do you ensure real-time comment updates?" — WebSocket/SSE connections for active viewers. Kafka consumer pushes new comments to a presence-aware notification service. Fallback: polling every 5s.
  3. "Why not store the entire comment tree as a single document?" — Document size limits, write contention on the root document, inability to paginate within the tree, and no partial updates.

Data Model & Scaling

  1. "How would you handle like counts without race conditions?" — Atomic increment in database (UPDATE comments SET like_count = like_count + 1 WHERE id = ?), idempotent like tracking in a separate comment_likes table with (user_id, comment_id) unique key.
  2. "What happens when a user deletes a comment with replies?" — Soft delete: set status to DELETED, show as "[deleted]" in UI while preserving reply tree structure. Hard delete only when all descendants are also deleted.
  3. "How do you partition comments across databases?" — By entity_id ensures all comments for one entity are co-located. Cross-entity queries (e.g., "all comments by user X") use a secondary index or a separate user-comments table.

Moderation & Edge Cases

  1. "How do you handle edit history?" — Store edits in an append-only comment_edits table. Show "edited" badge in UI. Moderation re-scores on edit.
  2. "What about internationalization and emoji?" — UTF-8 storage, content-length limits in characters not bytes, and language-specific ML models for moderation.
  3. "How does cursor pagination work when like_count changes?" — The cursor encodes the values at read time. A comment's position can shift between pages, but no duplicates or gaps occur within a single pagination session.
  4. "How would you add a 'pin comment' feature?" — Add a pinned boolean and pinned_at timestamp. Pinned comments are fetched separately and prepended to results before pagination begins.
