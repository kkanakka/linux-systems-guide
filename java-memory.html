Java Memory Architecture — Deep Dive

Mapping JVM Internals to Linux Process Memory

This guide maps every Java component to its exact location in the Linux process memory layout. We trace how ZooKeeper and Espresso (LinkedIn's NoSQL store) use Text, Data, BSS, Heap, Stack, and Kernel memory — with diagrams for every scenario.

1. Java Process Memory Layout — Where Everything Lives

A running Java program is a regular Linux process. Its virtual address space has the same segments as any C program (text, data, BSS, heap, stack) — but the JVM carves up the heap into specialized sub-regions and adds large off-heap native allocations.

Segment-by-Segment Breakdown

TEXT Machine Code — Read-Only, Shared

  • JVM binary code: libjvm.so, the core JVM runtime (interpreter loop, GC algorithms, class loader)
  • JIT-compiled methods: After the JVM detects a “hot” method (called thousands of times), the C1 (client) or C2 (server) compiler compiles it from bytecode to native machine instructions. These go into the CodeCache (technically in native memory, but conceptually “text”)
  • Interpreter dispatch loop: The bytecode interpreter itself is compiled C++ code in the text segment
  • This region is read-only and can be shared between processes running the same JVM binary

DATA Initialized Global/Static Variables

  • JVM version strings, default configuration constants
  • Signal handler function pointers
  • Static final fields from loaded Java classes (after class loading, the JVM places resolved constant pool entries here)
  • Values are read from the executable file when the JVM binary is loaded

BSS Uninitialized Global/Static Variables

  • Large static arrays before initialization
  • JVM internal counters that start at zero
  • Key advantage: the binary doesn’t need to store these bytes — the kernel just records the size needed and fills with zeros at startup

HEAP Dynamic Memory — The Largest Region

In a Java process, “heap” has two meanings:

Java Heap (-Xmx)Native Heap (malloc)
Eden: Where new objects are born. new Request(), new byte[1024] — short-lived
Survivor S0/S1: Objects that survived at least 1 minor GC cycle
Old Gen (Tenured): Long-lived objects promoted after surviving many GC cycles. DataNode tree, HashMap, SessionTracker 		Metaspace: Class metadata, method vtables (-XX:MaxMetaspaceSize)
DirectByteBuffer: Netty I/O buffers, allocated via sun.misc.Unsafe
JIT CodeCache: Compiled native methods (-XX:ReservedCodeCacheSize)
Thread-local storage for native (C/C++) code
Managed by GC (G1, ZGC, etc.) Outside GC control. Managed by malloc/free or Netty reference counting

STACK Per-Thread, Auto-Managed

  • Each Java thread gets its own stack, typically 512KB–1MB (-Xss)
  • Contains method call frames with local variables, return addresses, operand values
  • Primitives (int, long, boolean) are stored directly on the stack
  • Object references are on the stack, but the actual objects are on the heap
  • 200 threads × 1MB = 200MB of stack memory!
  • If recursion goes too deep: StackOverflowError

KERNEL SPACE Not Accessible to JVM

  • Socket buffers (sk_buff) for TCP connections
  • Page cache (cached disk I/O)
  • File descriptor table
  • epoll interest lists
  • The JVM interacts with kernel space only through system calls
Total Process Memory = Heap (-Xmx) + Metaspace + Direct Buffers + Code Cache + Thread Stacks × N + Native libs + Kernel overhead
Example: 4G + 256M + 512M + 240M + 200×1M + 100M ≈ 5.3GB

2. Stack vs Heap — What Goes Where in Java

Stack: Per-Thread, Automatic

When ZooKeeper’s RequestProcessor chain calls PrepRequestProcessor.processRequest() → SyncRequestProcessor.processRequest() → FinalRequestProcessor.processRequest(), each call pushes a frame onto that thread’s stack:

┌─────────────────────────────────────┐  ← Stack Pointer (top)
│ serialize() frame                   │
│   int offset = 0    (primitive)     │
│   ByteBuffer buf    (ref → off-heap)│
│   return address                    │
├─────────────────────────────────────┤
│ processRequest() frame              │
│   Request req       (ref → heap)    │
│   int txnType = 1   (primitive)     │
│   byte[] data       (ref → heap)    │
│   return address                    │
├─────────────────────────────────────┤
│ main() frame                        │
│   String[] args     (ref → heap)    │
│   int port = 2181   (primitive)     │
│   return address                    │
└─────────────────────────────────────┘  ← Stack Bottom

When serialize() returns, its frame is popped — all its locals vanish instantly. No GC needed.

Heap: Shared, GC-Managed

Objects created with new live on the heap. References to them are stored on the stack (or in other heap objects):

WhatWhere on HeapLifetime

new byte[1024]Young Gen (Eden)Short — temporary buffer
DataNode("/app/config")Old Gen (Tenured)Long — lives for process lifetime
HashMap<String,DataNode>Old Gen (Tenured)Long — the ZK data tree
SessionTrackerOld Gen (Tenured)Long — tracks all client sessions

3. Text, Data, BSS, Heap, Stack — Java Edition

Mapping C Concepts to Java

In Chapter 6, we saw how a C program’s variables map to memory segments. Here’s the Java equivalent:

Global uninitialized (char globBuf[65536])static byte[] buffer (null until assigned)BSS (reference is null/zero)
Local variable (int result)int result = x * xStack (primitive directly)
Heap allocation (malloc(1024))new byte[1024]Java Heap (Eden → Survivor → Old Gen)
Function pointerMethod reference / vtableMetaspace (off-heap)
Text / codeJIT-compiled bytecodeCodeCache (off-heap, executable)

The JVM adds a layer of indirection. In C, you manage all memory directly. In Java:

  • javac compiles .java → .class (bytecode, not native machine code)
  • The JVM interpreter executes bytecodes one by one (this interpreter lives in the text segment as compiled C++)
  • The JIT compiler (C1/C2) detects hot methods and compiles them to native x86/ARM code, storing the result in the CodeCache
  • The GC manages the Java heap, moving objects between young and old gen

4. fork() in Java — The Dangerous Copy

What Happens When a Java Process Forks

Java apps almost never call fork() directly. But when ZooKeeper does Runtime.exec("some command"), the JVM internally calls fork() + exec(). Here’s the dangerous sequence:

  1. fork() duplicates the entire virtual address space:
    • 4GB heap, 1GB of Netty direct buffers, all 200 thread stacks
    • With Copy-on-Write (COW), physical pages aren’t actually copied until modified
    • But the page table entries are all duplicated — for a large process, this alone can require hundreds of MB
  2. Only ONE thread survives:
    • If your ZooKeeper has 200 threads and one calls fork(), the child gets only that one thread
    • The other 199 threads simply vanish
    • Any mutexes they held are now permanently locked in the child
    • This includes JVM internal locks, GC locks, classloader locks
  3. exec() must follow immediately:
    • exec() replaces the entire address space with a new program
    • This cleans up the dangerous half-forked state
    • That’s why Runtime.exec() does fork()+exec() as an atomic pair
OOM Kill Risk: Redis (which forks for persistence) and large JVMs can trigger the OOM killer during fork(). Even with COW, the kernel needs to account for the potential memory if all pages were written. The vm.overcommit_memory sysctl controls how aggressive the kernel is about allowing this. Set to 1 (always overcommit) for Redis/large JVM workloads, or 0 (heuristic, default) for conservative behavior. 
# Check overcommit settings
cat /proc/sys/vm/overcommit_memory    # 0=heuristic, 1=always, 2=never
cat /proc/sys/vm/overcommit_ratio     # percentage of RAM (when mode=2)

# See how much memory is committed vs available
cat /proc/meminfo | grep -i commit
# CommitLimit:     8388608 kB
# Committed_AS:    5242880 kB  ← total memory "promised" to processes

# Check for OOM kills

How clone() Creates Threads

clone() is the same system call that fork() uses, but with different flags:

fork()clone(SIGCHLD)Nothing — full copy (COW)

Threads ARE processes that share memory. In Linux, there is no fundamental distinction — both are task_struct entries. The clone() flags determine how much is shared.

6. CPU Cores, NUMA, and cgroups

CPU Core Assignment

The Linux CFS (Completely Fair Scheduler) maintains a per-core run queue implemented as a red-black tree sorted by virtual runtime. Each Java thread is a kernel-schedulable entity (a task_struct). The scheduler picks which core runs which thread based on fairness.

  • Threads are free to migrate between cores unless pinned with taskset or sched_setaffinity()
  • In ZooKeeper, you’d typically pin the SyncRequestProcessor thread (the one flushing to disk) to a specific core to avoid cache thrashing
  • GC threads should also be pinned to avoid NUMA penalties

NUMA — Non-Uniform Memory Access

On a dual-socket server (common for ZooKeeper/Espresso production boxes), each socket has its own memory controller:

Access TypeLatencyExample
Local (same NUMA node)~80nsCore 0 accessing RAM on Socket 0
Remote (cross-node via QPI/UPI)~150nsCore 0 accessing RAM on Socket 1

Almost 2x slower for cross-node access!

The JVM’s -XX:+UseNUMA flag tells the GC to allocate young gen on the NUMA node where the allocating thread runs. Without it, a thread on core 4 (node 1) might be accessing heap pages physically located on node 0, paying the cross-node penalty on every object access.

7. cgroups + Page Cache — The Hidden Memory Trap

What Counts Against memory.max

The kernel tracks memory.current for a cgroup as:

memory.current = anon RSS + file-backed pages (page cache) + kernel slab memory

This is one of the most misunderstood aspects of cgroups. Page cache (disk cache) of multiple processes is shared and counts against the same cgroup memory limit.

  • When ZooKeeper reads its transaction log, those cached file pages are charged to the cgroup
  • When Espresso reads MySQL data files, those pages also count
  • Both processes’ cached disk I/O is summed into one memory.current

What Happens at the Limit

  1. First: Kernel reclaims page cache (drops clean cached pages) — I/O performance degrades but nothing crashes
  2. If still not enough: The OOM killer picks a process to terminate!

Container Sizing Formula

Container memory = JVM heap + off-heap (Metaspace + direct buffers + thread stacks) + page cache headroom + kernel overhead
ComponentZooKeeper ExampleEspresso Example
JVM Heap (-Xmx)2GB4GB
Metaspace256MB512MB
Thread Stacks200 × 1MB = 200MB300 × 1MB = 300MB
Direct Buffers128MB512MB
Page Cache headroom~500MB (txn logs)~1GB (MySQL data)
Kernel overhead~100MB~200MB
Total minimum~3.2GB~6.5GB
Recommended cgroup3.5–4GB7–8GB
Common mistake: Setting memory.max = -Xmx. If your heap is 4GB and your cgroup limit is 4GB, you’re guaranteed to get OOM killed — there’s zero room for Metaspace, stacks, direct buffers, and page cache!

8. Netty I/O — Two Worlds of Memory

Why Netty Uses Off-Heap (Direct) Buffers

When performing network I/O, data must eventually reach the kernel’s socket buffers. The path depends on buffer type:

Buffer TypeI/O PathCopies
Heap byte[]JVM heap → copy to native temp buffer → kernel → NIC2 copies
Direct ByteBufferDirect buffer → kernel → NIC1 copy
sendfile()Page cache → NIC (DMA)0 copies (zero-copy!)

The extra copy for heap buffers exists because the GC can move objects in memory during compaction. The kernel needs a stable address for its DMA transfer. Direct buffers are pinned at a fixed native address.

Netty’s Memory Management

  • PooledByteBufAllocator: Netty maintains arenas of pre-allocated direct memory, slicing them into buffers as needed (avoids expensive malloc/free per request)
  • Reference counting: Each ByteBuf has a reference count. retain() increments, release() decrements. When it hits 0, memory is returned to the pool
  • This is NOT garbage collected. Forgetting to call release() causes a native memory leak that won’t show up in Java heap dumps!

Kernel Side

For each TCP connection, the kernel maintains:

  • struct sock — socket state (TCP state machine, window size, RTT)
  • sk_buff chain for send buffer (default ~87KB, tunable via net.core.wmem_default)
  • sk_buff chain for receive buffer (default ~87KB, tunable via net.core.rmem_default)
  • epoll interest list entry (red-black tree in kernel)
  • File descriptor table entry

Kernel Layer (Shared)

  • Page cache: Both the Espresso JVM’s file I/O and MySQL’s data file reads are cached here

  • File descriptors: Data files + sockets + pipes

The request flow: Client → Netty (off-heap) → Router (heap) → MySQL query → InnoDB (its process) → Disk
