Section 01

Overview & Scale

Problem Statement: Design a cloud file storage and synchronization service that allows users to store files in the cloud, sync them across multiple devices, and share them with others. The system must handle massive scale while ensuring data integrity, low-latency sync, and efficient bandwidth usage.

Core Requirements

Functional Requirements

  • Upload, download, and delete files
  • Automatic sync across all linked devices
  • File versioning — revert to previous versions
  • Share files/folders with other users
  • Offline editing with sync on reconnect
  • Support files up to 10 GB

Non-Functional Requirements

  • Consistency: All devices see the same file state
  • Durability: Zero data loss (99.999999999% durability)
  • Availability: 99.9% uptime SLA
  • Low Latency: Sync within seconds of a change
  • Bandwidth Efficiency: Minimize data transfer
  • Security: End-to-end encryption at rest and in transit

Section 02

High-Level Architecture

Core Components

The system is decomposed into four major backend services, each with a single responsibility:

ComponentResponsibilityKey Tech
Block Server Splits files into chunks, computes hashes, uploads/downloads blocks to cloud storage SHA-256, CDC, S3/GCS
Metadata Service Stores file tree, chunk manifests, user info, sharing permissions, version history MySQL/PostgreSQL, Redis cache
Notification Service Pushes real-time sync events to connected clients when files change WebSocket, Long Polling
Cloud Storage Stores raw file chunks with high durability and availability S3, Azure Blob, GCS
Figure 1 — High-level architecture showing the client, Block Server, Metadata Service, Notification Service, and Cloud Storage.

Sync Flow Walkthrough

8-Step File Sync Flow

  1. File Change Detected: The desktop client’s filesystem watcher detects a file modification (create, update, or delete) in the synced folder.
  2. Chunking & Hashing: The client’s Block Server module splits the file into 4 MB chunks and computes a SHA-256 hash for each chunk.
  3. Diff Calculation: The client compares local chunk hashes against the last-known manifest from the Metadata Service. Only new or modified chunks are marked for upload.
  4. Chunk Upload: Changed chunks are uploaded to Cloud Storage via the Block Server API. Each chunk is stored by its content hash as the key, enabling deduplication.
  5. Metadata Update: Once all chunks are persisted, the client sends the updated file manifest (ordered list of chunk hashes, file path, size, modified timestamp) to the Metadata Service.
  6. Version Created: The Metadata Service creates a new version record, atomically updating the file’s current state while preserving the previous version for rollback.
  7. Notification Broadcast: The Metadata Service publishes a change event to the Notification Service, which fans it out to all other devices linked to the same account (or shared folder collaborators).
  8. Remote Sync: Other clients receive the notification, fetch the new manifest from Metadata Service, identify which chunks they’re missing, and download only those chunks from Cloud Storage.

Why This Design Works at Scale

  • Decoupled data & metadata paths: Large file chunks go directly to Cloud Storage; only small metadata goes through the Metadata Service, keeping it lightweight.
  • Content-addressable storage: Chunks keyed by SHA-256 hash enable global deduplication — identical chunks uploaded by different users are stored only once.
  • Stateless Block Servers: Any Block Server instance can handle any request; horizontal scaling is trivial.
  • Push-based sync: Notification Service ensures near-instant propagation without clients polling.
Section 03

File Chunking & Delta Sync

Chunking Strategy

Files are split into fixed-size 4 MB chunks. Each chunk is independently hashed, stored, and transferred. This is the foundation of Dropbox’s bandwidth efficiency and deduplication.

PropertyValueRationale
Chunk size4 MBBalance between metadata overhead (too small) and re-upload waste (too large)
Hash algorithmSHA-256Collision-resistant; 256-bit hash = negligible collision probability at 100B+ files
Last chunk≤ 4 MBFinal chunk can be smaller; stored as-is
Chunk namingContent hashContent-addressable: sha256(chunk_bytes) is the storage key
File: presentation.pptx (12 MB)

┌──────────┬──────────┬──────────┐
│ Chunk 0  │ Chunk 1  │ Chunk 2  │
│  4 MB    │  4 MB    │  4 MB    │
│ hash: a3 │ hash: 7f │ hash: e1 │
└──────────┴──────────┴──────────┘

Manifest: [a3f8c2..., 7fb41e..., e19d03...]

Fixed vs Content-Defined Chunking (CDC)

ApproachProsCons
Fixed-Size (4 MB) Simple, predictable, fast Byte insertion shifts all subsequent chunk boundaries → many chunks change
Content-Defined (CDC) Insertion-resilient; chunk boundaries determined by content via rolling hash (Rabin fingerprint) More complex, variable-size chunks, slower

Dropbox uses CDC in production for better delta efficiency, but fixed-size is acceptable for interview scope.

Delta Sync

Key Insight: Only Upload Changed Chunks

When a user edits a 1 GB file and changes only a few bytes, the entire file does not get re-uploaded. The client:

  1. Re-chunks the modified file
  2. Computes SHA-256 for each new chunk
  3. Compares against the previous manifest
  4. Uploads only chunks whose hashes differ
Before edit:  [chunk_A] [chunk_B] [chunk_C] [chunk_D]
After edit:   [chunk_A] [chunk_B'] [chunk_C] [chunk_D]
                           ↑
                     only this chunk changed

Upload: 1 chunk (4 MB) instead of full file (16 MB)
Bandwidth saved: 75%

Delta Sync Algorithm

  1. Local manifest lookup: Client retrieves the stored manifest [hash_A, hash_B, hash_C, hash_D]
  2. New manifest computation: Client chunks the updated file → [hash_A, hash_B', hash_C, hash_D]
  3. Diff: Compare element-wise. Position 1 differs: hash_B ≠ hash_B'
  4. Upload: Upload only chunk_B' to Cloud Storage
  5. Commit: Send updated manifest to Metadata Service

Content Deduplication

Cross-User Deduplication

Because chunks are stored by content hash, if two users upload the same file (or the same chunk appears in different files), it is stored only once in Cloud Storage.

ScenarioWithout DedupWith Dedup
100 users upload same 100 MB file10 GB stored100 MB stored
Email attachment sent to 1000 recipients5 GB (5 MB × 1000)5 MB stored
OS installer across enterprise500 GB5 GB stored

The Metadata Service maintains a reference count per chunk hash. A chunk is garbage-collected only when its reference count drops to zero.

Sync Protocol & Conflict Resolution

Client-Side Watcher

Each Dropbox desktop client runs a filesystem watcher that monitors the synced folder for changes in real time. This is the entry point for all local-to-cloud sync operations.

OSWatcher APINotes
macOSFSEventsCoalesced events, efficient for large trees
LinuxinotifyPer-directory watches; limited by max_user_watches
WindowsReadDirectoryChangesWRecursive watch support natively

Watcher Responsibilities

  • Detect file create, modify, delete, rename events
  • Debounce rapid changes (e.g., auto-save every 2 seconds)
  • Queue events for the sync engine to process sequentially
  • Handle moved/renamed files by detecting matching content hashes
  • Maintain a local SQLite database of file states for crash recovery

Push Notifications

Long Polling vs WebSocket

MethodMechanismTradeoffs
Long Polling Client sends HTTP request; server holds it open until there’s a change or timeout (60s). Client reconnects immediately after response. Simpler infrastructure, works through all proxies/firewalls, slightly higher latency
WebSocket Persistent bidirectional TCP connection between client and Notification Service Lower latency, less overhead per message, requires sticky sessions or connection state management

Dropbox historically used long polling for its Notification Server due to simplicity and firewall compatibility. Modern implementations may use WebSocket for lower latency.

Notification Flow

  1. Client A modifies a file and commits new metadata to the Metadata Service
  2. Metadata Service writes a change event to the message queue (e.g., Kafka)
  3. Notification Service consumes the event and looks up all active sessions for the affected user/shared folder
  4. For long polling: the held HTTP response is released with a “changes available” signal
  5. Client B receives the notification, queries Metadata Service for the updated manifest, and downloads only missing chunks

Conflict Resolution

The Conflict Problem

Two users (or two devices of the same user) edit the same file while offline or before sync completes. When both come online, the server has two divergent versions.

Strategy 1: Last-Writer-Wins (LWW)

  • The version with the latest timestamp overwrites the other
  • Simple to implement
  • Risk: silent data loss for the “losing” edit
  • Acceptable for low-collaboration single-user scenarios

Strategy 2: Conflict Copy

  • Both versions are preserved; the second is saved as file (conflicted copy).ext
  • User manually resolves the conflict
  • No data loss, but creates user friction
  • This is what Dropbox actually does

Conflict Detection Mechanism

Each file has a version vector (or simpler: a monotonically increasing version number maintained by the Metadata Service). When a client pushes an update, it includes the base_version it was editing from. If base_version < current_version, a conflict is detected.

Client A: edit file v3 → upload → server accepts → file becomes v4
Client B: edit file v3 → upload → base_version=3 < current_version=4 → CONFLICT
  → Server creates "file (Client B's conflicted copy)" as v5

LinkedIn Mapping & Interview Questions

LinkedIn Technology Mapping

How Dropbox’s architecture maps to LinkedIn’s internal infrastructure:

Dropbox ComponentLinkedIn EquivalentWhy
Cloud Storage (S3) Ambry LinkedIn’s distributed blob store for large immutable objects (media, documents). Content-addressable, replicated across data centers.
Metadata Service (MySQL) Espresso LinkedIn’s document-oriented NoSQL store with strong consistency, secondary indexes, and change capture — ideal for file metadata and version records.
Notification Service (Kafka) Kafka LinkedIn invented Kafka. Used for reliable, ordered event streaming — file change events, sync notifications, audit logging.
Block Server API (REST) Rest.li LinkedIn’s REST framework with automatic schema validation, backward-compatible API evolution, and built-in routing.
Chunk dedup index (Redis) Couchbase Distributed cache layer for fast hash lookups to check if a chunk already exists before uploading.
User auth & ACL Nuage LinkedIn’s centralized authorization service for fine-grained permission checks on shared files/folders.

LinkedIn Sync Flow (Mapped)

  1. Client detects file change, chunks it, computes SHA-256 hashes
  2. Client calls Rest.li Block Server API with chunk hashes to check existence
  3. Block Server queries Couchbase dedup cache — returns list of missing chunks
  4. Client uploads missing chunks to Ambry blob store
  5. Block Server commits updated file manifest to Espresso
  6. Espresso change capture emits event to Kafka
  7. Notification consumers read from Kafka and push to connected clients via WebSocket
  8. Other clients download missing chunks from Ambry

Common Interview Questions

Design & Architecture

  1. How would you handle a 10 GB file upload?
    Chunk into 4 MB blocks, upload in parallel with resumable upload support. If connection drops, only retry failed chunks. Use pre-signed URLs for direct-to-storage uploads.
  2. Why separate metadata from file content?
    Metadata is small, structured, and needs strong consistency (ACID). File content is large, unstructured, and needs high throughput. Different storage engines are optimal for each.
  3. How do you ensure data durability?
    Store chunks with 3+ replicas across availability zones. Use erasure coding for cold storage. Verify integrity with SHA-256 on read. 11 nines durability target.

Scaling & Performance

  1. How does deduplication save storage?
    Content-addressable chunks mean identical data is stored once. At enterprise scale, Dropbox reported 60%+ storage savings from dedup alone.
  2. How would you scale the Metadata Service?
    Shard by user_id for user-owned files. For shared folders, use a separate shard key (e.g., folder_id). Cache hot metadata in Redis/Couchbase.
  3. What if the Notification Service goes down?
    Clients fall back to periodic polling (e.g., every 60s). Kafka retains events, so when the service recovers, no events are lost. Clients also do a full sync on reconnect.

Conflict & Consistency

  1. How do you handle concurrent edits to the same file?
    Optimistic concurrency: each update includes a base_version. If another update landed first, detect conflict and create a conflict copy. Never silently overwrite.
  2. How do you handle offline edits?
    Client queues all changes locally in a journal. On reconnect, replays changes in order against the server. Conflicts are resolved per the conflict copy strategy.
  3. Should you use strong or eventual consistency?
    Strong consistency for metadata (users must see accurate file state). Eventual consistency is acceptable for chunk storage (chunks are immutable once written, so there is no update conflict).

Advanced Topics (Bonus Points)

  • Compression: Compress chunks before upload (LZ4 for speed, zstd for ratio). Skip for already-compressed formats (JPEG, MP4, ZIP).
  • Encryption: Client-side encryption with per-user keys. Server stores encrypted chunks and never sees plaintext.
  • Smart Sync: Keep file stubs locally (metadata only); download content on-demand to save local disk space.
  • Bandwidth throttling: Client-side rate limiting to avoid saturating user’s network connection.
  • Selective sync: Allow users to choose which folders sync to which devices.
  • LAN sync: Discover peers on the same LAN and transfer chunks locally instead of going through the cloud.
